## `std::vector`의 내부 구조
- `vector`는 세 개의 포인터(또는 포인터 + 정수 두 개)로 Heap 위의 연속 배열을 관리하는 컨테이너
```
┌─────────────────────────────────────────────────────────────────┐
│   Stack                                                         │
│                                                                 │
│  ┌──────────────────────────────┐                               │
│  │     vector<int> v            │                               │
│  │                              │                               │
│  │  ┌────────────┬───────────┐  │                               │
│  │  │  _begin    │  포인터    │──┼──────────────────────────┐    │
│  │  ├────────────┼───────────┤  │                          │    │
│  │  │  _end      │  포인터    │──┼──────────────────┐       │    │
│  │  ├────────────┼───────────┤  │                  │       │    │
│  │  │  _end_cap  │  포인터    │──┼──────────────────┼───┐   │    │
│  │  └────────────┴───────────┘  │                  │   │   │    │
│  └──────────────────────────────┘                  │   │   │    │
│                                                    │   │   │    │
└────────────────────────────────────────────────────┼───┼───┼────┘
                                                     │   │   │
┌────────────────────────────────────────────────────┼───┼───┼────┐
│  Heap                                              │   │   │    │
│                                                    │   │   │    │
│  ┌──────┬──────┬──────┬──────┬────────┬────────┐   │   │   │    │
│  │  1   │  2   │  3   │  4   │        │        │   │   │   │    │
│  │ v[0] │ v[1] │ v[2] │ v[3] │        │        │   │   │   │    │
│  └──────┴──────┴──────┴──────┴────────┴────────┘   │   │   │    │
│     ↑                            ↑                 │ ↑ │   │    │
│  _begin                        _end               _end_cap │    │
│     │                            └─────────────────┘ └─┘   │    │
│     └──────────────────────────────────────────────────────┘    │
│  ◀─────── size() = 4 ───────▶                                  │
│  ◀────────────── capacity() = 6 ──────────────▶                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
- `size()`: 현재 저장된 원소 개수
- `capacity()`: 재할당 없이 저장할 수 있는 최대 원소 개수
```cpp
std::vector<int> v = {1, 2, 3, 4};
v.reserve(6);

std::cout << v.size();      // 4
std::cout << v.capacity();  // 6
```

## `push_back`의 내부 동작
### size < capacity인 경우(빠른 경로)
```
push_back(5) 호출 시 (size=4, capacity=6)

Heap
┌──────┬──────┬──────┬──────┬──────┬──────┐
│  1   │  2   │  3   │  4   │  5   │(미사용)│
└──────┴──────┴──────┴──────┴──────┴──────┘
                            ↑ 이 위치에 5를 배치 후 _end++

비용: O(1) - Heap 접근 1회 + 포인터 이동 1회
```

### size == capacity인 경우(재할당, 느린 경로)
```
push_back(7) 호출 시 (size=6, capacity=6)

1단계: 새 배열 할당 (capacity * 2 = 12)
        Heap에서 12칸 짜리 연속 블록을 새로 malloc

2단계: 기존 원소 이동
        기존 배열 → 새 배열로 이동 생성자 호출 (noexcept이면 이동, 아니면 복사)

3단계: 기존 배열 해제
        기존 Heap 블록 free

4단계: 포인터 갱신
        _begin, _end, _end_cap 모두 새 배열 기준으로 갱신

5단계: 새 원소 배치
        _end 위치에 7 배치 후 _end++

비용: O(N) - 기존 원소를 전부 이동/복사
```
> ⚠️ 재할당 후 포인터 무효화: 재할당이 발생하면 `_begin`이 가리키는 주소가 바뀜
>
> 재할당 전에 얻어둔 포인터나 이터레이터는 모두 댕글링 포인터가 됨
```cpp
std::vector<int> v = {1, 2, 3};
int* p = &v[0];     // v의 첫 원소를 가리키는 포인터

v.push_back(4);     // 재할당이 발생할 수 있음
// p는 이제 해제된 메모리를 가리킴 → Undefined Behavior
std::cout << *p     // ❌
```

## 분할 상환 분석(Amortized Analysis)
- `push_back`은 재할당 시 $O(N)$이지만, 평균 비용을 따지면 $O(1)$임
- 이를 분할 상환 분석으로 증명함

### 핵심 아이디어 - 2배 확장 전략
- capacity가 가득 찰 때마다 2배로 늘리는 전략을 사용함
```
N개의 push_back을 수행할 때 총 이동 비용 계산 (N=16 기준)

capacity 1  → 2  : 원소 1개 이동
capacity 2  → 4  : 원소 2개 이동
capacity 4  → 8  : 원소 4개 이동
capacity 8  → 16 : 원소 8개 이동
                 ─────────────────
총 이동 횟수   :  1 + 2 + 4 + 8 = 15 = N - 1

N번의 push_back에 대한 총 비용 = N(삽입) + (N-1)(이동) < 2N
→ 평균 비용 = 2N / N = O(1) 상수
```

### 2배가 아닌 1.5배를 쓰는 구현도 있는 이유
|   확장 배율   |   재할당 횟수 |   총 이동 비용    |   메모리 낭비 |
|--------------|-------------|-----------------|--------------|
|   2배 |   $O(\log{N})$    |   $O(N)$  |   최대 capacity의 50% |
|   1.5배   |   $O(\log_{1.5}N)$    |   $O(N)$  |   최대 capacity의 33% |

- 이론적 총 이동 비용은 둘 다 $O(N)$이지만, 1.5배는 메모리 낭비를 줄이는 대신 재할당 횟수가 약간 늘어남
- GCC(`libstdc++`)는 2배, MSVC는 1.5배 전략을 사용함

### 시각화
```
N = 8번의 push_back 시나리오

작업   │ size │ capacity │ 이동 비용 │ 누적 비용
───────┼──────┼──────────┼───────────┼──────────
pb(1)  │  1   │    1     │     0     │    1
pb(2)  │  2   │    2     │     1     │    3     ← capacity 1→2, 원소 1개 이동
pb(3)  │  3   │    4     │     2     │    6     ← capacity 2→4, 원소 2개 이동
pb(4)  │  4   │    4     │     0     │    7
pb(5)  │  5   │    8     │     4     │   12     ← capacity 4→8, 원소 4개 이동
pb(6)  │  6   │    8     │     0     │   13
pb(7)  │  7   │    8     │     0     │   14
pb(8)  │  8   │    8     │     0     │   15

평균 비용 = 15 / 8 ≈ 1.875 → O(1)
```

## `reserve`와 `shrink_to_fit`
### `reserve(n)` — 재할당 미리 방지
- 삽입할 원소 수를 미리 알고 있을 때 사용
- capacity만 늘리고 size는 그대로임
```cpp
std::vector<int> v;
v.reserve(1'000'000);       // Heap에 1,000,000개 분량 미리 확보

for (int i=0; i<1'000'000; i++){
    v.push_back(i);         // 재할당 0회, 모든 push_back이 O(1)
}
```
```
reserve 없이 push_back 100만 번:
재할당 횟수 = log₂(1,000,000) ≈ 20회
총 이동 비용 ≈ 1,000,000번의 원소 이동 발생

reserve(1,000,000) 후 push_back 100만 번:
재할당 횟수 = 0회
```

### `shrink_to_fit()` — 남는 capacity 반환
capacity를 size에 맞게 줄이는 요청(구현체가 무시할 수 있음)
```cpp
std::vector<int> v;
v.reserve(10000);

for (int i=0; i<10; i++)    v.push_back(i);

std::cout << v.size();      // 10
std::cout << v.capacity();  // 10000

v.shrink_to_fit();

std::cout << v.capacity();  // 10 (구현체마다 다를 수 있음)
```

## 구조체 멤버로 `vector`를 쓰면 느린 이유
- PS에서 "구조체 배열(Array of Structs)"에 `vector`를 멤버로 두는 패턴이 느린 이유를 두 가지 관점에서 분석함

### 문제 패턴
```cpp
// ❌ 느린 패턴: vector를 멤버로 갖는 구조체 배열
struct Node {
    int val;
    std::vector<int> children;      // vector가 멤버
};

Node graph[100001];
```

### 원인1 — 각 `vector`마다 별도의 Heap 할당 발생
```
graph 배열(스택 or BSS)
┌──────────────────┬──────────────────┬──────────────────┐
│  Node[0]         │  Node[1]         │  Node[2]         │
│  val: 1          │  val: 2          │  val: 3          │
│  vec: ptr ───────│──────────────────│──────── ptr ─────│──┐
│       size=3     │  vec: ptr ───────│──┐    size=2     │  │
│       cap=4      │       size=5     │  │    cap=4      │  │
└──────────────────┴──────────────────┴──│───────────────┘  │
                                         │                  │
Heap (분산된 위치)                        │                   │
  [Node[1]의 children 배열] ←────────────┘                   │
  [어딘가 다른 위치]                                          │
  [Node[2]의 children 배열] ←────────────────────────────────┘
```

### 원인2 — `vector` 객체 자체의 크기 오버헤드
```cpp
// vector 객체 하나의 크기
sizeof(std::vector<int>)    // 24 바이트 (포인터 3개: begin, end, end_cap)

// int 배열 포인터 하나의 크기
sizeof(int*)                // 8 바이트
```
- children이 없는 리프 노드도 24바이트를 점유함
- 구조체 배열 전체가 무거워져 캐시에 올라오는 노드 수가 감소

### 해결 패턴 — CSR (Compressed Sparse Row) 구조 또는 인접 배열
```cpp
// ✅ PS 실전 패턴: 인접 배열 + 범위 인덱스
// 모든 간선을 단일 배열에 저장, 노드별 시작 위치를 별도 관리

const int MAXN = 100001;
const int MAXE = 200001;

int head[MAXN];     // head[u]: u의 첫 번째 간선 인덱스
int to[MAXE];       // to[i]:   i번 간선의 도착 노드
int nxt[MAXE];      // nxt[i]:  i번 간선 다음 간선 인덱스 (연결 리스트)
int edge_cnt = 0;

void add_edge(int u, int v){
    to[++edge_cnt] = v;
    nxt[edge_cnt] = head[u];
    head[u] = edge_cnt;
}

// 노드 u의 인접 노드 순회
for (int e = head[u]; e; e = nxt[e]){
    int v = to[e];
    // ...
}
```
```
메모리 구조 비교

vector 멤버 방식              인접 배열 방식
─────────────────────         ─────────────────────────────────────
Node[0] ──→ Heap 어딘가        head[]: [0][3][5][7]...  (연속 배열)
Node[1] ──→ Heap 다른 곳       to[]:   [2][3][4][5][6]... (연속 배열)
Node[2] ──→ Heap 또 다른 곳    nxt[]:  [0][2][0][4][0]... (연속 배열)
... (분산)                     → 모든 데이터가 연속 메모리에 위치
```

### 성능 비교
|   항목    |   `vector` 멤버   |   인접 배열   |
|----------|-------------------|--------------|
|   동적 할당 횟수  |   노드 수 × N회   |   1회(`head`, `to`, `nxt` 선언)   |
|   메모리 분산 |   분산(캐시 미스 빈번)    |   연속(캐시 친화적)   |
|   구조체 크기 |   24바이트 오버헤드/노드  |   없음    |
|   구현 복잡도 |   낮음    |   중간    |
|   사용 적합 상황  |   간단한 코드, 소규모 |   PS 실전, 대규모 그래프  |

## 개념 연결 지도
```
vector 내부 동작
    │
    ├── 내부 구조
    │     ├── Heap 위의 연속 배열
    │     └── begin / end / end_cap 포인터 3개
    │
    ├── push_back
    │     ├── size < capacity → O(1) (빠른 경로)
    │     └── size == capacity → 재할당 → O(N) (느린 경로)
    │           ├── 새 배열 할당 (2배)
    │           ├── 이동 생성자로 원소 이전
    │           ├── 기존 배열 해제
    │           └── 포인터 무효화 주의
    │
    ├── 분할 상환 분석
    │     ├── 2배 확장 → 총 이동 비용 < 2N
    │     └── N번 push_back 평균 비용 = O(1)
    │
    ├── reserve(n)
    │     └── 재할당 횟수를 0으로 만들어 push_back을 항상 O(1)로 유지
    │
    └── 구조체 멤버 vector의 문제
          ├── 노드마다 별도 Heap 할당 → 단편화
          ├── 분산 메모리 → 캐시 미스
          └── 해결: 인접 배열(head/to/nxt) 또는 reserve 선활용
```
