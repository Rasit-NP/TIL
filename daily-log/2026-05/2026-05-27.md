## C++ 프로젝트를 Docker로 빌드하는 이유
- C++은 빌드 산출물이 호스트 환경에 강하게 종속됨
- 의존성 충돌 발생 요인
    - glibc 버전 차이로 인한 런타임 오류
    - 컴파일러 버전(gcc 9 vs. 11)에 따른 ABI 호환성 문제
    - 시스템 라이브러리(`libssl`, `libboost` 등)의 설치 경로 차이
- Docker로 얻는 효과
    - 빌드 환경을 이미지로 고정하여 재현 가능한 빌드 보장
    - CI/CD 파이프라인과 로컬 개발 환경의 결과물 일치
    - 배포 시 OS 패키지 의존성을 호스트에 설치할 필요 없음

## 단일 스테이지 Dockerfile의 한계
```dockerfile
FROM ubuntu:22.04

RUN apt-get update && apt-get install -y \
    build-essential cmake git \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY . .

RUN cmake -S . -B build -DCMAKE_BUILD_TYPE=Release \
    && cmake --build build -j

CMD ["./build/my_app"]
```
- 동작은 하지만 최종 이미지에 컴파일러, 빌드 도구, 소스 코드가 모두 포함됨
- 일반적으로 수백 MB ~ 1GB를 초과하는 이미지가 생성됨
- 보안 측면에서도 불필요한 도구가 공격 표면을 넓힘

## 멀티 스테이지 빌드 전략
- 빌드 환경(Build Stage)과 실행 환경(Runtime Stage)을 분리
- 최종 이미지에는 빌드 산출물(실행 파일과 필수 공유 라이브러리)만 포함
```dockerfile
# Stage 1: Build
FROM ubuntu:22.04 AS builder

RUN apt-get update && apt-get install -y \
    build-essential cmake ninja-build \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /src
COPY CMakeLists.txt ./
COPY src/ ./src/

RUN cmake -S . -B build -G Ninja \
    -DCMAKE_BUILD_TYPE=Release \
    && cmake --build build -j

# Stage 2: Runtime
FROM ubuntu:22.04 AS runtime

RUN apt-get update && apt-get install -y \
    libstdc++6 \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY --from=builder /src/build/my_app ./

CMD ["./my_app"]
```
- `AS builder`로 이름을 붙이면 `--from=builder`로 해당 스테이지의 산출물만 선택적으로 복사 가능
- 빌드 캐시·중간 오브젝트 파일·디버깅 심볼이 최종 이미지에서 제외됨

## 정적 링킹을 활용한 경량화
- 동적 링킹 시 런타임 스테이지에 호환 가능한 공유 라이브러리를 반드시 포함해야 함
- 정적 링킹을 적용하면 최소 베이스 이미지(distroless, alpine, scratch) 사용 가능
```cmake
# CMakeLists.txt
add_executable(my_app src/main.cpp)
target_link_options(my_app PRIVATE -static -static-libgcc -static-libstdc++)
```
```dockerfile
# Runtime Stage - 정적 링킹 시
FROM gcr.io/distroless/static-debian12

WORKDIR /app
COPY --from=builder /src/build/my_app ./

CMD ["./my_app"]
```
- distroless 이미지는 shell과 패키지 매니저가 없어 보안에 유리함
- 단, 디버깅 시 컨테이너 내부 접근이 어려워지므로 운영 환경 한정으로 사용 권장

## 빌드 캐시 최적화
- Docker는 레이어 단위로 캐시되므로 `COPY`와 `RUN`의 순서가 빌드 시간을 좌우함
- 자주 바뀌는 파일은 늦게 복사하는 것이 핵심 원칙
```dockerfile
FROM ubuntu:22.04 AS builder

RUN apt-get update && apt-get install -y \
    build-essential cmake ninja-build \
    && rm -rf /var/lib/apt/lists/*

WORKDIR /src

# 의존성 정의 파일만 먼저 복사
COPY CMakeLists.txt ./
COPY cmake/ ./cmake/

# 의존성 fetch 단계만 별도 캐시
RUN cmake -S . -B build -G Ninja \
    -DCMAKE_BUILD_TYPE=Release

# 소스 코드는 마지막에 복사
COPY src/ ./src/
RUN cmake --build build -j
```
- 소스 코드만 변경된 경우 의존성 다운로드 레이어는 캐시에서 재사용됨
- vcpkg, Conan, FetchContent를 사용하는 프로젝트일수록 캐시 분리 효과가 큼

## cache를 활용한 증분 빌드
- 컨테이너는 매번 클린 빌드를 수행하므로 빌드 시간이 길어짐
- BuildKit의 캐시 마운트를 활용하면 컨테이너 외부에 ccache 캐시를 영속화 가능
```dockerfile
# syntax=docker/dockerfile:1.4
FROM ubuntu:22.04 AS builder

RUN apt-get update && apt-get install -y \
    build-essential cmake ninja-build ccache \
    && rm -rf /var/lib/apt/lists/*

ENV CMAKE_CXX_COMPILER_LAUNCHER=ccache
ENV CCACHE_DIR=/ccache

WORKDIR /src
COPY . .

RUN --mount=type=cache,target=/ccache \
    cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release \
    && cmake --build build -j
```
- `--mount=type=cache`는 빌드 중에만 마운트되며 최종 이미지에는 포함되지 않음
- 두 번째 빌드부터는 변경되지 않은 컴파일 단위가 ccache에서 즉시 복원됨

## docker-compose를 활용한 개발 환경 구성
- 빌드 환경 자체를 컨테이너로 제공하여 팀원 간 환경 차이를 제거
```yaml
services:
  builder:
    build:
      context: .
      target: builder
    volumes:
      - .:/src
      - build-cache:/src/build
      - ccache:/ccache
    working_dir: /src
    command: cmake --build build -j

volumes:
  build-cache:
  ccache:
```
- `target: builder`로 멀티 스테이지 중 빌드 단계까지만 사용
- 소스 코드는 볼륨 마운트로 동기화하여 호스트에서 편집한 결과를 즉시 반영

## .dockerignore 작성
- 빌드 컨텍스트에 불필요한 파일을 제외하여 전송 시간 단축 및 캐시 무효화 방지
```
build/
cmake-build-*/
.git/
.vscode/
.idea/
*.user
CMakeCache.txt
CMakeFiles/
```
- 특히 로컬 `build/` 디렉토리를 제외하지 않으면 호스트의 빌드 산출물이 컨테이너에 복사되어 캐시가 매번 깨짐
