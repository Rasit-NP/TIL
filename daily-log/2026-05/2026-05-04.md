- Refresh Token 전달 및 저장 전략
    - JSON Body vs. HttpOnly Cookie

## 저장 전략
- JWT 인증 체계에서 Access Token은 수명이 짧고, Refresh Token은 수명이 길기 때문에 탈취되었을 때의 피해 규모가 전혀 다름
- Refresh Token을 어디에 어떻게 저장하느냐는 곧 공격 표면(attack surface)의 크기를 결정함
- 클라이언트 유형(SPA, 모바일, SSR 등)과 보안 요구 수준에 따라 전략이 달라지며, 잘못된 선택은 XSS-CSRF 취약점으로 직결됨

## 공격 벡터 기초 개념
### XSS(Cross-Site Scripting)
- 공격자가 악성 스크립트를 페이지에 주입하여 JavaScript가 실행되는 모든 저장소에 접근하는 공격
- `localStorage`, `sessionStorage`, JS로 읽히는 쿠키가 대상이 됨
- `HttpOnly` 플래그가 붙은 쿠키는 JavaScript에서 접근 불가능하므로 XSS로부터 안전함

### CSRF(Cross-Site Request Forgery)
- 인증된 사용자의 브라우저를 이용해 사용자 의지와 무관한 요청을 서버로 전송시키는 공격
- 쿠키는 브라우저가 요청에 자동으로 첨부하기 때문에, HttpOnly 쿠키도 CSRF에는 취약할 수 있음
- `SameSite` 속성과 CSRF 토큰으로 방어함

## 전략 1 : JSON Body로 전달 + localStorage/sessionStorage 저장
### 동작 방식
```
[클라이언트]                              [서버]
    |                                      |
    |-- POST /api/auth/login ------------->|
    |                                      |-- 인증 처리
    |<-- 200 OK { accessToken,           --| 
    |             refreshToken }           |
    |                                      |
    |-- localStorage.setItem("refresh"...) | (클라이언트가 직접 저장)
```

### 재발급 흐름
```
[클라이언트]                              [서버]
    |                                      |
    |-- POST /api/auth/refresh ----------->|
    |   { refreshToken: "..." }            |  ← JSON Body에 담아 전송
    |                                      |
    |<-- 200 OK { accessToken, ... } ------|
```

### 특징 요약
|   항목    |   내용    |
|----------|-----------|
|   XSS 취약 여부   |   취약 — JS로 `localStorage` 읽기 가능    |
|   CSRF 취약 여부  |   안전 — 서버가 자동으로 붙이지 않음  |
|   구현 난이도 |   낮음 — 클라이언트가 JS로 자유롭게 제어  |
|   적합한 환경 |   모바일 앱, React Native, Flutter, Electron  |

### 보안상 한계
- XSS 공격이 성공하면 `localStorage.getItem("refreshToken")`으로 즉시 탈취 가능
- 내부 JS 번들에 악성 코드가 삽입되거나, `eval()` 계열 함수가 악용될 수 있음
- SPA에서 npm 패키지 공급망 공격(supply chain attack)이 성공하면 전체 토큰 노출

## 전략 2 : HttpOnly Cookie로 전달 및 저장
### 동작 방식
```
[클라이언트]                              [서버]
    |                                      |
    |-- POST /api/auth/login ------------->|
    |                                      |-- 인증 처리
    |<-- 200 OK                          --|
    |   Set-Cookie: refresh=xxx;           |
    |     HttpOnly; Secure;                |
    |     SameSite=Strict; Path=/api/auth/refresh
    |                                      |
    | (브라우저가 쿠키를 자동으로 저장)         |
```

### 재발급 흐름
```
[클라이언트]                              [서버]
    |                                      |
    |-- POST /api/auth/refresh ----------->|
    |   Cookie: refresh=xxx                |  ← JS가 토큰 값을 모름
    |                                      |
    |<-- 200 OK (새 AccessToken 반환) ------|
    |   Set-Cookie: refresh=yyy (갱신)      |
```

### Spring Boot 구현 예시
#### 로그인 응답 시 쿠키 설정
```java
@PostMapping("/login")
public ResponseEntity<LoginResponse> login(
        @RequestBody LoginRequest request,
        HttpServletResponse response) {
    
    AuthToken tokens = authService.login(request);
    
    ResponseCookie refreshCookie = ResponseCookie.from("refreshToken", tokens.refreshToken())
            .httpOnly(true)             // JS 접근 차단
            .secure(true)               // HTTPS 전용
            .sameSite("Strict")         // 크로스사이트 요청 시 쿠키 미전송
            .path("/api/auth/refresh")  // Refresh 엔드포인트에만 전송
            .maxAge(Duration.ofDays(7))
            .build();
    
    response.addHeader(HttpHeaders.SET_COOKIE, refreshCookie.toString());
    
    // AccessToken만 Body에 담아 반환
    return ResponseEntity.ok(new LoginResponse(tokens.accessToken()));
}
```

#### 재발급 엔드포인트에서 쿠키 얻기

```java
import org.springframework.http.HttpHeaders;

@PostMapping("/refresh")
public ResponseEntity<TokenResponse> refresh(
        @CookieValue(name = "refreshToken", required = false) String refreshToken,
        HttpServletResponse response
) {
  if (refreshToken == null) {
    throw new CustomException(ErrorCode.REFRESH_TOKEN_NOT_FOUND);
  }

  AuthTokens newTokens = authService.reissue(refreshToken);

  // 새 RefreshToken으로 쿠키 갱신 (RTR 전략)
  ResponseCookie newCookie = ResponseCookie.from("refreshToken", newTokens.refreshToken())
          .httpOnly(true)
          .secure(true)
          .sameSite("Strict")
          .path("/api/auth/refresh")
          .maxAge(Duration.ofDays(7))
          .build();

  response.addHeader(HttpHeaders.SET_COOKIE, newCookie.toString());
  
  return ResponseEntity.ok(new TokenResponse(newTokens.accessToken()));
}
```

#### 로그아웃 시 쿠키 무효화
```java
@PostMapping("/logout")
public ResponseEntity<Void> logout(HttpServletResponse response){
    // 쿠키 maxAge를 0으로 설정하여 즉시 만료
    ResponseCookie expiredCookie = ResponseCookie.from("refreshToken", "")
            .httpOnly(true)
            .secure(true)
            .sameSite("Strict")
            .path("/api/auth/refresh")
            .maxAge(0)
            .build();
    
    response.addHeader(HttpHeaders.SET_COOKIE, expiredCookie.toString());
    return ResponseEntity.noContent().build();
}
```

### 특징 요약
|   항목    |   내용    |
|----------|-----------|
|   XSS 취약 여부   |   안전 — JS에서 쿠키 값 접근 불가 |
|   CSRF 취약 여부  |   조건부 안전 — `SameSite=Strict`/`Lax` + CSRF 토큰 병행 필요 |
|   구현 난이도 |   중간 — 서버 측 쿠키 설정 및 CORS 설정 필요  |
|   적합한 환경 |   웹 브라우저 기반 SPA, SSR 서비스    |

## 쿠키 속성 상세 분석
### Secure
- `Secure` 플래그가 설정된 쿠키는 HTTPS 연결에서만 전송됨
- HTTP에서 Refresh Token이 평문으로 전송되는 것을 방지함
- 개발 환경에서는 HTTP를 허용해야 할 수 있어 `localhost` 예외 처리가 필요함

### HttpOnly
- `HttpOnly` 플래그가 설정된 쿠키는 `document.cookie`로 접근 불가
- XSS 공격이 성공하더라도 Refresh Token 탈취 불가
- 단, HTTP 요청에는 자동 첨부되므로 CSRF로부터는 보호받지 못함

### SameSite
```
SameSite=Strict     →   크로스사이트 컨텍스트에서 쿠키를 절대 전송하지 않음
                        (다른 사이트에서의 링크 클릭으로 접근 시에도 미전송)

SameSite=Lax        →   최상위 탐색(Top-level navigation, GET 요청)에는 전송 허용
                        (POST, PUT 등 상태 변경 요청에는 미전송)

SameSite=None       →   크로스사이트에서도 전송 허용
                        (반드시 Secure 플래그와 함께 사용, iframe/OAuth 등 필요 시)
```
|   SameSite 값 |   CSRF 방어 수준  |   크로스 도메인 사용 가능 여부    |
|---------------|-----------------|-------------------------------|
|   `Strict`    |   강함    |   불가    |
|   `Lax`   |   중간    |   부분 허용(GET 한정) |
|   `None` + `Secure`   |   없음    |   가능    |

### Path 제한
- `Path=/api/auth/refresh`로 설정하면 해당 경로에 요청할 때만 쿠키가 전송됨
- `/api/data`같은 일반 API 요청에는 쿠키가 첨부되지 않아 불필요한 노출을 최소화함

## 전략 종합 비교
|   비교 항목   |   JSON Body + localStorage    |   HttpOnly Cookie |
|-------------|--------------------------------|------------------|
|   XSS 방어    |   ❌ 취약 |   ✅ 안전 |
|   CSRF 방어   |   ✅ 안전 |   ⚠️ SameSite 설정 필요   |
|   구현 복잡도 |   낮음    |   중간    |
|   크로스 도메인   |   용이    |   제한적(`SameSite=None`) 필요    |
|   모바일 앱 지원  |   ✅ 적합 |   ⚠️ WebView 의존 |
|   브라우저 자동 처리  |   ❌ JS 직접 관리 |   ✅ 브라우저가 자동 관리 |
|   토큰 만료 후 갱신 처리  |   클라이언트 로직 필요    |   서버 Set-Cookie로 자동 갱신 |

## RTR(Refresh Token Rotation) 전략
### 개념
- Refresh Token을 1회 사용 후 즉시 폐기하고 새 토큰으로 교체하는 방식
- 탈취된 Refresh Token이 사용되는 순간, 기존 사용자의 재발급 요청이 실패하여 탈취 사실을 감지할 수 있음

### 흐름
```
[정상 클라이언트]                         [서버 (Redis)]
    |                                         |
    |-- POST /refresh (RT: "token-A") ------->|
    |                                         |-- "token-A" 유효 확인
    |                                         |-- "token-A" 즉시 무효화
    |                                         |-- 새 "token-B" 생성 및 저장
    |<-- 200 OK { AT: new, RT: "token-B" } ---|

(만약 공격자가 탈취한 token-A를 나중에 사용하면)
[공격자]                                  [서버 (Redis)]
    |                                         |
    |-- POST /refresh (RT: "token-A") ------->|
    |                                         |-- "token-A" 없음 → 이미 폐기됨
    |                                         |-- 이상 감지 → 전체 세션 무효화 가능
    |<-- 401 Unauthorized --------------------|
```

### Redis를 활용한 RTR 구현 예시
```java
@Service
@RequiredArgsConstructor
public class RefreshTokenService {

    private final RedisTemplate<String, String> redisTemplate;
    private static final String PREFIX = "refresh:";

    public void save(Long userId, String refreshToken, Duration ttl) {
        redisTemplate.opsForValue()
            .set(PREFIX + userId, refreshToken, ttl);
    }

    // RTR: 사용 시 즉시 교체
    public String rotate(Long userId, String incomingToken) {
        String stored = redisTemplate.opsForValue().get(PREFIX + userId);

        if (stored == null || !stored.equals(incomingToken)) {
            // 이미 사용된 토큰 또는 탈취 감지
            redisTemplate.delete(PREFIX + userId); // 모든 세션 강제 만료
            throw new CustomException(ErrorCode.INVALID_REFRESH_TOKEN);
        }

        // 유효하면 새 토큰 생성 후 교체 저장
        String newToken = generateRefreshToken(userId);
        redisTemplate.opsForValue()
            .set(PREFIX + userId, newToken, Duration.ofDays(7));

        return newToken;
    }

    public void delete(Long userId) {
        redisTemplate.delete(PREFIX + userId);
    }
}
```

## 크로스 도메인 환경에서의 고려사항
### SPA(api.example.com) + 백엔드(api.example.com) — 동일 도메인
- `SameSite=Strict` 사용 가능
- 별도 CORS 설정 불필요
- 가장 단순하고 안전한 구성

### 프론트(app.example.com) + 백엔드(api.example.com) — 서브도메인 분리
- `SameSite=Lax` 또는 `SameSite=None; Secure` 필요
- `Domain=.example.com`으로 설정하면 서브 도메인 간 쿠키 공유 가능
- CORS `allowCredentials(true)` + `allowedOrigins` 명시 필수
```java
@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://app.example.com")
            .allowedMethods("GET", "POST", "PUT", "DELETE")
            .allowCredentials(true)  // 쿠키 전송 허용
            .maxAge(3600);
    }
}
```
> `allowCredentials(true)` 설정 시 `allowedOrigins("*")`는 사용 불가
>
> 반드시 도메인을 명시해야 함

### 완전히 다른 도메인 — `SameSite=None; Secure`
- OAuth, 서드파티 통합, 마이크로프론트엔드에서 발생하는 케이스
- 브라우저의 Third-party cookie 제한 정책(Safari ITP, Chrome Privacy Sandbox)으로 향후 지원이 제한될 수 있음
- 가능하면 동일 도메인 구조로 아키텍처를 변경하는 것을 권장

## 실무 권장 전략
```
브라우저 기반 SPA
    └─ HttpOnly Cookie (SameSite=Strict + Secure + Path 제한)
    └─ RTR 적용 + Redis 저장
    └─ Access Token은 메모리(변수)에만 보관 (localStorage 사용 금지)

모바일 앱 (iOS, Android, React Native)
    └─ JSON Body 전달 + Secure Storage 저장
       (iOS: Keychain, Android: EncryptedSharedPreferences)
    └─ localStorage는 절대 사용 금지

크로스 도메인이 불가피한 경우
    └─ BFF(Backend For Frontend) 패턴 도입을 검토
       → BFF가 HttpOnly Cookie를 관리하고, 프론트와는 세션 기반 통신
```

### Access Token 저장 위치 원칙
- 메모리(JS 변수/상태)에만 저장하는 것이 가장 안전
- 페이지 새로고침 시 소멸되므로, `/api/auth/refresh/` 호출로 재발급받는 silent refresh 구현이 필요함
- `localStorage`에 Access Token을 저장하는 것은 XSS에 그대로 노출되므로 지양
```
[앱 초기화]
  └─ /api/auth/refresh 호출 (HttpOnly 쿠키 자동 첨부)
  └─ 새 AccessToken 수신 → 메모리 변수에 저장
  └─ 이후 API 호출 시 Authorization 헤더에 첨부

[AccessToken 만료 시 (401 응답)]
  └─ 자동으로 /api/auth/refresh 재호출
  └─ 새 AccessToken 발급 → 원래 요청 재시도
```
