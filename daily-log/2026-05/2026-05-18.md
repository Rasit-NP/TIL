## GoogleTest
- Google이 개발한 C++ 단위 테스트 프레임워크
- xUnit 아키텍처를 따르며, 테스트 작성과 실행, 결과 보고를 표준화된 방식으로 제공함
- Header-only가 아닌 라이브러리 형태이므로 빌드 시스템에 통합이 필요함
- 풍부한 단언(assertion) 매크로와 테스트 픽스처(fixture), 모킹 라이브러리(GoogleMock)까지 제공함
- 핵심 특징
    - 테스트 자동 등록(매크로 기반)
    - 풍부한 단언 매크로
    - 테스트 픽스처를 통한 공통 셋업/정리
    - 파라미터화 테스트, 타입 파라미터화 테스트 지원
    - CMake와의 통합이 잘 되어 있음

## 프로젝트에 GoogleTest 추가하기
- 권장 방식은 CMake의 `FetchContent`를 이용해 소스를 직접 가져와 빌드에 포함하는 방식
- 시스템 설치 의존성 없이 재현 가능한 빌드를 보장함
```cmake
cmake_minimum_required(VERSION 3.14)
project(my_project CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# GoogleTest 가져오기
include(FetchContent)
FetchContent_Declare(
    googletest
    URL https://github.com/google/googletest/archive/refs/tags/v1.14.0.zip
)
# Windows에서 CRT 충돌 방지
set(gtest_force_shared_crt ON CACHE BOOL "" FORCE)
FetchContent_MakeAvailable(googletest)

enable_testing()

# 테스트 실행 파일 정의
add_executable(my_tests
    tests/calculator_test.cpp
)

target_link_libraries(my_tests
    PRIVATE
        GTest::gtest_main       # main 함수 포함된 진입점
)

# CTest 등록
include(GoogleTest)
gtest_discover_tests(my_tests)
```
> `GTest::gtest_main`을 링크하면 별도의 `main()`을 작성하지 않아도 됨
>
> 직접 `main()`을 정의하고 싶다면 `GTest::gtest`만 링크할 것

### 디렉토리 구성 예시
```
my_project/
├── CMakeLists.txt
├── src/
│   ├── calculator.cpp
│   └── calculator.h
└── tests/
    └── calculator_test.cpp
```

## 기본 테스트 작성
- 가장 단순한 단위는 `TEST(TestSuiteName, TestName)` 매크로
- 첫 번째 인자는 테스트 그룹(스위트) 이름, 두 번째 인자는 개별 테스트 이름
- 두 이름의 조합은 전체 빌드에서 유일해야 함
```cpp
// calculator.h
int add(int a, int b);
int divide(int a, int b);
```
```cpp
// calculator_test.cpp
# include <gtest/gtest.h>
# include "calculator.h"

TEST(CalculatorTest, AddPositiveNumbers){
    EXPECT_EQ(add(2, 3), 5);
    EXPECT_EQ(add(10, 20), 30);
}

TEST(CalculatorTest, AddNegativeNumbers){
    EXPECT_EQ(add(-1, -1), -2);
    EXPECT_EQ(add(-5, 3), -2);
}
```
- 테스트 함수는 자동으로 전역 테스트 레지스트리에 등록됨
- 명시적인 호출 없이 `main`에서 한 번에 실행됨

## ASSERT vs. EXPECT
- 단언 매크로는 두 계열로 나뉨
- 두 계열의 차이는 실패 시 동작 방식에 있음

|   구분    | 실패 시 동작  |   사용 시점   |
|----------|-------------|---------------|
|   `ASSERT_*`  |   현재 테스트를 즉시 중단 |   이후 코드가 의미 없거나 크래시를 유발할 때  |
|   `EXPECT_*`  |   실패를 기록하되 계속 진행   |   가능한 한 많은 정보를 수집하고 싶을 때  |

```cpp
TEST(VectorTest, NotEmpty){
    std::vector<int> v = {1, 2, 3};

    // 비어있으면 v[0] 접근이 의미 없으므로 ASSERT로 즉시 중단
    ASSERT_FALSE(v.empty());

    // 여기서부터는 EXPECT로 여러 조건을 모두 검사
    EXPECT_EQ(v.size(), 3);
    EXPECT_EQ(v[0], 1);
    EXPECT_EQ(v[2], 3);
}
```
> 기본은 `EXPECT_*`로 작성하고, 후속 코드가 위험할 때만 `ASSERT_*`를 사용하는 것이 일반적인 가이드라인

### 주요 단위 매크로
|   매크로  |   의미    |
|----------|----------|
|   `EXPECT_EQ(a, b)`   |   `a == b`    |
|   `EXPECT_NE(a, b)`   |   `a != b`    |
|   `EXPECT_LT/LE/GT/GE`    |   대소 비교   |
|   `EXPECT_TRUE(cond)`/`EXPECT_FALSE(cond)`    |   bool 검사   |
|   `EXPECT_STREQ(a, b)`    |   C-style 문자열 동등 비교    |
|   `EXPECT_FLOAT_EQ(a, b)`/`EXPECT_DOUBLE_EQ(a, b)`    |   부동소수점 근사 비교    |
|   `EXPECT_NEAR(a, b, abs_err)`    |   지정한 오차 범위 내 비교    |
|   `EXPECT_THROW(stmt, ex)`    |   지정한 예외 발생 여부   |

> 부동소수점은 `EXPECT_EQ` 대신 `EXPECT_DOUBLE_EQ` 또는 `EXPECT_NEAR`를 사용해야 함
>
> IEEE 754의 정밀도 한계로 직접적인 동등 비교는 불안정함

## 테스트 픽스처(Test Fixture)
- 여러 테스트에서 공통적인 셋업/정리 로직을 공유하기 위한 클래스 기반 구조
- `::testing::Test`를 상속받아 정의하고, `TEST_F` 매크로로 테스트를 작성함
- 각 테스트마다 픽스처 인스턴스가 새로 생성됨
    - 즉, 테스트 간에 상태가 공유되지 않음

```cpp
# include <gtest/gtest.h>
# include "user_repository.h"

class UserRepositoryTest : public ::testing::Test{
protected:
    // 각 TEST_F 실행 전 호출됨
    void SetUp() override{
        repo = std::make_unique<UserRepository>();
        repo->add({1, "Alice"});
        repo->add({2, "Bob"});
    }

    // 각 TEST_F 실행 후 호출됨
    void TearDown() override{
        repo.reset();
    }

    std::unique_ptr<UserRepository> repo;
};

TEST_F(UserRepositoryTest, FindExistingUser){
    auto user = repo->find(1);
    ASSERT_TRUE(user.has_value());
    EXPECT_EQ(user->name, "Alice");
}

TEST_F(UserRepositoryTest, FindNonExistingUser){
    auto user = repo->find(999);
    EXPECT_FALSE(user.has_value());
}
```
> `TEST` 대신 `TEST_F`를 사용해야 픽스처 멤버에 접근 가능
>
> 첫 번째 인자는 픽스처 클래스 이름과 일치해야 함

### `SetUpTestSuite`/`TearDownTestSuite`
- 전체 테스트 스위트 단위로 한 번만 실행되는 셋업/정리가 필요할 때 사용
- DB 커넥션, 외부 리소스 초기화 등 비용이 큰 작업에 적합
- 단, 테스트 간 격리가 깨질 수 있으므로 신중하게 사용해야 함
```cpp
class DatabaseTest : public ::testing::Test{
protected:
    static void SetUpTestSuite(){
        // 전체 스위트 시작 전 1회만 실행
        db_connection = connect_to_test_db();
    }

    static void TearDownTestSuite(){
        db_connection->close();
    }

    static DbConnection* db_connection;
};
```

## 테스트 실행과 필터링
- 빌드된 테스트 바이너리를 직접 실행할 수도 있고, CTest를 통해 실행할 수도 있음
```bash
# CMake 빌드 후 직접 실행
./my_tests

# 특정 패턴의 테스트만 실행
.my_tests --gtest_filter=CalculatorTest.*

# 특정 테스트 제외
./my_tests --gtest_filter=-*Slow*

# 실패한 테스트만 다시 실행
./my_tests --gtest_filter=CalculatorTest.AddNegativeNumbers

# CTest로 실행 (CMake 통합 시)
ctest --output-on-failure
```

## 주의점
### Death와 멀티스레드
- 예외 종료를 검증하는 `ASSERT_DEATH`는 fork 기반으로 동작함
- 멀티스레드 환경에서는 동작이 불안정할 수 있어 사용을 신중히 결정해야 함

### 테스트는 결정론적이어야 함
- 시간, 난수, 외부 네트워크 등 비결정적 요소에 직접 의존하면 안됨
- 가능한 한 의존성을 추상화하고, 테스트에서는 가짜 객체(fake/mock)로 대체할 것

### 한 테스트에는 한 가지 행위만 검증
- 한 `TEST` 안에서 너무 많은 시나리오를 다루면 실패 원인 파악이 어려워짐
- 시나리오별로 테스트를 분리하는 것이 디버깅과 회귀 추적에 유리함
