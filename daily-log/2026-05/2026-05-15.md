## 하이브리드 암호화
- 대칭키 암호화의 속도와 비대칭키 암호화의 안전한 키 전달을 결합한 방식
- 본문은 빠른 AES로 암호화하고, 그 AES 키 자체는 RSA로 암호화해 전달함
- TLS 1.3, PGP, JWE, AWS KMS 봉투 암호화 등 현대 보안 시스템의 사실상 표준 구조

### 등장 배경
- 대칭키 단독 사용
    - 빠르고 효율적이지만, 키를 어떻게 안전하게 상대에게 전달할지가 문제(Key Distribution Problem)
- 비대칭키 단독 사용
    - 키 분배 문제는 없지만 RSA-2048이 AES-256보다 약 1000배 이상 느림
    - 한 번에 암호화 가능한 평문 크기에도 제한이 있음(RSA-2048 + OAEP-SHA256 기준 약 190바이트)
- 해결책
    - "데이터는 빠른 대칭키로, 그 대칭키는 안전한 비대칭키로"
    - 짧은 대칭키만 RSA로 보호하면 RSA의 크기 제한과 속도 문제 모두 해결

## 동작 원리
```
[송신자]                                        [수신자]
   │                                              │
   ├─ ① 1회용 AES 세션 키 K 생성                    │
   ├─ ② AES-GCM(K, IV)로 본문 암호화                │
   ├─ ③ RSA-OAEP(수신자 공개키)로 K 암호화           │
   │                                              │
   │── [EncKey || IV || Ciphertext || Tag] ──────▶│
   │                                              │
   │                                              ├─ ④ 개인키로 EncKey 복호화 → K 복원
   │                                              ├─ ⑤ K + IV + Tag로 본문 복호화 + 무결성 검증
   │                                              ▼
                                              [평문 복원]
```

### 보안 속성 정리
|   속성    |   보장 수단   |
|----------|--------------|
|   기밀성(Confidentiality) |   AES-GCM 암호문  |
|   무결성(Integrity)   |   GCM의 인증 태그(Auth Tag)   |
|   인증(Authenticity) - 송신자 확인    |   별도 디지털 서명 필요(RSA-PSS 등)   |
|   키 보호 |   RSA-OAEP로 세션 키 자체 암호화  |

> 하이브리드 암호화 자체는 "이 메시지를 누가 보냈는지" 보장하지 않음
>
> 송신자 인증이 필요하면 별도의 디지털 서명을 함께 적용해야 함

## AES-GCM
- AES(Advanced Encryption Standard)를 GCM(Galois/Counter Mode) 운용 모드로 사용한 것
- **AEAD(Authenticated Encryption with Associated Data)**에 속함
    - 한 번의 연산으로 암호화와 무결성 검증을 모두 수행
- 내부적으로 CTR 모드 기반이라 병렬 처리가 가능 → 매우 빠름

### 운용 모드 비교
|   모드    |   특징    |   추천 여부   |
|----------|-----------|--------------|
|   ECB |   각 블록을 독립적으로 암호화 |   ❌ -- 같은 평문 = 같은 암호문, 패턴 노출    |
|   CBC |   이전 블록과 XOR 후 암호화   |   ⚠️ -- 무결성 별도 필요, 패딩 오라클 공격    |
|   CTR |   카운터를 암호화해 키 스트림 생성    |   ⚠️ -- 무결성 별도 필요  |
|   GCM |   CTR + GHASH 인증    |   ✅ -- 암호화 + 무결성 동시 보장 |

### 입출력 파라미터
|   파라미터    |   설명    |
|--------------|-----------|
|   Key |   16/24/32 바이트 대칭키  |
|   IV(Nonce)   |   12바이트 권장, 재사용 금지  |
|   AAD(Additional Auth Data)   |   암호화하지 않고 무결성만 보호할 메타데이터  |
|   Auth Tag    |   16바이트 인증 태그(복호화 시 자동 검증) |
> GCM에서 같은 키 + 같은 IV로 두 번 암호화하면 치명적임
>
> 키 스트림이 노출되어 두 평문의 XOR이 그대로 드러남 → 사실상 평문 복원 가능
>
> IV는 매 암호화마다 새로운 난수로 생성해야 함

### Java 구현
```java
public class AesGcmExample{

    private static final int GCM_IV_LENGTH = 12;        // 12바이트 권장
    private static final int GCM_TAG_LENGTH = 128;      // 비트 단위

    public static byte[] encrypt(byte[] plaintext, SecretKey key, byte[] iv) throws Exception{
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        GCMParameterSpec spec = new GCMParameterSpec(GCM_TAG_LENGTH, iv);
        cipher.init(Cipher.ENCRYPT_MODE, key, spec);

        // 결과: [Ciphertext || AuthTag]
        return cipher.doFinal(plaintext);
    }

    public static byte[] decrypt(byte[] ciphertext, SecretKey key, byte[] iv) throws Exception{
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        GCMParameterSpec spec = new GCMParameterSpec(GCM_TAG_LENGTH, iv);
        cipher.init(Cipher.DECRYPT_MODE, key, spec);

        // 태그가 일치하지 않으면 AEADBadTagException 발생
        return cipher.doFinal(ciphertext);
    }

    // IV는 반드시 매번 새로 생성
    public static byte[] generateIv(){
        byte[] iv = new byte[GCM_IV_LENGTH];
        new SecureRandom().nextBytes(iv);
        return iv;
    }

    public static SecretKey generateAesKey() throws Exception{
        KeyGenerator keyGen = KeyGenerator.getInstance("AES");
        keyGen.init(256);
        return keyGen.generateKey();
    }
}
```

## RSA-OAEP
- RSA 비대칭키 암호화에 OAEP(Optimal Asymmetric Encryption Padding) 패딩을 적용한 방식
- 단순 RSA(텍스트북 RSA)는 결정론적이라 같은 평문이면 같은 암호문이 나옴 → 사전 공격에 취약
- OAEP는 평문에 **무작위 시드 + 해시 기반 마스킹(MGF1)**을 추가해 매번 다른 암호문이 나오게 함

### 키 길이 권장
|   키 길이 |   보안 수준(대칭키 환산)  |   상태    |
|----------|------------------------|----------|
|   RSA-1024    |   ~80bit  |   ❌ 사용 금지    |
|   RSA-2048    |   ~112bit |   ✅ 현재 표준    |
|   RSA-3072    |   ~128bit |   ✅ 장기 사용 권장   |
|   RSA-4096    |   ~152bit |   ✅ 매우 안전(비용 큼)   |

### PKCS#1 v1.5 vs. OAEP
|   항목    |   PKCS#1 v1.5 |   OAEP    |
|----------|---------------|-----------|
|   표준    |   구버전(RFC 2313)    |   현행 권장(RFC 8017) |
|   무작위성    |   부분적  |   해시 + MGF 기반 충분    |
|   공격 저항   |   Bleichenbacher 공격 취약    |   안전    |
|   권장 여부   |   ❌ 신규 시스템에서 금지 |   ✅  |

### 평문 크기 제한
- RSA-OAEP에서 한 번에 암호화 가능한 평문 최대 크기 = `keySize - 2*hashSize - 2`(바이트)
- RSA-2048 + SHA-256 기준 : 256-64-2=190 바이트
- AES 키(32바이트) + IV(12바이트) 정도를 감싸기엔 충분
    - 하이브리드 암호화에서 비대칭키 부분이 세션 키 전용이어야 하는 또 다른 이유

### Java 구현
```java
public class RsaOaepExample{
    
    public static KeyPair generateRsaKeyPair() throws Exception {
        KeyPairGenerator gen = KeyPairGenerator.getInstance("RSA");
        gen.initialize(2048);
        return gen.generateKeyPair();
    }

    public static byte[] encryptKey(byte[] aesKeyBytes, PublicKey publicKey) throws Exception {
        Cipher cipher = Cipher.getInstance("RSA/ECB/OAEPWithSHA-256AndMGF1Padding");

        OAEPParameterSpec oaepParams = new OAEPParameterSpec(
            "SHA-256",                      // 메인 해시
            "MGF1",                         // 마스크 생성 함수
            MGF1ParameterSpec.SHA256,       // MGF1 내부 해시
            PSource.PSpecified.DEFAULT
        );

        cipher.init(Cipher.ENCRYPT_MODE, publicKey, oaepParams);
        return cipher.doFinal(aesKeyBytes);
    }

    public static byte[] decryptKey(byte[] encryptedKey, PrivateKey privateKey) throws Exception{
        Cipher cipher = Cipher.getInstance("RSA/ECB/OAEPWithSHA-256AndMGF1Padding");

        OAEPParameterSpec oaepParams = new OAEPParameterSpec(
            "SHA-256", "MGF1", MGF1ParameterSpec.SHA256, PSource.PSpecified.DEFAULT
        );

        cipher.init(Cipher.DECRYPT_MODE, privateKey, oaepParams);
        return cipher.doFinal(encryptedKey);
    }
}
```
> `"RSA/ECB/OAEPWithSHA-256AndMGF1Padding"`에서 ECB는 운용 모드가 아니라 JCA 문자열의 관례적 표기
>
> 실제로는 RSA가 단일 블록으로 동작하므로 운용 모드 개념이 적용되지 않음

## 전체 통합 예시
```java
public class HybridEncryption{

    // 송신자 측 -- 평문을 암호화해 패키지 생성
    public static EncryptedPackage encrypt(byte[] plaintext, PublicKey receiverPublicKey) throws Exception{

        // 1. 1회용 AES-256 세션 키 생성
        SecretKey sessionKey = AesGcmExample.generateAesKey();

        // 2. IV 생성 (매번 새로 생성)
        byte[] iv = AesGcmExample.generateIv();

        // 3. AES-GCM으로 본문 암호화
        byte[] ciphertext = AesGcmExample.encrypt(plaintext, sessionKey, iv);

        // 4. RSA-OAEP로 세션 키 자체를 암호화
        byte[] encryptedKey = RsaOaepExample.encryptKey(
            sessionKey.getEncoded(),
            receiverPublicKey
        );

        return new EncryptedPackage(encryptedKey, iv, ciphertext);
    }

    // 수신자 측 -- 패키지를 복호화해 평문 복원
    public static byte[] decrypt(EncryptedPackage pkg, PrivateKey receiverPrivateKey) throws Exception {

        // 1. 자신의 개인키로 세션 키 복원
        byte[] aesKeyBytes = RsaOaepExample.decryptKey(
            pkg.encryptedKey(),
            receiverPrivateKey
        );
        SecretKey sessionKey = new SecretKeySpec(aesKeyBytes, "AES");

        // 2. 복원한 세션 키 + IV로 본문 복호화 (태그 자동 검증)
        return AesGcmExample.decrypt(pkg.ciphertext(), sessionKey, pkg.iv());
    }

    public record EncryptedPackage(
        byte[] encryptedKey,    // RSA-OAEP로 암호화된 세션 키
        byte[] iv,              // AES-GCM IV
        byte[] ciphertext       // AES-GCM 암호문 + 태그
    ) {}
}
```

## 실무에서의 응용
- TLS 1.3 Handshake: (EC)DHE로 세션 키 교환 → AES-GCM(또는 ChaCha20-Poly1305)으로 본문 암호화
- AWS KMS Envelope Encryption: 1회용 DEK(Data Ecryption Key)로 본문을 AES-GCM 암호화, DEK는 KMS 마스터 키로 암호화해 함께 저장
- JWE(JSON Web Encryption): 헤더의 `alg`에 키 암호화 방식(`RSA-OAEP-256` 등), `enc`에 본문 암호화 방식(`A256GCM` 등) 명시

## 주의사항 및 안티 패턴
- 직접 구현하지 말 것
    - 암호학은 검증된 표준 라이브러리(JCA, BouncyCastle, libsodium 등)를 사용해야 함
    - IV 재사용, 패딩 처리, 타이밍 공격 방어 등 사소한 실수가 치명적인 취약점이 됨
- IV(Nonce)는 매 암호화마다 새로 생성
    - 같은 키 + 같은 IV로 두 번 암호화 = 보안 붕괴
    - IV는 비밀일 필요는 없으므로 그대로 전송해도 됨
- 세션 키는 1회용
    - 매번 새로운 세션 키를 생성해야 Forward Secrecy에 가까운 효과를 얻음
    - 과거 한 번의 키가 노출되어도 다른 통신은 영향받지 않음
- 가능하다면 ECDHE + AES-GCM 조합 우선 고려
    - TLS 1.3은 RSA 키 교환을 제거하고 (EC)DHE 기반 키 교환만 허용
    - 매 세션마다 임시 키쌍을 만들어 진정한 Forward Secrecy를 보장하기 때문
    - RSA 기반 하이브리드는 PGP, JWE 등 일부 영역과 레거시 시스템에서 여전히 유효�
