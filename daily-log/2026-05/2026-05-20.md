## 메모리 계층 구도
- CPU는 RAM에서 데이터를 직접 읽지 않음
- 속도 차이를 극복하기 위해 여러 단계의 캐시를 계층적으로 사용함
```
CPU 코어
CPU 코어
 │
 ├── 레지스터         ~1 사이클    수십 B
 ├── L1 캐시         ~4 사이클    32 ~ 64 KB
 ├── L2 캐시         ~12 사이클   256 KB ~ 1 MB
 ├── L3 캐시         ~40 사이클   수 MB (코어 공유)
 │
RAM                  ~200 사이클  수십 GB
```
- L1에서 L3로 갈수록 느리고 크며, RAM은 캐시보다 수십 배 느림
- 캐시에 없는 데이터를 RAM에서 가져오는 것을 캐시 미스(Cache Miss)라고 함
- 캐시에서 바로 찾는 것을 캐시 히트(Cache Hit)라고 함

## 캐시 라인(Cache Line)
- CPU는 데이터를 1바이트씩 가져오지 않음
- 64바이트 단위(캐시 라인)로 한 번에 가져옴
```
RAM에서 int a[16] 중 a[0] 하나를 요청했을 때

캐시 라인 (64바이트 = int 16개)
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│ a[0] │ a[1] │ a[2] │ a[3] │ a[4] │ a[5] │ a[6] │ a[7] │  ...
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
   ↑
요청한 원소       나머지도 캐시에 함께 올라옴

→ a[1], a[2], ... a[15] 접근 시 캐시 히트
```
- 연속된 메모리를 순서대로 접근하면 한 번의 캐시 미스로 여러 원소를 캐시에 올릴 수 있음
- 이 성질을 공간 지역성(Spatial Locality)이라고 함

## Cache Locality(캐시 지역성)
- 캐시를 효율적으로 사용하기 위한 두 가지 접근 패턴

### 공간 지역성(Spatial Locality)
- 한 주소를 접근하면, 인접한 주소도 곧 접근할 가능성이 높다는 성질
```cpp
// 공간 지역성을 활용하는 코드
int arr[10000];
long long sum = 0;
for (int i=0; i<10000; i++){
    sum += arr[i];      // 연속 주소를 순서대로 접근 → 캐시 히트 연속
}
```

### 시간 지역성(Temporal Locality)
- 한 번 접근한 데이터는 가까운 미래에 다시 접근할 가능성이 높다는 성질
```cpp
// 시간 지역성을 활용하는 코드
int count = 0;          // 루프마다 count에 접근 → 레지스터/L1에 계속 유지
for (int i=0; i<10000; i++){
    count += arr[i];
}
```

## `vector<vector<int>>` vs. 2D 배열
- 캐시 지역성의 차이가 실제 성능에 미치는 영향을 잘 보여주는 대표 사례

### `vector<vector<int>>`의 메모리 구조
```
vector<vector<int>> mat (스택/Heap)
┌────────┐
│ mat[0] │ ──→  Heap 어딘가: [1][2][3][4]
├────────┤
│ mat[1] │ ──→  Heap 다른 곳: [5][6][7][8]
├────────┤
│ mat[2] │ ──→  Heap 또 다른 곳: [9][10][11][12]
└────────┘

각 행이 Heap의 서로 다른 위치에 분산됨
→ mat[0][3] 다음에 mat[1][0]을 읽으면 캐시 미스
```

### 2D 배열의 메모리 구조
```
int mat[3][4] (전역/스택)
┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
│[0][0]│[0][1]│[0][2]│[0][3]│[1][0]│[1][1]│[1][2]│[1][3]│[2][0]│[2][1]│[2][2]│[2][3]│
└──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
    ↑───────────────────────── 연속된 단일 메모리 블록 ─────────────────────────────↑

→ mat[0][3] 다음 mat[1][0]도 이미 캐시에 올라와 있음 → 캐시 히트
```

### 행 우선 순회 vs. 열 우선 순회
- 2D 배열에서 순회 순서만 바꿔도 성능 차이가 발생함
```cpp
int mat[1000][1000];

// 행 우선 순회 (Row-Major) — C++에서 올바른 방향
for (int i=0; i<1000; i++){
    for (int j=0; j<1000; j++){
        sum += mat[i][j]        // 메모리 순서대로 접근 → 캐시 히트 연속
    }
}

// 열 우선 순회 (Column-Major) — 캐시를 역행하는 방향
for (int j=0; j<1000; j++){
    for (int i=0; i<1000; i++){
        sum += mat[i][j];       // 1000칸씩 건너뛰며 접근 → 캐시 미스 연속
    }
}
```
> 행렬 곱셈에서 단순한 3종 루프 행렬 곱셈(`i-j-k` 순서)은 내부 루프에서 `B[k][i]`를 열 방향으로 접근하므로 캐시 미스가 빈번하게 발생함
>
> `i-k-j` 순서로 루프를 바꾸면 같은 $O(N^3)$ 알고리즘이라도 실제 속도가 수 배 차이날 수 있음

## Big-O에서의 상수 계수
- Big-O는 입력 크기 N이 커질 때의 점근적 성장률만 나타냄
- 상수 계수와 낮은 차수 항을 제거하기 때문에, 같은 O 표기라도 실제 속도는 크게 다를 수 있음

### 실제 비용 함수 비교
```
알고리즘 A: T(N) = 1000N      → O(N)
알고리즘 B: T(N) = 2N         → O(N)
알고리즘 C: T(N) = N log N    → O(N log N)

N = 1,000,000 일 때:
  A: 1,000,000,000 (10억 연산)
  B: 2,000,000     (200만 연산)
  C: 20,000,000    (2천만 연산)

→ B < C < A 순서로 빠름
→ O(N log N)인 C가 O(N)인 A보다 빠름
```

### 캐시 미스가 상수 계수를 키우는 방식
```
vector<vector<int>> 순회:
  실제 비용 = 캐시 미스 패널티(~200 사이클) × N × M
            = 200 * N * M
            → O(NM) 이지만 상수 계수 200

int arr[N][M] 순회:
  실제 비용 = 캐시 히트(~4 사이클) × N × M
            = 4 * N * M
            → O(NM) 이지만 상수 계수 4

→ 복잡도는 동일하나 실제 속도 차이 50배 가능
```

### 알고리즘 선택 시 상수 계수를 고려해야 하는 경우
|   상황    |   현상    |   원인    |
|   `new`/`delete` 반복 |   $O(1)$인데 느림   |   free list 탐색 + 단편화 |
|   분산된 포인터 추적  |   선형 순회인데 느림  |   캐시 미스 누적  |
|   열 우선 행렬 접근   |   $O(N^2)$인데 매우 느림  |   캐시 라인 낭비  |
|   정렬된 배열 이진 탐색   |   $O(\log{N})$인데 느릴 수 있음   |   예측 불가능한 접근 패턴 |
