## GDB
- GNU Project에서 개발한 명령줄 기반 디버거(GNU Debugger)
- C/C++ 외에도 Fortran, Go, Rust 등 다양한 컴파일 언어를 지원함
- 실행 중인 프로그램의 내부 상태를 들여다보고 흐름을 제어할 수 있는 도구
- 핵심적으로 할 수 있는 작업
    - 특정 지점에서 프로그램을 멈추기(중단점 설정)
    - 한 줄씩 단계별로 실행하기
    - 변수, 메모리, 레지스터의 현재 값 확인하기
    - 함수 호출 스택 추적하기
    - 비정상 종료(Segmentation Fault) 시점의 상태 분석하기

### `printf` 디버깅과의 비교
- `printf`/`cout`을 코드 곳곳에 삽입하는 방식
    - 매번 코드를 수정하고 재컴파일해야 함
    - 검사하고 싶은 변수마다 출력 코드를 추가해야 함
    - 운영 코드에 디버깅 코드가 섞일 위험이 있음
- GDB를 사용하는 방식
    - 코드를 수정하지 않고 실행 시점에 임의 지점에서 멈춰 상태를 관찰
    - 임의의 변수, 메모리 주소, 표현식을 즉석에서 평가
    - 재컴파일 없이 중단점을 추가하거나 제거 가능

## 디버그 빌드
- GDB가 변수명, 함수명, 줄 번호 등 소스 코드와의 대응 정보를 알기 위해서는 디버그 심볼(debug symbol)이 바이너리에 포함되어야 함
- 디버그 심볼은 `-g` 옵션으로 생성됨
```bash
# 디버그 심볼 포함 컴파일
g++ -g -O0 -o app main.app
```
- `-g`
    - 디버그 심볼을 ELF 바이너리에 포함시킴
- `O0`
    - 컴파일러 최적화를 끔
    - 최적화가 켜져 있으면 변수가 레지스터에만 존재하거나, 코드가 재배치되어 줄 번호가 일치하지 않을 수 있음
- 디버깅 중에는 반드시 `-g -O0` 조합으로 빌드해야 정확한 추적이 가능
> `-g` 없이 빌드된 바이너리도 GDB로 열 수 있지만, 변수명·줄 번호 없이 어셈블리 수준에서만 디버깅이 가능

## GDB 실행 방법
```bash
# 1. 일반 실행 - 프로그램과 함께 시작
gdb ./app

# 2. 인자를 전달해 실행
gdb --args ./app input.txt 42

# 3. 실행 중인 프로세스에 붙이기 (attach)
gdb -p <PID>

# 4. 코어 덤프 파일 분석
gdb ./app core
```
- `gdb`만 입력하면 `(gdb) 프롬프트가 뜨고, 여기서 명령어를 입력하며 디버깅을 진행함
- TUI(Text User Interface) 모드는 `gdb -tui ./app` 또는 실행 후 `Ctrl+X A`로 진입할 수 있음
    - 소스 코드와 명령 창이 분할되어 표시되므로 입문자에게 권장

## 핵심 명령어
### 실행 제어
|   명령    |   단축    |   설명    |
|----------|-----------|----------|
|   `run [args]`    |   `r` |   프로그램을 처음부터 실행    |
|   `continue`  |   `c` |   현재 멈춘 지점부터 다음 중단점까지 진행 |
|   `next`  |   `n` |   다음 줄로 이동(함수는 한 번에 실행) |
|   `step`  |   `s` |   다음 줄로 이동(함수 내부로 진입)    |
|   `finish`    |   `fin`   |   현재 함수가 끝날 때까지 실행    |
|   `until <line>` |   `u` |   지정한 줄까지 실행  |
|   `kill`  |   `k` |   현재 디버깅 중인 프로세스 종료  |
|   `quit`  |   `q` |   GDB 종료    |
- next와 `step`의 차이
    - `next`는 함수 호출을 하나의 단위로 보고 한 번에 통과
    - `step`은 함수 호출을 만나면 그 함수의 첫 줄로 진입
    - 라이브러리 함수처럼 들어갈 필요 없는 호출은 `next`로, 직접 작성한 함수의 동작을 확인할 때는 `step`을 사용

## 중단점(Breakpoint)
```
(gdb) break main            # main 함수 시작 지점에 중단점 설정
(gdb) break 42              # 현재 파일의 42번째 줄에 설정
(gdb) break util.cpp:88    # 특정 파일의 88번째 줄에 설정
(gdb) break MyClass::foo    # 클래스 멤버 함수에 설정
(gdb) info breakpoints      # 등록된 중단점 목록 확인
(gdb) delete 2              # 2번 중단점 삭제
(gdb) disable 2             # 2번 중단점 비활성화(삭제하지 않음)
(gdb) enable 2              # 비활성화된 중단점 재활성화
(gdb) clear 42              # 42번째 줄의 모든 중단점 제거
```

### 조건부 중단점
```
(gdb) break loop.cpp:30 if i == 100
(gdb) condition 1 x > 0     # 1번 중단점에 조건 추가
```
- 반복문 안에서 특정 조건일 때만 멈추고 싶을 때 매우 유용
- 매 반복마다 일일이 `continue`를 누르지 않아도 원하는 시점에 정확히 멈춤

### 상태 확인
|   명령    |   설명    |
|   `print <expr>`(`p`) |   변수나 표현식의 값 출력 |
|   `display <expr>`    |   매 정지 시마다 자동으로 표현식 값 출력  |
|   `info locals`   |   현재 스택 프레임의 지역 변수 목록   |
|   `info args` |   현재 함수의 인자 목록   |
|   `backtrace`(`bt`)   |   함수 호출 스택 출력 |
|   `frame <n>`(`f`)    |   n번 스택 프레임으로 이동    |
|   `list`(`l`) |   현재 위치 주변의 소스 코드 표시 |
|   `watch <var>`   |   변수 값이 변경될 때마다 멈춤    |

```
(gdb) print x           # x의 값
(gdb) print arr[5]      # 배열 원소
(gdb) print *ptr        # 포인터가 가리키는 값
(gdb) print obj.member  # 객체의 멤버
(gdb) print/x x         # 16진수로 출력 (/d 10진수, /t 2진수, /c 문자)
```

### `backtrace`와 스택 프레임
- 프로그램이 멈춘 시점에 어떤 함수들이 호출되어 있는지 위에서 아래로 표시
- Segmentation Fault가 발생했을 때 가장 먼저 확인해야 할 정보
```
(gdb) bt
#0  0x00005555555551a9 in process (data=0x0) at main.cpp:15
#1  0x00005555555551e3 in main () at main.cpp:23
```
- `#0`이 현재 멈춘 위치, 아래로 갈수록 호출한 쪽
- `frame 1`로 이동하면 `main` 함수의 컨텍스트에서 변수를 확인할 수 있음

## 실전 예시: Segmentation Fault 추적
```cpp
// main.cpp
# include <iostream>
using namespace std;

void process(int* data){
    cout << *data << endl;      // data가 nullptr이면 여기서 죽음
}

int main(){
    int* p = nullptr;
    process(p);
    return 0;
}
```
- 빌드와 실행
```bash
$ g++ -g -O0 -o app main.cpp
$ gdb ./app
```
- GDB 섹션
```
(gdb) run
Starting program: ./app

Program received signal SIGSEGV, Segmentation fault.
0x00005555555551a9 in process (data=0x0) at main.cpp:5
5           cout << *data << endl;

(gdb) bt
#0  0x00005555555551a9 in process (data=0x0) at main.cpp:5
#1  0x00005555555551e3 in main () at main.cpp:10

(gdb) print data
$1 = (int *) 0x0
```
- 죽은 정확한 줄과 호출 경로, 문제 인자(`data=0x0`)까지 한 번에 확인됨

## 코어 덤프 분석
- 프로그램이 비정상 종료될 때 메모리 상태를 파일로 남기는 기능
- 운영 환경에서 재현이 어려운 충돌을 사후에 분석할 때 유용
```bash
# 코어 덤프 활성화 (현재 셀 한정)
ulimit -c unlimited

# 코어 파일 저장 경로 확인
cat /prc/sys/kernel/core_pattern

# 실행 후 충돌 시 core 파일 생성
./app

# GDB로 분석
gdb ./app core
```
- GDB가 코어 파일을 열면 충돌 시점의 상태에서 멈춰 있는 것처럼 `bt`, `print` 등 모든 명령을 사용할 수 있음

## 자주 사용하는 활용 패턴
1. 무한 루프 디버깅
    - 프로그`램이 멈추지 않고 둘 때 `Crtl+C`로 GDB에 진입
    - `bt`로 현재 어느 함수에서 도는지 확인
    - `print`로 루프 변수 값을 검사`

2. 특정 반복 횟수에서 멈추기
    ```
    (gdb) break main.cpp:30 if i == 999
    ```
    - 100만 번 반복 중 999번째 반복에서만 정확히 멈춤
3. 변수 변경 추적
    ```
    (gdb) watch count
    ```
    - `count` 변수의 값이 바뀌는 순간 자동으로 멈춤
    - 의도치 않게 값이 변하는 버그를 추적할 때 유용

### `.gdbinit`을 통한 환경 설정
- 홈 디렉토리의 `~/.gdbinit` 또는 프로젝트 디렉토리의 `.gdbinit`에 자주 쓰는 명령을 미리 등록할 수 있음
```
# ~/.gdbinit
set print pretty on         # 구조체를 줄바꿈하여 보기 좋게 출력
set print array on          # 배열을 줄바꿈하여 출력
set pagination off          # 페이지 단위 일시 정지 비활성화
set history save on         # 명령어 히스토리 저장
```
- STL 컨테이너(`vector`, `map` 등)는 기본 출력이 읽기 어려움
- GCC에 포함된 STL 프리티 프린터(Python 스크립트)를 활성화하면 `vector<int>`의 내용도 한 눈에 표시됨

## 자주 마주치는 문제
### 변수가 `<optimized out>`으로 표시될 때
- 컴파일러 최적화로 인해 변수가 레지스터에만 존재하거나 제거됨
- `-O0`으로 다시 빌드하면 해결됨

### 줄 번호가 소스와 어긋날 때
- 빌드한 바이너리가 현재 소스 파일과 다른 버전임
- 코드 수정 후 재빌드하지 않은 경우가 대부분
- 깨끗이 재빌드(`make clean && make`)로 해결

### `No symbol table is loaded`
- `-g` 없이 빌드된 바이너리임
- 디버그 심볼을 포함해 다시 빌드해야 함�
