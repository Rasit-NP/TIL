## CMake
- 빌드 시스템 자체가 아니라, 빌드 시스템을 생성하는 메타 빌드 도구
- `CMakeLists.txt`라는 선언적 스크립트를 작성하면, 플랫폼별로 적절한 빌드 파일(Makefile, Ninja, Visual Studio 솔루션 등)을 생성함
- PS 코드는 `g++ a.cpp` 한 줄이면 끝이지만, 다중 파일·다중 라이브러리·플랫폼 호환·테스트·벤치마크가 얽힌 프로젝트는 C++에서는 빌드 그래프를 손으로 관리할 수 없음
- 빌드는 일반적으로 2단계로 진행됨
```
[ CMakeLists.txt ]   →   [ Generator(Makefile/Ninja/...) ]   →   [ 실행 파일 ]
      (작성)                    cmake -S . -B build            cmake --build build
```

### 빌드 명령 흐름
```bash
# 1. 빌드 디렉토리 생성 및 Configure (CMakeLists.txt → Makefile/Ninja 생성)
cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug

# 2. 실제 빌드 (생성된 빌드 시스템으로 컴파일·링크)
cmake --build build -j

# 3. (선택) 테스트 실행
ctest --test-dir build
```
> `-S`는 source root, `-B`는 build 디렉토리
>
> 빌드 산출물을 소스 트리 밖(out-of-source build)에 두는 것이 표준임

## 핵심 명령어
### `cmake_minimum_required`, `project()`
```cmake
cmake_minimum_required(VERSION 3.20)
project(http-server VERSION 0.1.0 LANGUAGES CXX)
```
- `cmake_minimum_required`
    - 사용한 CMake 최소 버전을 명시
    - 이 버전 미만에서는 빌드를 거부하고, 명시된 버전의 정책(Policy)을 기반으로 동작
- `project()`
    - 프로젝트 이름·버전·언어를 등록
    - 이 호출 시점부터 `CMAKE_PROJECT_NAME`, `PROJECT_SOURCE_DIR` 같은 핵심 변수가 채워짐
- 두 명령은 반드시 루트 CMakeLists.txt의 가장 위쪽에 위치해야 함

### `add_executable`, `add_library`
```cmake
# 실행 파일 타겟
add_executable(http-server src/main.cpp)

# 라이브러리 타겟 (STATIC, SHARED, INTERFACE 중 선택)
add_library(http-core STATIC src/server.cpp src/router.cpp)
```
- CMake는 타겟(Target) 중심으로 동작함
- 실행 파일과 라이브러리 모두 타겟이고, 이후의 모든 설정(`target_*`)이 타겟 단위로 걸림
- 라이브러리 종류
    - `STATIC`
        - `.a`/`.lib`
        - 링크 시 실행 파일에 통합됨
    - `SHARED`
        - `.so`/`.dll`
        - 런타임 동적 로딩
    - `INTERFACE`
        - 헤더 전용 라이브러리
        - 컴파일할 소스가 없고, include 경로·컴파일 옵션만 전파

### `target_include_directories`
```cmake
target_include_directories(http-core
    PUBLIC  ${CMAKE_CURRENT_SOURCE_DIR}/include     # 헤더가 외부에 노출됨
    PRIVATE ${CMAKE_CURRENT_SOURCE_DIR}/src         # 내부 구현에서만 사용
)
```
- 해당 타겟이 사용할 헤더 검색 경로를 지정함
- 전역 `include_directories()` 대신 타겟별로 지정하는 것이 모든 CMake의 원칙(scope leak 방지)

### `target_link_libraries`
```cmake
add_executable(http-server src/main.cpp)
target_link_libraries(http-server PRIVATE http-core)
```
- 타겟 간 의존성을 선언함
    - 라이브러리뿐만 아니라 외부 패키지(`fmt::fmt`, `Threads::Threads` 등)도 동일한 방식으로 연결됨
- 타겟을 연결하면 include 경로·컴파일 옵션·정의 매크로까지 함께 전파됨
    - 추이적 의존성(transitive dependency)의 핵심

### `target_compile_features`
```cmake
target_compile_features(http-core PUBLIC cxx_std_20)
```
- C++ 표준을 타겟 단위로 지정하는 모던 방식
- 전역 변수 `CMAKE_CXX_STANDARD`를 쓰는 구식 방법보다 안전함(다른 타겟에 영향 X)
- 표준뿐 아니라 특정 언어 기능(예: `cxx_constexpr`, `cxx_lambda`)도 지정 가능

### `add_subdirectory`
```cmake
# 루트 CMakeLists.txt
add_subdirectory(src)
add_subdirectory(tests)
```
- 디렉토리 단위로 CMakeLists.txt를 분리해 호출함
- 각 하위 디렉토리는 자신의 타겟을 정의하지만, 변수·타겟은 부모 스코프와 일정 부분 공유됨
- 프로젝트 규모가 커지면 루트 파일이 비대해지므로 반드시 분리 필요

## PUBLIC/PRIVATE/INTERFACE
> `target_*` 명령에 붙는 키워드
>
> 의존성이 타겟 자신과 그 타겟을 사용하는 쪽 중 어디까지 전파될지를 결정함

|   키워드  |   타겟 자신 빌드 시   |   이 타겟을 링크하는 쪽   |
|----------|---------------------|------------------------|
|   `PRIVATE`   |   사용함  |   사용 안 함  |
|   `INTERFACE` |   사용 안 함  |   사용함  |
|   `PUBLIC`    |   사용함  |   사용함  |

### 판단 기준
- 헤더 파일에 노출되는가?
    - 노출되면 `PUBLIC`, 아니면 `PRIVATE`
- 자기 자신은 안 쓰고 외부에만 강제하는가?
    - `INTERFACE`(헤더 전용 라이브러리에서 자주 사용)

### 예시
```cpp
// http-core/include/server.h   -- 외부에 노출되는 헤더
# include <boost/asio.hpp>      // boost는 헤더에 등장함 → PUBLIC 필요

class Server { ... };
```
```cpp
// http-core/src/router.cpp     -- 내부 구현
# include <fmt/format.h>        // fmt는 .cpp에서만 사용 → PRIVATE면 충분
```
```cmake
target_link_libraries(http-core
    PUBLIC  Boost::asio     # 헤더에 노출되므로 사용자도 알아야 함
    PRIVATE fmt::fmt        # 내부 구현 디테일, 사용자는 몰라도 됨
)
```

## 디렉토리 구조와 CMakeLists.txt 분리
### 골격
```
http-server/
├── CMakeLists.txt          # 루트
├── src/
│   ├── CMakeLists.txt
│   └── main.cpp
├── include/                # 헤더
├── tests/
│   ├── CMakeLists.txt
│   └── dummy_test.cpp
├── benchmarks/
├── .clang-format
├── .gitignore
└── README.md
```

### 루트 `CMakeLists.txt`
```cmake
cmake_minimum_required(VERSION 3.20)
project(http-server VERSION 0.1.0 LANGUAGES CXX)

# 전역적인 정책만 여기서 처리
set(CMAKE_EXPORT_COMPILE_COMMANDS ON)   # clangd, IDE 분석용 compile_commands.json 생성

# 하위 디렉토리로 위임
add_subdirectory(src)
add_subdirectory(tests)
```

### `src/CMakeLists.txt`
```cmake
add_executable(http-server main.cpp)

target_include_directories(http-server
    PRIVATE ${CMAKE_SOURCE_DIR}/include
)

target_compile_features(http-server PRIVATE cxx_std_20)

# 컴파일러 경고 옵션 (선택사항이지만 강력 권장)
target_compile_options(http-server PRIVATE
    -Wall -Wextra -Wpedantic
)
```

### `tests/CMakeLists.txt`
```cmake
enable_testing()

add_executable(dummy_test dummy_test.cpp)
target_compile_features(dummy_test PRIVATE cxx_std_20)

add_test(NAME dummy COMMAND dummy_test)
```
- `enable_testing()`은 루트에서 호출해도 되지만, 테스트 디렉토리에서 호출하는 것도 일반적
- `add_test`는 `ctest`로 실행할 테스트를 등록함

## 자주 막히는 지점
### Configure가 됐는데 Build에서 깨질 때
- 빌드 디렉토리에 캐시(`CMakeCache.txt`)가 남아 옛 설정이 반영됨
- 빌드 디렉토리 자체를 삭제하고 처음부터 재생성
```bash
rm -rf build
cmake -S . -B build
```

### 헤더를 못 찾는다고 할 때
- 가장 흔한 원인
    - `target_include_directories`를 빠뜨리거나, `PRIVATE`만 걸어놓고 의존 타겟에서 그 헤더를 사용함
- 헤더에 노출되는 라이브러리라면 `PUBLIC`으로 바꿀 것

### 옛 스타일 명령 사용
```cmake
# 구식(전역 적용, scope leak 발생)
include_directories(include)
link_libraries(fmt)
set(CMAKE_CXX_STANDARD 20)

# 모던 (타겟 단위로 격리)
target_include_directories(http-core PUBLIC include)
target_link_libraries(http-core PRIVATE fmt::fmt)
target_compile_features(http-core PUBLIC cxx_std_20)
```
- 옛 스타일은 작은 프로젝트에서는 잘 동작하지만, 타겟이 늘어나면 의존 관계가 꼬여 디버깅이 매우 어려워짐

## FAQ
### CMake와 Make의 차이
- Make는 빌드 시스템 자체로, `Makefile`에 명시된 규칙대로 컴파일·링크를 수행함
- CMake는 메타 빌드 도구로, `CMakeLists.txt`로부터 플랫폼에 맞는 빌드 시스템(Makefile, Ninja, MSBuild 등)을 생성함
- 같은 소스 트리를 Linux에서는 Make로, Windows에서는 Visual Studio 솔루션으로 빌드할 수 있는 것이 CMake의 강점

### `target_link_libraries`에서의 PUBLIC/PRIVATE/INTERFACE
- PRIVATE
    - 내부 구현에서만 사용
    - 헤더에 노출되지 않음
    - 사용자가 알 필요 없음
- PUBLIC
    - 헤더 파일에 사용 흔적이 노출됨
    - 사용자도 이 의존성을 알아야 함
- INTERFACE
    - 자기 자신은 빌드에 안 쓰고 외부에만 강제
    - 헤더 전용 라이브러리의 표준 패턴

### out-of-source build
- 소스 트리와 빌드 산출물을 분리하면 `git status`가 깨끗하게 유지됨
- 여러 빌드 구성(Debug/Release, 다른 컴파일러)을 동시에 별도 디렉토리에 둘 수 있음
- 빌드 산출물 폐기가 단순함(`rm -rf build` 한 줄)

### `CMAKE_EXPORT_COMPILE_COMMANDS`
- `compile_commands.json`을 생성해 clangd, ccls 같은 LSP가 정확한 컴파일 옵션으로 코드 분석을 수행할 수 있게 함
- 모던 C++ 개발 환경에서 사실상 필수 옵션
