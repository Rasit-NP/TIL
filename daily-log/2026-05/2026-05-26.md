## 🧠 오늘 배운 것
- GitHub Actions를 활용한 C++/CMake 프로젝트 CI 구성

## GitHub Actions
- GitHub가 제공하는 워크플로 자동화 플랫폼
- 저장소 이벤트(push, pull request, schedule 등)를 트리거로 빌드·테스트·배포 작업을 자동 실행함
- 별도 CI 서버를 운영하지 않고도 GitHub가 제공하는 가상 머신(runner)에서 작업이 수행됨
- 워크플로는 `.github/workflows/` 아래의 YAML 파일로 정의되며, 저장소 코드와 함께 버전 관리됨

### 구성 요소
- Workflow
    - YAML 파일 하나가 곧 하나의 워크플로
    - 트리거 이벤트와 실행할 작업을 정의함
- Job
    - 워크플로 안에서 실행되는 작업 단위
    - 기본적으로 Job끼리는 병렬 실행되며, `needs`로 순차 의존성을 걸 수 있음
- Step
    - Job 안에서 순차적으로 실행되는 명령 단위
    - 셸 명령(`run`) 또는 재사용 가능한 액션(`uses`)으로 구성됨
- Runner
    - Job이 실제로 실행되는 가상 머신
    - GitHub 호스팅(`ubuntu-latest`, `windows-latest`, `macos-latest`) 또는 자체 호스팅 선택 가능

## 워크플로 파일의 기본 구조
```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout source
        uses: actions/checkout@v4
    
      - name: Echo
        run: echo "Hello, Actions"
```

### `on` -- 트리거
- 워크플로가 언제 실행될지 정의함
- 자주 쓰이는 트리거
    - `push`
        - 지정한 브랜치에 커밋이 푸시될 때
    - `pull_request`
        - PR이 열리거나 업데이트될 때
    - `workflow_dispatch`
        - GitHub UI에서 수동 실행 버튼을 누를 때
    - `schedule`
        - cron 표현식으로 주기 실행
- 별도 폴링 없이 GitHub 내부 이벤트를 직접 구독하므로 즉시 트리거됨

### `jobs` -- 작업
- 하나 이상의 Job을 정의함
- 각 Job은 `runs-on`으로 실행 환경을 지정함
- Job 간 의존성을 두려면 `needs` 키워드를 사용함

### `steps` -- 단계
- 한 Job 안에서 위에서 아래로 순차 실행됨
- `uses`
    - 마켓플레이스 또는 저장소의 재사용 가능한 액션을 가져옴
- `run`
    - 셸 명령을 직접 실행함

## 핵심 액션
> 거의 모든 워크플로에서 반복적으로 등장하는 표준 액션들

### `actions/checkout@v4`
- 워크플로가 실행되는 runner에 저장소 코드를 가져옴
- 사실상 모든 워크플로의 첫 단계
- 기본 동작은 트리거 이벤트의 커밋 시점 코드를 단일 깊이로 클론함

### `actions/setup-*` 계열
- 언어·도구의 특정 버전을 runner에 설치함
- 예시
    - `actions/setup-node`
    - `actions/setup-python`
    - `actions/setup-java`
- C++의 경우 `ubuntu-latest`에 GCC, Clang, CMake가 이미 설치되어 있어 별도 setup 액션이 거의 필요 없음

## C++/CMake 프로젝트의 최소 워크플로

### 디렉토리 위치
```
my_project/
├── .github/
│   └── workflows/
│       └── ci.yml          ← 워크플로 파일
├── CMakeLists.txt
├── src/
└── tests/
```
- 워크플로 파일은 이미 `.github/workflows/` 아래에 위치해야 GitHub가 인식함
- 파일 이름은 자유이며 한 저장소에 여러 워크플로를 둘 수 있음

### 최소 워크플로
```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout source
        uses: actions/checkout@v4
    
      - name: Configure
        run: cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
    
      - name: Build
        run: cmake --build build -j
    
      - name: Test
        run: ctest --test-dir build --output-on-failure
```

- 로컬에서 사용하던 명령 세 줄을 그대로 옮긴 형태임
- `ubuntu-latest` runner에는 최신 CMake·GCC가 기본 설치되어 있어 별도 설치 단계가 필요 없음
- `--output-on-failure` 옵션은 테스트 실패 시 표준 출력을 그대로 보여주어 디버깅에 유리함

### 동작 흐름
```
[ git push ]
     ↓
GitHub가 .github/workflows/ci.yml 감지
     ↓
ubuntu-latest runner 할당
     ↓
checkout → configure → build → test
     ↓
실패 시 PR/커밋에 빨간 X 표시, 성공 시 초록 체크
```

## 워크플로 점진적 개선 방향
> 처음부터 모든 옵션을 도입할 필요는 없음. 빌드 시간·요구사항이 늘어날 때 단계적으로 추가

### Matrix Build
- 여러 OS·컴파일러·빌드 타입 조합에서 동시에 빌드·테스트
- C++ 같이 플랫폼 의존성이 큰 언어에서 특히 유용
```yaml
jobs:
  build-and-test:
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        build_type: [Debug, Release]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - run: cmake -S . -B build -DCMAKE_BUILD_TYPE=${{ matrix.build_type }}
      - run: cmake --build build -j
      - run: ctest --test-dir build --output-on-failure
```

### 빌드 캐싱
- GoogleTest 같은 `FetchContent` 의존성이 매번 다운로드되어 빌드 시간이 길어질 때 도입 고려
- `actions/cache`로 `build/_deps` 또는 ccache 디렉토리를 캐시함

### Sanitizer 빌드
- AddressSanitizer, UndefinedBehaviorSanitizer 등을 활성화한 별도 Job을 추가
- 빌드 시간이 늘어나므로 캐싱과 함께 도입하는 것이 자연스러움

## 주의점
### 워크플로는 코드와 함께 버전 관리됨
- `.github/workflows/` 안의 YAML이 곧 인프라 정의
- 워크플로 자체의 변경도 PR로 리뷰하는 것이 안전함

### Secret 노출 방지
- 토큰, API 키 등은 절대 YAML에 평문으로 적지 말 것
- GitHub 저장소 Settings의 Secrets 기능으로 등록한 뒤 `${{ secrets.NAME }}`으로 참조

### Runner의 상태는 매번 초기화됨
- Job이 끝나면 runner는 폐기되므로, 이전 빌드 결과를 다음 실행에서 그대로 쓸 수 없음
- 산출물을 보존하려면 `actions/upload-artifact`로 업로드하거나 캐시를 활용해야 함

### 무료 사용 한도
- 퍼블릭 저장소는 무료로 무제한 사용 가능
- 프라이빗 저장소는 월별 무료 분 한도가 정해져 있으므로, 매트릭스 폭을 무분별하게 늘리면 비용이 누적될 수 있음
