# C++ `std::map`과 `unordered_map`의 메모리 구조 차이
## 두 컨테이너의 내부 자료 구조
- `map`은 노드 기반의 균형 이진 탐색 트리(Red-Black Tree)로 구현됨
- `unordered_map`은 해시 테이블로 구현되며, 버킷 배열과 노드의 연결 리스트로 구성됨
- 둘 다 노드를 Heap에 동적 할당하지만, 노드를 묶는 방식과 추가로 유지하는 메타데이터가 다름

### `map`의 메모리 구조
- 각 노드는 키·값 외에 부모·좌측·우측 자식 포인터와 색상 비트를 함께 보유함
```
map<int, int> 노드 하나

┌──────────────────────────────┐
│  parent  (포인터)             │
│  left    (포인터)             │
│  right   (포인터)             │
│  color   (RED / BLACK)       │
│  pair<const Key, Value>      │
└──────────────────────────────┘
   → 노드마다 포인터 3개 + 색상 비트 + 정렬 오버헤드
```
- 노드들이 Heap 곳곳에 분산되어 트리 형태로 연결됨
- 탐색 시 포인터를 따라 트리를 내려가므로 캐시 미스가 빈번함

### `unordered_map`의 메모리 구조
- 버킷 배열은 연속된 메모리에 존재하지만, 각 버킷은 노드 연결 리스트의 시작점을 가리킴
```
unordered_map<int,int>

버킷 배열 (연속)
┌─────┬─────┬─────┬─────┐
│  0  │  1  │  2  │  3  │ ...
└──┬──┴─────┴──┬──┴─────┘
   │           │
   ▼           ▼
 [node]      [node] → [node]   (해시 충돌 시 같은 버킷에 연결)
   → 노드는 Heap에 분산, 버킷 배열만 연속
```
- 노드는 키·값과 다음 노드를 가리키는 포인터, 해시 캐시 값을 보유함
- 충돌이 발생한 키들은 동일 버킷의 연결 리스트로 묶임

## 메모리 사용량 비교
| 구분 | `map` | `unordered_map` |
|-----|--------|----------------|
| 노드 부가 정보 | 포인터 3개 + 색상 비트 | 포인터 1개 + 해시 캐시 |
| 별도 연속 배열 | 없음 | 버킷 배열 존재(부하율에 따라 재할당) |
| 정렬 유지 비용 | 삽입마다 회전·재균형 발생 | 없음 |
| 메모리 분산 | 트리 전체가 Heap에 분산 | 노드 분산 + 버킷 배열만 연속 |

## Rehash와 부하율
- `unordered_map`은 원소 수 대비 버킷 수의 비율인 부하율(load factor)이 임계값을 넘으면 버킷 배열을 키워 재해싱함
- 재해싱 시 모든 원소의 버킷 위치를 다시 계산하므로 일시적으로 비용이 큼
- `map`은 이런 일괄 재배치가 없는 대신, 삽입·삭제마다 국소적인 회전 연산으로 균형을 유지함
> 원소 수를 미리 알 수 있다면 `reserve`로 버킷을 선확보해 재해싱을 줄이는 것이 유리함

## 선택 기준
- 키 순서대로 순회하거나 범위 탐색이 필요하면 `map`이 적합함
- 순서가 무의미하고 단일 키 조회 속도가 중요하면 `unordered_map`이 유리함
- 다만 `unordered_map`은 노드별 추가 할당과 버킷 배열로 인해 메모리 사용량이 더 큰 경향이 있음

# Raw 소켓으로 TCP 서버 구현(`socket`/`bind`/`listen`/`accept`)
## 버클리 소켓
- 운영체제가 네트워크 통신을 파일과 동일한 추상화(파일 디스크립터)로 제공하는 시스템 콜 인터페이스
- 소켓을 열면 정수형 fd가 반환되며, 이후 `read`/`write`로 일반 파일처럼 데이터를 주고받음
- TCP 서버는 수동 소켓(passive socket)을 만들어 연결을 수신 대기하고, 연결마다 능동 소켓(connected socket)을 새로 발급받는 구조임

## 서버 측 연결 수립 흐름
> 클라이언트는 `socket`→`connect` 두 단계지만, 서버는 네 개의 시스템 콜을 거쳐야 함
```
socket()   → 통신 종단점 생성, fd 발급
   ↓
bind()     → fd에 IP/포트 결합
   ↓
listen()   → 수동 소켓으로 전환, 연결 대기 큐 생성
   ↓
accept()   → 완료된 연결을 꺼내 새 fd 발급 (블로킹)
   ↓
read/write → 발급된 fd로 데이터 송수신
```
- `socket`이 만든 fd와 `accept`가 반환하는 fd는 서로 다른 역할을 가짐
    - 전자는 "연결을 받기만 하는" 리스닝 소켓
    - 후자는 "실제 통신에 쓰이는" 연결 소켓

## 각 시스템 콜의 의미
### `socket(AF_INET, SOCK_STREAM, 0)`
- 첫 인자는 주소 체계(IPv4), 둘째 인자는 소켓 타입(스트림, 즉 TCP)을 지정함
- 셋째 인자 0은 해당 타입의 기본 프로토콜을 자동 선택함을 의미함
- 실패 시 -1을 반환하고 errno를 설정함

### `setsockopt(SO_REUSEADDR)`
- 소켓의 동작 옵션을 변경하는 시스템 콜
- `SO_REUSEADDR`를 켜면 TIME_WAIT 상태로 점유 중인 포트에 즉시 다시 바인딩할 수 있음
- 서버를 죽인 직후 재시작할 때 "Address already in use" 오류를 회피하는 핵심 옵션임

### `bind(8080)`
- 소켓을 특정 IP 주소와 포트 번호에 결합함
- `INADDR_ANY`를 쓰면 호스트의 모든 네트워크 인터페이스에서 수신함
- 포트 번호는 네트워크 바이트 순서(빅 엔디언)로 변환해야 하므로 `htons`를 거쳐야 함

### `listen(backlog=10)`
- 소켓을 능동(connect)에서 수동(accept) 모드로 전환함
- `backlog`는 완료된 연결을 대기시키는 큐의 최대 길이를 의미함
- 큐가 가득 차면 새 연결 요청이 거부되거나 무시됨

### `accept()`
- 대기 큐에서 완료된 연결 하나를 꺼내 새 fd로 반환함
- 큐가 비어 있으면 연결이 들어올 때까지 블로킹됨
- 반환된 fd로만 실제 데이터를 주고받음

## 구현 예시
```cpp
// src/main.cpp -- 의도적으로 RAII 없이 raw C 스타일로 작성
# include <sys/socket.h>
# include <netinet/in.h>
# include <arpa/inet.h>
# include <unistd.h>
# include <cstring>
# include <cstdio>
# include <cstdlib>

int main(){
    // 1. 소켓 생성
    int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
    if (listen_fd < 0){
        perror("socket");
        exit(EXIT_FAILURE);
    }

    // 2. SO_REUSEADDR 설정
    int opt = 1;
    if (setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt))<0){
        perror("setsockopt");
        exit(EXIT_FAILURE);
    }

    // 3. 주소 구조체 채우기
    sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = htonl(INADDR_ANY);   // 모든 인터페이스
    addr.sin_port = htons(8080);                // 호스트 -> 네트워크 바이트 순서

    // 4. 바인딩
    if (bind(listen_fd, reinterpret_cast<sockaddr*>(&addr), sizeof(addr))<0){
        perror("bind");
        exit(EXIT_FAILURE);
    }

    // 5. 리스닝
    if (listen(listen_fd, 10) < 0){
        perror("listen");
        exit(EXIT_FAILURE);
    }

    // 6. 연결 수락 (블로킹)
    sockaddr_in client_addr{};
    socklen_t client_len = sizeof(client_addr);
    int conn_fd = accept(listen_fd, reinterpret_cast<sockaddr*>(&client_addr), &client_len);

    if (conn_fd < 0){
        perror("accept");
        exit(EXIT_FAILURE);
    }

    // 7. 응답 전송 후 종료
    const char* msg = "Hello from server\n";
    write(conn_fd, msg, strlen(msg));

    close(conn_fd);
    close(listen_fd);
    return 0;
}
```
> 모든 시스템 콜 뒤에 반환값 검사와 `perror` 로깅을 붙임
>
> `perror`는 직전 errno에 해당하는 사람이 읽을 수 있는 메시지를 표준 에러로 출력함

## 동작 확인 방법
- 서버 실행 후 다른 터미널에서 `nc localhost 8080`으로 접속해 메시지 수신을 확인함
- `accept` 호출 직전에 두 번째 터미널에서 `strace -p <PID>`를 붙이면 프로세스가 `accept`에서 멈춰 있는 상태를 관찰할 수 있음

## 직접 부딪혀 봐야 하는 지점
> `SO_REUSEADDR`를 일부러 빼고 빌드해 보면 동작 원리가 체감됨
- 서버를 한 번 종료한 뒤 즉시 재시작하면 `bind`가 "Address already in use"로 실패함
- 이는 방금 닫은 연결이 TCP의 TIME_WAIT 상태로 포트를 일정 시간 점유하기 때문임
- 옵션을 다시 켜면 같은 포트에 바로 재바인딩되어 문제가 사라짐

## RAII 없이 작성하는 이유
- fd를 손으로 열고 닫아 봐야, 추상화가 무엇을 숨겨 주는지 명확히 인식할 수 있음
- 예외가 던져지거나 중간에 `return`하면 `close`가 누락되어 fd가 새는 문제가 보임
    - 바로 이 누수 지점이 다음 단계에서 RAII 래퍼를 설계하는 직접적인 동기가 됨
