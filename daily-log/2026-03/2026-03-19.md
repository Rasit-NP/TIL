- Redis를 활용한 캐싱 전략

## 캐싱
- 자주 요청되거나 계산 비용이 높은 데이터를 빠른 저장소에 미리 저장해두고, 이후 요청 시 원본 저장소(DB 등)를 거치지 않고 빠르게 응답하는 기법
```
Client -> [Cache Hit] -> Redis -> 빠른 응답 ✅
Client → [Cache Miss] → DB 조회 → Redis 저장 → 응답
```

### 핵심 용어
|   용어    |   설명    |
|----------|----------|
|   Cache Hit   |   요청한 데이터가 캐시에 존재 |
|   Cache Miss  |   요청한 데이터가 캐시에 없음 -> DB 조회 필요 |
|   TTL(Time To Live)   |   캐시 데이터의 유효 기간 |
|   Eviction    |   캐시가 가득 찼을 때 오래된 데이터를 제거하는 정책   |

## Redis를 캐시로 사용하는 이유
- 인메모리(In-Memory) 저장
    - 디스크 I/O 없이 마이크로초 단위 응답
- 다양한 자료구조
    - String, Hash, List, Set, Sorted Set 등 지원
- TTL 내장
    - `EXPIRE` 명령으로 자동 만료 처리
- Atomic 연산
    - 동시성 문제 없는 원자적 처리
- Pub/Sub, Lua 스크립트 등 고급 기능 지원

## 캐싱 전략 종류
### Cache-Aside(Lazy Loading)
- **읽기 중심** 워크로드에 적합
- 애플리케이션이 직접 캐시를 관리
```
Read 요청
  ├─ Cache Hit → 캐시에서 반환
  └─ Cache Miss → DB 조회 → 캐시에 저장 → 반환
```
- 장점
    - 실제로 사용되는 데이터만 캐시에 적재(메모리 효율적)
    - DB 장애 시에도 캐시로 일부 서비스 가능
- 단점
    - 최초 요청 시 Cache Miss로 인한 지연(Cold Start)
    - 캐시와 DB 간 데이터 불일치 가능성

### Write-Through
- 쓰기 요청 시 캐시와 DB를 동시에 업데이트
```
Write 요청
  └─ 캐시 업데이트 → DB 업데이트 (동기)
```
- 장점
    - 캐시와 DB 간 데이터 일관성 보장
    - 이후 읽기 요청은 항상 Cache Hit
- 단점
    - 쓰기 지연(Write Latency) 증가
    - 자주 읽히지 않는 데이터도 캐시에 적재 -> 메모리 낭비

### Write-Behind(Write-Back)
- 캐시에 먼저 쓰고, DB 반영은 비동기로 처리
```
Write 요청
  └─ 캐시 즉시 업데이트 → (비동기) DB 업데이트
```
- 장점
    - 쓰기 성능이 매우 빠름
    - DB 부하 감소(배치 처리 가능)
- 단점
    - 캐시 장애 시 미반영 데이터 유실 위험
    - 구현 복잡도 높음

### Read-Through
- 캐시가 DB 조회를 **대리(Proxy)**함
- Cache-Aside와 유사하지만, 캐시 계층이 자동으로 DB를 조회하며 채움
- 장점
    - 애플리케이션 코드 단순화
    - 캐시 미스 처리가 캐시 계층에서 자동화
- 단점
    - 초기 Cold Start 문제 동일
    - 캐시 라이브러리/프레임워크 의존성 높음

### 전략 비교 요약
| 전략 | 읽기 성능 | 쓰기 성능 | 일관성 | 복잡도 | 적합한 상황 |
|------|-----------|-----------|--------|--------|-------------|
| Cache-Aside | ✅ 높음 | 보통 | △ 약함 | 낮음 | 읽기 빈도 높은 데이터 |
| Write-Through | ✅ 높음 | △ 낮음 | ✅ 강함 | 보통 | 데이터 정합성 중요 |
| Write-Behind | ✅ 높음 | ✅ 높음 | △ 약함 | 높음 | 쓰기 부하가 큰 시스템 |
| Read-Through | ✅ 높음 | 보통 | △ 약함 | 보통 | 코드 단순화 원할 때 |

## TTL/만료 전략
- TTL(Time To Live)은 캐시 데이터의 **유효 기간**을 설정해 자동으로 삭제되도록 함
### Redis TTL 설정 방법
```bash
# 키 저장과 동시에 TTL 설정(초 단위)
SET user:1001 '{"name":"Alice"}' EX 3600

# 기존 키에 TTL 추가
EXPIRE user:1001 3600

# TTL 확인
TTL user:1001   # 남은 초 반환, -1이면 만료 없음, -2이면 키 없음
```

### TTL 선정 기준
|   데이터 유형 |   권장 TTL    |   이유    |
|-------------|--------------|-----------|
|   세션 정보   |   30분 ~ 2시간    |   보안상 짧게 유지    |
|   사용자 프로필   |   1시간 ~ 24시간 |    변경 빈도 낮음  |
|   뉴스/랭킹   |   1분 ~ 10분  |   실시간성 필요   |
|   정적 설정값 |   1일 ~ 7일   |   거의 변하지 않음    |

### EVICTION 정책(메모리 초과 시)
- Redis는 메모리가 가득 찼을 때 다음 정책 중 하나로 데이터를 제거
```bash
# redis.conf 또는 런타임 설정
CONFIG SET maxmemory-policy allkeys-lru
```
|   정책    |   설명    |
|----------|-----------|
|   `noeviction`    |   메모리 초과 시 에러 반환(기본값)    |
|   `allkeys-lru`   |   전체 키 중 가장 오래된 것 제거 (일반적으로 추천)    |
|   `volatile-lru`  |   TTL 설정된 키 중 LRU 제거   |
|   `allkeys-lfu`   |   전체 키 중 가장 적게 사용된 것 제거 |
|   `volatile-ttl`  |   TTL이 가장 짧은 키 먼저 제거    |

## 캐시 무효화 문제
- **DB가 업데이트되었을 때 캐시의 오래된 데이터(Stale Data)를 어떻게 처리하느냐**의 문제

### Stale Data(오래된 데이터)
- DB는 업데이트되었지만, 캐시의 TTL이 남아있어 이전 데이터를 계속 반환하는 상황
- 해결책
    - DB 업데이트 시 명시적으로 캐시 삭제
    - TTL을 짧게 설정
    - Write-Through 전략 채택

### Cache Stampede(캐시 스탬피드)
- 인기 있는 캐시 키가 만료되는 순간, **수많은 요청이 동시에 DB로 몰리는 현상**
- 해결책
    - Mutex Lock
        - 첫 번째 요청만 DB 조회, 나머지는 대기
    - Probabilistic Early Expiration
        - TTL 만료 전에 확률적으로 미리 갱신
    - 백그라운드 워커가 주기적으로 갱신

### Cache Penetration(캐시 관통)
- 존재하지 않는 키를 계속 요청하면, 캐시 미스가 반복되어 **DB에 부하**가 집중되는 현상
- 해결책
    - Null 값도 캐싱 (짧은 TTL로)
    - Bloom Filter로 존재 여부 사전 판별

### Cache Avalanche(캐시 사태)
- 다수의 캐시 키가 **동시에 만료**되면서 DB에 대량 요청이 쏟아지는 현상
- 해결책
    - TTL에 랜덤 지터(Jitter) 추가
    - 캐시 워밍업(Cache Warming)으로 서비스 시작 전 미리 적재

## 정리 및 선택 기준
- 읽기가 많고, 정합성보다 속도가 중요
    - Cache-Aside (Lazy Loading)
- 쓰기 후 즉시 캐시도 최신 상태여야 함
    - Write-Through
- 쓰기 성능이 최우선이고 데이터 유실을 감수할 수 있음
    - Write-Behind
- 캐시 코드를 애플리케이션에서 분리하고 싶음
    - Read-Through
