# 2026-03-30

## 🧠 오늘 배운 것
- 동적 할당 vs. 메모리 풀

## 문제 상황
### `new` 반복 호출의 비용
- 동적 세그먼트 트리에서 쿼리 $Q$번, 좌표 범위 $N=10^9$일 때, 최대 $Q\times 2\log_2 N$번의 `new` 호출이 발생함
- `new` 호출 한 번에서 내부적으로 다음 과정이 순서대로 실행됨
    1. **free list** 순회
        - heap에서 적합한 빈 블록 탐색
    2. **블록 분할 및 메타데이터 기록**
        - 크기$\cdot$상태 헤더 저장
    3. **OS 시스템 콜**(블록 부족 시)
        - `brk`/`mmap`으로 추가 메모리 요청
    4. **포인터 반환**
        - heap 어딘가의 임의 위치를 가리키는 주소 반환

### 단편화(Fragmentation)
- 할당$\cdot$해체를 반복하면 heap 곳곳에 작은 빈 공간이 생겨 연속 블록 확보가 어려워짐
- 메모리 풀은 큰 배열 하나를 미리 선점하므로 이 문제를 처음부터 회피함

## 캐시 미스
### 메모리 계층 구조
- CPU는 메모리를 **64바이트 캐시 라인** 단위로 읽음
- 계층별 접근 비용은 아래와 같음

|   계층    |   접근 비용   |   용량    |
|----------|--------------|-----------|
|   레지스터    |   ~1 사이클   |   수십 B  |
|   L1 캐시 |   ~4 사이클   |   32 ~ 64 KB |
|   L2 캐시 |   ~12 사이클  |   256 KB ~ 1 MB   |
|   L3 캐시 |   ~40 사이클  |   수 MB   |
|   RAM |   ~200 사이클 |   수십 GB |

### `new` 반복 할당 시의 문제
- `new`로 할당된 노드는 heap 곳곳에 분산되어, 다음 노드 접근 시마다 RAM까지 내려가는 캐시 미스가 발생함
```cpp
// new 반복 시 - 노드 주소가 분산됨
TreeNode* a = new TreeNode();   // → 0x7f3a48c0
TreeNode* b = new TreeNode();   // → 0x2c1b0040  (전혀 다른 위치)
TreeNode* c = new TreeNode();   // → 0x9d04ff20  (캐시 라인 밖)

// 메모리 풀 시 - 노드가 배열에 연속 배치됨
// pool[1] | pool[2] | pool[3] | pool[4] …
// 하나를 읽으면 이웃들도 캐시 라인에 함께 올라옴
```

### 하드웨어 프리페처 무력화
- CPU의 **하드웨어 프리페처**는 순차 접근 패턴을 감지하면 다음 캐시 라인을 미리 로드함
- 배열 기반 풀은 이 기능을 자동으로 활성화하지만, 분산된 포인터 추적은 패턴이 없어 프리페처가 무력화됨

## 메모리 풀 구현
### 핵심 아이디어
- 큰 배열을 미리 선점하고, 인덱스 카운터를 증가시키는 것만으로 '할당'을 흉내 냄
- `allocNode()`는 실질적으로 **정수 덧셈 1번 + 구조체 초기화**만 수행함
```cpp
// MAX_NODE 계산 : Q × 2log₂N × 여유배수(2)
const int MAX_NODE = 25'000'000;

struct TreeNode {
    int lc, rc;
    long long val, lazy;
} pool[MAX_NODE];

int pool_idx = 1;   // 0은 null 노드로 예약

int allocNode() {
    pool[pool_idx] = {0, 0, 0LL, 0LL};
    return pool_idx++;  // 정수 덧셈 1번이 전부
}
```

### 포인터 대신 인덱스를 사용하는 이유
- 인덱스(`int`)는 포인터(`TreeNode*`)보다 4바이트 작아 캐시 라인 하나에 더 많은 노드가 올라옴

|   항목    |   포인터(`TreeNode*`) |   인덱스(`int`)   |
|----------|----------------------|-------------------|
|   크기    |   8 바이트    |   4 바이트 (50% 절약) |
|   접근 방식   |   절대 주소 역참조    |   `pool[idx]`(배열 인덱싱)    |
|   캐시 효율   |   분산 주소 -> 미스 빈번  |   연속 배열 -> 히트 우세  |

### 전역 배열을 사용해야 하는 이유
- 스택은 보통 8MB로 제한되어 있어 대용량 배열 선언 시 런타임 에러가 발생함
- 초기값이 0인 전역 배열은 BSS 세그먼트에 위치함
    - 실행 파일에는 크기 정보만 기록되고, 실제 메모리는 OS가 실행 시점에 제공함
    - 대용량 배열을 선언해도 실행 파일 크기에 영향을 주지 않음

```cpp
// ❌ 지역(스택) 배열 — 스택 오버플로 발생
void solve() {
    TreeNode pool[25'000'000];  // → 런타임 에러
}

// ✅ 전역 배열 — BSS 세그먼트에 위치
TreeNode pool[25'000'000];
```

## 풀 크기 계산 공식
- 너무 작으면 런타임 에러, 너무 크면 MLE가 발생하므로 정확한 계산이 필요함
```
MAX_NODE = Q × (2 × ⌈log₂ N⌉) × 여유 배수(2)

예) Q = 200,000 / N = 10^9
    log₂(10^9) ≈ 30
    → 200,000 × 60 × 2 = 24,000,000
    → MAX_NODE = 25,000,000 으로 설정
```

## 성능 비교

|   항목    |   `new` 반복  |   메모리 풀   |
|----------|---------------|-------------|
|   할당 비용   |   수십 ~ 수백 ns/1회  |   ~수 ns/1회  |
|   메모리 위치 |   분산 (임의 주소)    |   연속 (배열 순서)    |
|   캐시 미스   |   빈번    |   드묾    |
|   단편화  |   발생    |   없음    |
|   전체 초기화 |   모든 노드 `delete` 필요 |   `pool_idx = 1` 한 줄    |

```cpp
#include <iostream>
#include <vector>
#include <chrono>
using namespace std;
using namespace chrono;

// ── 공통 구조체 ──────────────────────────────────────
struct Node {
    int lc, rc;
    long long val, lazy;
};

// ── 메모리 풀 ────────────────────────────────────────
const int MAX_NODE = 5'000'000;
Node pool[MAX_NODE];
int pool_idx = 1;

int allocNode() {
    pool[pool_idx] = {0, 0, 0LL, 0LL};
    return pool_idx++;
}

// ── 측정 함수 ────────────────────────────────────────
long long bench_new(int n) {
    auto start = high_resolution_clock::now();

    vector<Node*> nodes;
    nodes.reserve(n);
    for (int i = 0; i < n; i++) {
        nodes.push_back(new Node{0, 0, 0LL, 0LL});
    }
    int x;
    for (int i=0; i<n; i++){
        Node* node = nodes[i];
        x = node -> lc;
    }

    auto end = high_resolution_clock::now();

    for (auto* p : nodes) delete p;  // 메모리 해제
    return duration_cast<milliseconds>(end - start).count();
}

long long bench_pool(int n) {
    pool_idx = 1;  // 풀 초기화

    auto start = high_resolution_clock::now();

    for (int i = 0; i < n; i++) {
        allocNode();
    }
    int x;
    for (int i=0; i<n; i++){
        Node& node = pool[i];
        x = node.lc;
    }

    auto end = high_resolution_clock::now();
    return duration_cast<milliseconds>(end - start).count();
}

void run(int& N){
    cout << "=============================\n";
    cout << "할당 횟수: " << N << "회\n";
    cout << "-----------------------------\n";
    cout << "new 반복:   " << bench_new(N)  << " ms\n";
    cout << "메모리 풀:  " << bench_pool(N) << " ms\n";
    cout << "=============================\n";
}

// ── main ─────────────────────────────────────────────
int main() {
    
    for (int n = 1; n <= (1<<20); n <<= 1){
        run(n);
        cout << endl;
    }

    return 0;
}
/*
=============================
할당 횟수: 131072회
-----------------------------
new 반복:   1102 ms
메모리 풀:  0 ms
=============================

=============================
할당 횟수: 1048576회
-----------------------------
new 반복:   1849 ms
메모리 풀:  8 ms
=============================
*/
```

## 주의 사항
- 메모리 풀을 사용할 때 `reserve`를 사용하지 않고 `push_back`을 사용하게 되면, 메모리 풀의 `Node`를 포인터로 관리할 때 오류가 생길 수 있음
- `push_back` 메서드를 실행할 때, 벡트의 크기가 가득 차게 되면, 새로운 벡터를 생성하고 내용을 복사함
- 이 때, 기존의 벡터 주소를 저장하고 있는 포인터는 잘못된 값을 가리키게 됨
- 할당할 `Node`의 갯수를 예상한 후, `nodePool.reserve(N)`으로 충분한 메모리를 확보해 오류를 막을 수 있음

## 개념 연결 지도
```
큰 N  →  동적 세그먼트 트리 필요
              ↓
         new 반복 호출
           ├─ heap free list 탐색  →  수백 ns / 회
           ├─ 메모리 단편화
           └─ 분산 주소  →  캐시 미스  →  최대 ~200 사이클 / 접근
                                            └─ 하드웨어 프리페처 무력화
              ↓  (메모리 풀 도입)
         pool[] 전역 배열  +  인덱스 카운터
           ├─ 할당 비용 ≈ 0
           ├─ BSS 세그먼트  →  실행 파일 크기 유지
           ├─ 연속 메모리  →  캐시 지역성 확보
           └─ 순차 접근 패턴  →  하드웨어 프리페처 활성화
```
