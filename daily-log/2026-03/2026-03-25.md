- Spring AOP와 프록시 메커니즘

## 프록시
- Spring의 `@Transactional`, `@Cacheable`, `@Async` 같은 애노테이션은 비즈니스 로직에 부가 기능을 끼워 넣는 구조
- 이를 소스 코드 수정 없이 가능하게 하는 것이 AOP(Aspect-Oriented Programming)이고, 그 구현체가 바로 **프록시 객체**
- Spring은 실제 빈 대신 프록시를 컨테이너에 등록하고, 모든 호출을 프록시가 가로채 부가 로직을 실행한 뒤 실제 객체로 위임



## JDK Dynamic Proxy vs. CGLIB
- Spring은 두 가지 방식으로 프록시를 생성
    - 어떤 방식을 쓸지는 대상 클래스가 인터페이스를 구현하는지에 따라 결정됨
    - Spring Boot 2 이후부터는 기본값이 CGLIB으로 변경됨

|   |   JDK Dynamic Proxy   |   CGLIB   |
|---|---|---|
|   프록시 생성 방식    |   인터페이스 구현 |   클래스 상속(바이트코드 조작)    |
|   인터페이스 필요 여부    |   필수    |   불필요  |
|   제약    |   인터페이스 없으면 불가  |   `final` 클래스$\cdot$메서드 불가  |
|   Spring Boot 기본값  |   -  |   ✅ (2.x 이후)   |

## Self-invocation(내부 호출)
- 프록시 메커니즘에서 가장 자주 마주치는 함정
- 같은 클래스 내부에서 `this.method()`를 호출하면 프록시를 거치지 않음
```java
@Service
public class OrderService {

    // 외부에서 이 메서드를 호출하면 프록시를 거침 -> @Transactional 적용
    public void placeOrder(Order order){
        // ❌ this.로 내부 호출 → 프록시를 거치지 않음 -> @Transactional 무시됨
        this.saveOrder(order);
    }

    @Transactional
    public void saveOrder(Order order){
        // DB 저장 로직
    }
}
```

### 해결책
- 구조적 분리(권장)
    - 책임을 다른 빈으로 분리해 프록시를 거치도록 만듦
    ```java
    @Service
    @RequiredArgsConstructor
    public class OrderService {

        private final OrderPersistenceService orderPersistenceService;
        
        public void placeOrder(Order order){
            // 외부 빈 호출 → 프록시 경유 -> @Transactional 정상 적용 ✅
            orderPersistenceService.saveOrder(order);
        }
    }

    @Service
    public class OrderPersistenceService {

        @Transactional
        public void saveOrder(Order order){
            // DB 저장 로직
        }
    }
    ```
- 자기 자신을 주입(ApplicationContext 또는 `@Lazy`)
    - 구조 분리가 어려울 때의 차선책
    - 자기 자신의 프록시를 주입받아 호출
    ```java
    @Service
    public class OrderService {

        @Autowired
        @Lazy   // 순환 참조 방지를 위해 지연 주입
        private OrderService self;

        public void placeOrder(Order order){
            // self는 프록시 -> @Transactional 정상 적용 ✅
            self.saveOrder(order);
        }

        @Transactional
        public void saveOrder(Order order){
            // DB 저장 로직
        }
    }
    ```
    > 자기 주입 방식은 코드 가독성을 해치므로, 구조적 분리를 우선적으로 고려

## 빈 생성 시 프록시가 만들어지는 시점
- Spring 컨테이너 초기화 시, `BeanPostProcessor`가 빈 생성 후 AOP 대상을 감지하면 실제 빈 대신 프록시를 컨테이너에 등록
```
① @Component 스캔 → ② 빈 인스턴스 생성 → ③ BeanPostProcessor 개입
→ ④ AOP 대상이면 프록시 생성 → ⑤ 컨테이너에 프록시 등록
```

## `final` 키워드 주의
- CGLIB은 클래스를 **상속**해 프록시를 만들기 때문에, `final` 클래스나 `final` 메서드에는 프록시가 적용되지 않음
```java
@Service
public final class PaymentService { // ❌ final → CGLIB 프록시 생성 불가

    @Transactional
    public void pay() { ... }
}

@Service
public class PaymentService {

    @Transactional
    public final void pay() { ... } // ❌ final 메서드 → AOP 미적용
}
```

## FAQ
- Spring AOP가 프록시 방식인 이유
    - 컴파일·로드 타임 바이트코드 위빙(AspectJ 방식) 없이, IoC 컨테이너의 빈 생명주기를 활용해 런타임에 부가 기능을 끼워 넣을 수 있기 때문
- `@Transactional`이 같은 클래스 내부 호출에서 동작하지 않는 이유
    - Spring AOP는 프록시 기반이라 외부 호출만 가로챌 수 있음
    - `this.method()` 호출은 프록시를 우회해 실제 빈에 직접 접근하므로 AOP 어드바이스가 적용되지 않음
- JDK Dynamic Proxy와 CGLIB의 차이
    - JDK Proxy는 인터페이스를 구현해 프록시를 만들고, CGLIB는 클래스를 상속해 서브클래스를 생성
    - Spring Boot 2.x 이후 기본값은 CGLIB이며, `final` 클래스$\cdot$메서드에는 CGLIB도 적용이 불가능
