# 2026-03-23

## 🧠 오늘 배운 것
- Redis vs. JVM 인메모리 캐싱

## 데이터 접근 경로
### Redis
```
요청 -> [네트워크] -> [Redis 서버] -> [직렬화/역직렬화] -> 반환
```
- 지연: 1~5ms(로컬 네트워크), 5~20ms(원격)
- 네트워크를 거치므로 반드시 직렬화(객체 -> JSON/Binary) 비용이 발생

### JVM 인메모리
```
요청 -> [JVM Heap에서 직접 읽기] -> 반환
```
- 지연: 수 $\mu s$
- 같은 JVM 안에 있으므로 네트워크, 직렬화 필요 없음

## GC 영향
```
[JVM Heap 구조]
┌──────────────────────────────────────────┐
│  Young Generation   │   Old Generation   │
│  (새로 생성된 객체) │  (오래 살아남은 객체)   │
│                     │                    │
│                     │  ← 캐시 데이터가     │
│                     │    여기에 장기 체류  │
└──────────────────────────────────────────┘
```
- JVM 인메모리 캐시는 속도는 빠르지만 캐시 데이터가 JVM Heap을 점유함
- Old Generation이 가득 차면 **Major GC(Full GC)**가 발생
    - 이 순간 JVM이 잠시 멈추는 **Stop-the-World(STW)** 현상이 발생
    - 데이터가 많아질수록 STW 시간이 길어지고 -> **간헐적 응답 지연**의 원인이 됨
- Redis는 데이터를 Heap 외부에 보관하므로 GC 압박이 전혀 없음

## 스케일아웃 시 차이
- 서버를 여러 대 운영(스케일아웃)할 때 JVM 캐시는 **치명적인 문제**가 생김
- 각 인스턴스는 **독립된 JVM**을 가지므로, 캐시도 인스턴스마다 별도로 존재

### JVM 인메모리 -- 캐시 불일치 발생
```
[변경 전]
Instance A: data 캐시 v1
Instance B: data 캐시 v1

[데이터 변경 후 — A에서 @CacheEvict 실행]
Instance A: data 캐시 v2  ← 최신 데이터
Instance B: data 캐시 v1  ← Stale(낡은) 데이터! 불일치 발생
```
- 로드 밸런서가 요청을 A/B에 번갈아 보내면, 같은 API가 다른 결과를 반환하는 버그가 생김

### Redis -- 모든 인스턴스 동일 뷰 유지
```
[변경 전]
Instance A ←→ Redis (v1)
Instance B ←→ Redis (v1)

[데이터 변경 후]
Instance A ←→ Redis (v2)  ← 모두 동일
Instance B ←→ Redis (v2)  ← 모두 동일
```
- 캐시가 Redis 하나에만 있으므로, 어느 인스턴스에서 요청이 와도 항상 같은 데이터를 봄
>`@CacheEvict`는 JVM 인스턴스에서만 동작함
>다중 인스턴스 환경에서는 Stale Cache 문제가 필연적으로 발생

## 항목별 비교
| 항목 | JVM 인메모리 | Redis |
|---|---|---|
| 접근 속도 | ✅ 수 μs | ❌ 1~20ms |
| 직렬화 비용 | ✅ 없음 (객체 참조 그대로) | ❌ JSON/Binary 변환 필수 |
| GC 영향 | ❌ Heap 점유 → GC 압박 | ✅ Heap 외부 → GC 무관 |
| 인스턴스 간 공유 | ❌ 불가 (각 JVM 독립) | ✅ 가능 (모든 인스턴스 동일 뷰) |
| 용량 한계 | ❌ JVM Heap 크기에 제한 | ✅ 별도 서버 메모리 |
| 장애 시 데이터 | ❌ 앱 재시작 시 소멸 | ✅ 영속화 옵션 있음 (AOF/RDB) |
| TTL 지원 | ❌ 기본 없음 (Caffeine은 가능) | ✅ 네이티브 지원 |
| 운영 복잡도 | ✅ 별도 인프라 불필요 | ❌ Redis 서버 운영 필요 |
| 네트워크 장애 영향 | ✅ 없음 | ❌ Redis 다운 시 전파 |

## 구현 코드
### JVM 인메모리 -- Caffeine 설정
- Caffeine은 Spring Boot에서 가장 널리 쓰이는 로컬 캐시 구현체
- `expireAfterWrite`(쓴 뒤 N분 후 만료), `maximumSize`(최대 항목 수) 등을 선정할 수 있음
```groovy
// build.gradle
implementation 'org.springframework.boot:spring-boot-starter-cache'
implementation 'com.github.ben-manes.caffeine:caffeine'
```
```java
// CacheConfig.java
@Configuration
@EnableCaching  // Spring Cache 기능 활성화
public class CacheConfig {

    @Bean
    public CacheManager cacheManager() {
        Caffeine<Object, Object> caffeine = Caffeine.newBuilder()
            .maximumSize(1_000)                         // 항목이 1,000개를 넘으면 오래된 것부터 제거
            .expireAfterWrite(10, TimeUnit.MINUTES)     // 저장 후 10분이 지나면 자동 만료
            .recordStats();                             // 히트율 통계 수집 (모니터링용)
        
        CaffeineCacheManager cacheManager = new CaffeineCacheManager("wines");
        cacheManager.setCaffeine(caffeine);
        return cacheManager;
    }
}
```
```java
// WineService.java
@Service
public class WineService {

    private final WineRepository wineRepository;

    public WineService(WineRepository wineRepository){
        this.wineRepository = wineRepository;
    }

    // 처음 호출 시 DB 조회 후 "wines" 캐시에 저장
    // 이후 호출은 캐시에서 즉시 반환
    @Cacheable(cacheNames = "wines")
    public List<Wine> findAll() {
        return wineRepository.findAll();
    }

    // "wines" 캐시의 모든 항목을 제거
    // 데이터 변경 후 캐시를 강제로 비울 때 사용
    @CacheEvict(cacheNames = "wines", allEntries = true)
    public void refresh() { /*호출만 해도 캐시가 비워짐*/}
}
```
>### 주의 -- Self-invocation
>같은 클래스 안에서 `findAll()`을 직접 호출하면 `@Cacheable`이 동작하지 않음
>Spring Cache는 AOP 프록시로 구현되어 있어서, **외부 빈에서 호출해야** 프록시를 거침
>
>```java
>// ❌ 동작 안 함 — 같은 클래스 내 self-invocation
>public void someMethod() {
>    this.findAll();
>}
>
>// ✅ 동작함 — 외부 빈에서 호출
>@RestController
>public class WineController {
>   private final WineService wineService;
>
>   public WineController(WineService wineService){
>       this.wineService = wineService;
>   }
> 
>   @GetMapping
>   public List<Wine> get() {
>       return wineService.findAll();   // 프록시를 거침
>   }
>}
>```

### Redis -- Spring Data Redis 설정
```groovy
// build.gradle
implementation 'org.springframework.boot:spring-boot-starter-data-redis'
implementation 'org.springframework.boot:spring-boot-starter-cache'
```
```yaml
# application.yml
spring:
  data:
    redis:
      host: localhost
      port: 6379
      timeout: 2000ms         # 2초 안에 Redis 응답이 없으면 에러
      lettuce:
        pool:
          max-active: 8       # 동시에 사용할 수 있는 최대 커넥션 수
          max-idle: 4
```
```java
// RedisCacheConfig.java
@Configuration
@EnableCaching
public class RedisCacheConfig {

    private final RedisConnectionFactory redisConnectionFactory;

    public RedisCacheConfig(RedisConnectionFactory redisConnectionFactory){
        this.redisConnectionFactory = redisConnectionFactory;
    }

    @Bean
    public RedisCacheManager cacheManager() {
        RedisCacheConfiguration defaultConfig = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))       // 기본 TTL : 10분
            .serializeValuesWith(
                RedisSerializationContext.SerializationPair.fromSerializer(
                    new GenericJackson2JsonRedisSerializer()    // 객체를 JSON으로 직렬화
                )
            )
            .disableCachingNullValues();            // null은 캐싱하지 않음
        
        return RedisCacheManager.builder(redisConnectionFactory)
            .cacheDefaults(defaultConfig)
            .withCacheConfiguration(
                "wines",
                // wines 캐시만 TTL을 1시간으로 별도 설정
                defaultConfig.entryTtl(Duration.ofHours(1))
            )
            .build();
    }
}
```
>### 직렬화 주의
>`GenericJackson2JsonRedisSerializer`는 저장 시 클래스 경로(`@class`)를 JSON에 포함시킴
>```json
>{"@class": "com.example.wine.WineResponse", "id": 1, "name": "Château Margaux"}
>```
>패키지 구조나 클래스명이 바뀌면 **역직렬화 오류**가 발생
>타입을 명시하는 방식이 더 안전함
>```java
>// 더 안전한 방법 -- 타입 명시
>new Jackson2JsonRedisSerializer<>(WineResponse.class)
>```

### Redis Pub/Sub -- 다중 인스턴스 캐시 무효화
- 스케일아웃 환경에서도 JVM 캐시를 유지하고 싶다면, Redis의 발행/구독(Pub/Sub) 기능으로 인스턴스 간 무효화 신호를 전파할 수 있음
```
[동작 원리]
데이터 변경 발생
    → Instance A가 Redis "cache:invalidate" 채널에 메시지 발행
    → Redis가 모든 구독 인스턴스에게 메시지 전달
    → 각 인스턴스가 자신의 로컬 캐시를 비움

Instance A ──publish──▶ Redis "cache:invalidate" ──▶ Instance A (자기 자신도 수신)
                                                   ──▶ Instance B
                                                   ──▶ Instance C
```
```java
// CacheInvalidationPublisher.java — 무효화 신호를 Redis에 발행
@Component
public class CacheInvalidationPublisher {

    private final StringRedisTemplate redisTemplate;

    public CacheInvalidationPublisher(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    public void publish(String cacheName) {
        redisTemplate.convertAndSend("cache:invalidate", cacheName);
    }
}
```
```java
// CacheInvalidationSubscriber.java — Redis 메시지를 수신해 로컬 캐시를 제거
@Component
public class CacheInvalidationSubscriber implements MessageListener {

    private final CacheManager cacheManager;

    public CacheInvalidationSubscriber(CacheManager cacheManager) {
        this.cacheManager = cacheManager;
    }

    @Override
    public void onMessage(Message message, byte[] pattern) {
        String cacheName = message.toString();
        Cache cache = cacheManager.getCache(cacheName);
        if (cache != null) {
            cache.clear();  // 이 인스턴스의 로컬 Caffeine 캐시만 비워짐
        }
    }
}
```
```java
// RedisListenerConfig.java — 구독 채널 등록
@Configuration
public class RedisListenerConfig {

    private final RedisConnectionFactory connectionFactory;
    private final CacheInvalidationSubscriber subscriber;

    public RedisListenerConfig(RedisConnectionFactory connectionFactory,
                               CacheInvalidationSubscriber subscriber) {
        this.connectionFactory = connectionFactory;
        this.subscriber = subscriber;
    }

    @Bean
    public RedisMessageListenerContainer messageListenerContainer() {
        RedisMessageListenerContainer container = new RedisMessageListenerContainer();
        container.setConnectionFactory(connectionFactory);
        container.addMessageListener(subscriber, new ChannelTopic("cache:invalidate"));
        return container;
    }
}
```
```java
// WineService.java — 데이터 변경 시 무효화 발행
@Service
public class WineService {

    private final WineRepository wineRepository;
    private final CacheInvalidationPublisher publisher;

    public WineService(WineRepository wineRepository,
                       CacheInvalidationPublisher publisher) {
        this.wineRepository = wineRepository;
        this.publisher = publisher;
    }

    @Cacheable(cacheNames = "wines")
    public List<Wine> findAll() {
        return wineRepository.findAll();
    }

    @Transactional
    public Wine update(Wine wine) {
        Wine saved = wineRepository.save(wine);
        publisher.publish("wines");  // 모든 인스턴스에 "wines 캐시를 비워라" 전파
        return saved;
    }
}
```
## 실무 적용 판단 기준
- 아래 흐름에 따라 어떤 캐시를 선택할지 결정
```
인스턴스가 2개 이상으로 스케일아웃되는가?
    │
    ├─ Yes ──→ Redis
    │          (JVM 캐시는 인스턴스 간 불일치 필연적)
    │
    └─ No
         │
         캐시 데이터 크기가 수백 MB를 초과하는가?
             │
             ├─ Yes ──→ Redis
             │          (Heap 한계, OOM 위험)
             │
             └─ No
                  │
                  데이터 변경 빈도가 높은가? (분 단위 이하)
                      │
                      ├─ Yes ──→ Redis
                      │          (정교한 TTL 설정 또는 Pub/Sub 무효화 필요)
                      │
                      └─ No ──→ JVM 인메모리 (Caffeine)
                                 단, 스케일아웃 계획이 있다면 Redis 미리 고려
```
### 상황별 요약
| 상황 | 선택 | 이유 |
|---|---|---|
| 단일 인스턴스 + 읽기 전용 + 소용량 | JVM (Caffeine) | 속도 최우선, 별도 인프라 불필요 |
| 다중 인스턴스 (스케일아웃 환경) | Redis | 인스턴스 간 캐시 공유 필수 |
| 캐시 데이터 대용량 또는 영속화 필요 | Redis | Heap 한계 회피, AOF/RDB 영속화 |
| 응답 시간이 극도로 중요한 단일 서비스 | JVM (Caffeine) | μs 단위 지연 확보 |
| 여러 서비스가 같은 캐시를 공유해야 할 때 | Redis | 서비스 간 공유 가능 |
| 빠른 프로토타이핑 / 개발 초기 단계 | JVM (Caffeine) | 세팅이 간단, 나중에 Redis로 교체 쉬움 |
