# Docker Hub
### 개요
- Docker Hub는 Docker 이미지를 저장하고 공유하는 공식 레지스트리
- `docker run`, `Dockerfile`, Docker Compose 등 다양한 방식으로 활용할 수 있음
- 멀티 컨테이너 애플리케이션을 Docker Hub의 이미지 기반으로 손쉽게 정의하고 배포할 수 있음

### Docker Hub
- Docker가 공식 운영하는 컨테이너 이미지 레지스트리
- GitHub가 소스코드를 저장하고 공유하는 플랫폼이라면, Docker Hub는 그 역할을 컨테이너 이미지에 대해 수행
- 이미지를 저장(push)하고, 다운로드(pull)할 수 있는 중앙 저장소
- 공개(Public) 이미지는 로그인 없이도 누구나 pull 가능
- `docker pull nginx`처럼 이미지 이름만 지정하면 기본적으로 Docker Hub에서 가져옴

### Public vs. Private 저장소
|   구분    |   Public  |   Private |
|----------|-----------|-----------|
|열람 권한  | 누구나 pull 가능  | 허가된 계정만 접근    |
|주요 용도 | 오픈소스 배포, 팀 공유    | 사내 서비스, 유료 소프트웨어  |
|무료 플랜 제공 | ✅ 무제한 | ✅ 1개 (무료 기준)    |

> 주의 : 무료 플랜에서 비활성 Private 이미지는 일정 기간 후 삭제될 수 있음. 중요한 이미지는 유료 플랜을 사용하거나 별도 레지스트리(AWS ECR, GitHub Container Registry)를 고려

### Official Image vs. 개인 이미지
|   구분    |   Official Image  |   개인/조직 이미지    |
|----------|--------------------|--------------------|
|   이미지 이름 형식   |   `nginx`, `postgres`, `node` |   `username/image-name`   |
|   관리 주체   |   Docker 공식 또는 공인된 벤더    |   개인 또는 조직  |
|   보안 검토   |   Docker가 주기적으로 스캔$\cdot$검증 |   직접 관리 책임  |
|   신뢰도  |   높음(프로덕션 사용 권장)    |   출처 확인 필요  |
|   예시    |   `nginx:1.25-alpine` |   `myusername/my-app:v1.0.0`  |
> 팁 : 베이스 이미지로 Official Image를 사용하고, 그 위에 앱 코드를 올려 개인 이미지를 만드는 패턴이 일반적

## Docker Hub 이미지 Pull/Push
### Pull
```bash
# 공식 이미지 (latest)
docker pull nginx

# 특정 태그 지정
docker pull nginx:1.25-alpine

# 개인/조직 이미지
docker pull myusername/my-app:v1.0.0
```
> 프로덕션 환경에서는 `latest` 태그 대신 명시적인 버전 태그를 사용해 예기치 않은 업데이트를 방지

### Push
```bash
# Docker Hub 로그인
docker login

# 로컬 이미지에 태그 지정 (Hub 저장소명과 일치시켜야 함)
docker tag my-app:latest myusername/my-app:v1.0.0

# 푸시
docker push myusername/my-app:v1.0.0
```

### CI/CD 환경에서 로그인(비대화형)
```bash
echo "$DOCKER_PASSWORD" | docker login -u "$DOCKER_USERNAME" --password-stdin
```

## Nginx + PostgreSQL 예제
### 디렉토리 구조
```
project/
├── docker-compose.yml
├── nginx/
│   └── default.conf
└── .env
```

### `.env`
```env
POSTGRES_DB=mydb
POSTGRES_USER=myuser
POSTGRES_PASSWORD=secret
```

### `nginx/default.conf`
```nginx
server {
    listen 80;

    location / {
        proxy_pass http://app:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real_IP $remote_addr;
    }
}
```

### `docker-compose.yml`
```yaml
version: "3.9"

services:
  nginx:
    image: nginx:1.25-alpine          # Docker Hub 공식 이미지
    container_name: nginx_proxy
    ports:
      - "80:80"
    volumes:
      - ./nginx/default.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - app
    restart: unless-stopped

  app:
    image: myusername/my-app:v1.0.0   # Docker Hub 개인 이미지
    container_name: my_app
    expose:
      - "8080"
    environment:
      - DATABASE_URL=postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@db:5432/${POSTGRES_DB}
    depends_on:
      db:
        condition: service_healthy
    restart: unless-stopped

  db:
    image: postgres:16-alpine          # Docker Hub 공식 이미지
    container_name: postgres_db
    env_file:
      - .env
    volumes:
      - pg_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped

volumes:
  pg_data:
```

### 핵심 포인트
|   항목    |   설명    |
|----------|-----------|
|   `image:`    |   Docker Hub에서 pull할 이미지와 태그를 지정  |
|   `env_file:` |   `.env` 파일을 통해 민감한 정보를 코드와 분리    |
|   `depends_on: condition: service_healthy`    |   DB가 준비된 후 앱이 기동되도록 순서 보장    |
|   `volumes:`  |   컨테이너가 재시작되어도 DB 데이터가 유지됨  |
|   `restart: unless-stopped`   |   수동으로 내리지 않는 한 자동 재시작 |

## Docker Compose 외 Docker Hub 사용
- Docker Compose 없이도 Docker Hub 이미지를 직접 활용하는 두 가지 주요 방법이 있음

### `docker run` - Hub 이미지 직접 실행
```bash
# nginx 컨테이너 즉시 실행 (포트 바인딩 포함)
docker run -d -p 80:80 --name my-nginx nginx:1.25-alpine

# PostgreSQL 컨테이너 실행 (환경변수 전달)
docker run -d \
    --name my-postgres \
    -e POSTGRES_USER=myuser \
    -e POSTGRES_PASSWORD=secret \
    -e POSTGRES_DB=mydb \
    -p 5432:5432 \
    postgres:16-alpine

# 일회성 실행 후 자동 삭제 (--rm)
docker run --rm -it node:20-alpine node --version
```
- 이미지를 별도로 pull하지 않아도 `docker run` 실행 시 로컬에 이미지가 없으면 Docker Hub에서 자동으로 pull
- 빠른 테스트, 일회성 작업, 단일 컨테이너 실행에 적합
|   옵션    |   설명    |
|----------|-----------|
|   `-d`    |   백그라운드(detached) 실행   |
|   `-p host:container` |   호스트-컨테이너 포트 바인딩 |
|   `-e KEY=VALUE`  |   환경변수 전달   |
|   `--rm`  |   컨테이너 종료 시 자동 삭제  |
|   `-it`   |   대화형 터미널 연결  |

### `Dockerfile`의 `FROM` - Hub 이미지를 베이스로 커스텀 이미지 빌드
```dockerfile
# Official Image를 베이스로 지정
FROM node:20-alpine

# 작업 디렉토리 설정
WORKDIR /app

# 의존성 복사 (레이어 캐시 활용)
COPY package*.json ./
RUN npm ci --only=production

# 앱 소스 복사
COPY . .

# 포트 노출
EXPOSE 8080

# 실행 명령
CMD ["node", "server.js"]
```
- `Dockerfile`의 `FROM` 지시어는 Docker Hub에서 베이스 이미지를 가져와 그 위에 커스텀 레이어를 쌓음

### 멀티 스테이지 빌드 - 이미지 크기 최적화
```dockerfile
# Stage 1: 빌드 (Hub의 빌드 도구 이미지 사용)
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: 실행 (Hub의 경량 이미지 사용)
FROM node:20-alpine AS runner
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
EXPOSE 8080
CMD ["node", "dist/server.js"]
```
- 빌드 환경과 실행 환경을 분리해 최종 이미지를 가볍게 만듦
- `AS builder`처럼 스테이지에 이름을 붙이면 `--from=builder`로 특정 스테이지의 결과물만 선택적으로 복사할 수 있음
    - 빌드 도구, 소스맵, 테스트 파일 등이 최종 이미지에 포함되지 않아 보안과 성능 모두 개선됨

## 자주 쓰는 명령어
### 기동/종료
```bash
# 백그라운드로 전체 서비스 기동 (이미지 없으면 자동 pull)
docker compose up -d

# 특정 서비스만 기동
docker compose up -d nginx

# 서비스 종료 (컨테이너 삭제, 볼륨은 유지)
docker compose down

# 볼륨까지 함께 삭제 (데이터 초기화)
docker compose down -v
```

### 이미지 관리
```bash
# 최신 이미지로 재배포 (Hub에서 다시 pull 후 재기동)
docker compose pull
docker compose up -d

# 특정 서비스 이미지만 pull
docker compose pull db
```

### 상태 확인/디버깅
```bash
# 실행 중인 서비스 상태 확인
docker compose ps

# 특정 서비스 로그 (실시간)
docker compose logs -f app

# 컨테이너 내부 접속
docker compose exec db psql -U myuser -d mydb

# 리소스 사용량 확인
docker stats
```

### 빌드 관련(커스텀 이미지 사용 시)
```bash
# 이미지 빌드 후 가동
docker compose up -d --build

# 빌드만 수행 (기동 없이)
docker compose build --no-cache app
```

## 주의사항
- 이미지 업데이트가 반영 안될 때
    - `docker compose pull` 후 `up -d`를 실행
    - 같은 태그를 재사용하면 로컬 캐시가 남아 업데이트되지 않을 수 있음
- 포트 충돌
    - `docker compose ps`와 `lsof -i :80`으로 사용 중인 포트를 먼저 확인
- healthcheck 실패로 앱이 뜨지 않을 때
    - `docker compose logs db`로 DB 가동 로그를 확인
    - `interval`, `retries` 값을 조정
