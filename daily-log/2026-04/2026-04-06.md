- JVM GC 동작 원리

## GC(Garbage Collection)란?
- JVM이 힙(Heap) 메모리에서 더 이상 사용되지 않는 객체를 자동으로 탐지하고 제거하는 메모리 관리 메커니즘
- C/C++와 달리 개발자가 직접 `free()` 또는 `delete`를 호출하지 않아도 됨
    - 메모리 누수(Memory Leak)와 댕글링 포인터(Dangling Pointer) 문제를 방지할 수 있음
- GC는 **Stop-The-World(STW)** 이벤트를 수반함
    - GC가 실행되는 동안 애플리케이션 스레드가 일시 정지됨
    - GC 튜닝의 핵심 목표는 이 STW 시간을 최소화하는 것

## JVM 메모리 구조(힙 영역)
> GC를 이해하려면 먼저 힙(Heap)의 구조를 알아야 함

### Heap의 영역 구분
```
+----------------------------+---------------------------+
|        Young Generation    |      Old Generation       |
|  +---------+---+---+       |                           |
|  |  Eden   | S0| S1|       |   (Tenured Space)         |
|  +---------+---+---+       |                           |
+----------------------------+---------------------------+
         Minor GC 대상                Major GC 대상
```

### Young Generation
- 새롭게 생성된 객체가 할당되는 공간
- 대부분의 객체는 금방 접근 불가능한 상태가 됨(약 98% 이상)
    - **약한 세대 가설(Weak Generational Hypothesis)**에 기반
- 세 가지 영역으로 구성됨
    - **Eden 영역** : `new` 키워드로 생성된 객체가 처음 위치하는 곳
    - **Survivor 0 (S0)** : Minor GC 이후 살아남은 객체가 이동
    - **Survivor 1 (S1)** : S0과 번갈아가며 사용됨 (항상 하나는 비어있음)

### Old Generation (Tenured)
- Young 영역에서 오래 살아남은 객체가 이동하는 곳
- Young 영역보다 크게 할당되며, GC 빈도가 낮음

### Metaspace(JDK 8+)
- 클래스 메타데이터를 저장하는 영역
- JDK 7 이하의 PermGen 영역을 대체함
    - PermGen은 힙 크기가 고정되어 `OutOfMemoryError`가 자주 발생했음
    - Metaspace는 네이티브 메모리를 사용하며 자동으로 크기가 조정됨

## GC의 기본 동작 원리
### Reachability(도달 가능성)
- GC는 객체가 **GC Root**로부터 도달 가능한지 여부로 생사를 판단함
- GC Root의 종류
    - 현재 실행 중인 스레드의 스택 변수
    - 정적(static) 변수
    - JNI(Java Native Interface) 참조

### Mark and Sweep
1. **Mark 단계** : GC Root에서 시작해 참조 그래프를 순회하며 살아있는 객체에 표시
2. **Sweep 단계** : 표시되지 않은 객체를 메모리에서 제거
3. **Compact 단계** (선택적) : 메모리 파편화(Fragmentation) 방지를 위해 살아있는 객체를 연속된 공간으로 재배치

## Minor GC(Young GC) 동작 과정
> Young Generation에서 발생하는 GC. 빠르고 자주 실행됨
1. Eden 영역이 꽉 차면 Minor GC 트리거
2. Eden과 현재 사용 중인 Survivor 영역(S0)에서 Mark 수행
3. 살아남은 객체를 빈 Survivor 영역(S1)으로 복사
4. 살아남은 객체의 **Age(나이)**를 1 증가시킴
5. Eden과 S0을 완전히 비움(Sweep)
6. 다음 Minor GC 시에는 S0과 S1의 역할이 바뀜

### 객체의 승격(Promotion)
- 객체의 Age가 특정 임계값(`-XX:MaxTenuringThreshold`, 기본값 15)을 넘으면 Old Generation으로 이동
- Survivor 영역이 가득 차도 Old Generation으로 직접 이동됨

## Major GC/Full GC
> Old Generation에서 발생하는 GC. 느리고 STW 시간이 긺
- **Major GC** : Old Generation만 수집
- **Full GC** : Young + Old + Metaspace 전체를 수집
    - 가장 긴 STW를 유발하므로, 최대한 발생 빈도를 낮추는 것이 중요

## GC 알고리즘의 종류
### Serial GC
```bash
# 활성화 옵션
-XX:+UseSerialGC
```
- 단일 스레드로 GC를 수행
- STW 동안 다른 모든 스레드가 멈춤
- 소규모 애플리케이션이나 싱글 코어 환경에 적합

### Parallel GC(Throughput GC)
```bash
-XX:+UseParallelGC
-XX:ParallelGCThreads=<N>   # GC 스레드 수 지정
```
- Young GC를 여러 스레드가 병렬로 수행
- 처리량(Throughput) 극대화가 목표
- JDK 8의 기본 GC

### CMS GC(Concurrent Mark Sweep)
```bash
-XX:+UseConcMarkSweepGC     # JDK 9+에서 Deprecated
```
- 애플리케이션 스레드와 GC 스레드가 **동시에** 실행됨
- STW 시간을 최소화하여 응답 지연(Latency)을 줄이는 것이 목표
- 단점
    - 메모리 파편화 발생(Compact 미수행)
    - CPU 리소스 소모가 큼
    - JDK 9에서 Deprecated, JDK 14에서 제거됨

### G1 GC(Garbage-First GC)
```bash
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200    # 목표 STW 시간 (ms)
```
- JDK 9부터 기본 GC
- 힙을 고정 크기의 **Region**으로 분할하여 관리
    - 각 Region은 Eden, Survivor, Old, Humongous 역할을 동적으로 담당
    - **Humongous Region** : 단일 객체의 크기가 Region 크기의 50% 이상일 때 사용
- GC 대상 Region을 가비지가 가장 많은 곳(Garbage-First)부터 수집
- 예측 가능한 STW 시간 목표를 설정할 수 있음

### G1GC의 동작 단계
1. **Initial Mark(STW)** : GC Root에서 직접 참조되는 객체를 표시(Minor GC에 편승)
2. **Root Region Scanning** : Survivor 영역에서 Old 영역에 대한 참조를 스캔
3. **Concurrent Mark** : 힙 전체에서 살아있는 객체를 동시(Concurrent)에 마킹
4. **Remark(STW)** : Concurrent Mark 단계에서 누락된 변경 사항을 최종 확정
5. **Cleanup(STW + Concurrent)** : 각 Region의 가비지 비율을 계산하고 빈 Region을 회수
6. **Evacuation(STW)** : 가비지 비율이 높은 Region의 살아있는 객체를 다른 Region으로 복사

### ZGC(Z Garbage Collector)
```bash
-XX:+UseZGC             # JDK 15+에서 Production 준비 완료
-XX:+ZGenerational      # JDK 21+에서 Generational ZGC(권장)
```
- STW 시간을 **1ms 미만**으로 유지하는 것이 목표
- 힙 크기에 관계없이 일정한 저지연을 제공
    - 수 TB의 대용량 힙에도 적용 가능
- **Colored Pointers**와 **Load Barrier** 기술을 사용하여 대부분의 작업을 Concurrent하게 처리

### Shenandoah GC
```bash
-XX:+UseShenandoahGC    # OpenJDK에서 사용 가능
```
- Red Hat이 개발한 저지연 GC
- ZGC와 유사하게 Concurrent Compaction을 지원
- Compact 단계까지 애플리케이션이 실행 중에 처리

## GC 성능 지표 이해
### GC 로그 활성화
```bash
# JDK 9+
-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m
```

### 주요 지표
- **Throughput(처리량)** : 전체 실행 시간 중 GC가 아닌 애플리케이션이 실행된 비율
    - `Throughput = (1 - GC 시간 / 전체 시간) * 100`
- **Latency(지연 시간)** : GC로 인한 STW의 최대$\cdot$평균 시간
- **Footprint(메모리 사용량)** : GC가 사용하는 메모리의 총량
> 처리량, 지연 시간, 메모리 사용량은 트레이드 오프 관계이므로 애플리케이션의 특성에 따라 GC를 선택해야 함

## 실무에서의 GC 튜닝 포인트
### 힙 크기 설정
```bash
-Xms<size>      # 초기 힙 크기  (예 : -Xms2g)
-Xmx<size>      # 최대 힙 크기  (예 : -Xmx4g)
```
- `Xms`와 `Xmx`를 같은 값으로 설정하면 힙 리사이징으로 인한 STW를 방지할 수 있음

### OOM(OutOfMemoryError) 대응
```bash
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/heapdump.hprof
```
- OOM 발생 시 힙 덤프를 자동으로 생성하여 원인 분석에 활용
- **MAT(Memory Analyzer Tool)** 또는 **VisualVM**으로 덤프를 분석

### GC 알고리즘 선택 기준
|   상황    |   권장 GC |
|----------|-----------|
|   배치, 대용량 처리(처리량 우선)  |   Parallel GC |
|   일반적인 서버 애플리케이션  |   G1 GC(기본값)   |
|   응답 속도가 매우 중요한 서비스  |   ZGC/Shenandoah  |
|   대용량 힙(수십 GB 이상) |   ZGC |
