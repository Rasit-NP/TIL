# 게임 서버 ↔ 클라이언트 랜덤 연산 동기화
## 게임에서의 랜덤
- 일반 서버-클라이언트 통신에서 랜덤은 단순히 `Math.random()` 한 번이면 끝
- 게임에서는 클라이언트(렌더링)와 서버(검증)가 동일한 결과를 독립적으로 재현해야 하는 경우가 존재
- 어떤 방식으로 랜덤을 다루느냐에 따라 공정성(Anti-cheat), 레이턴시, 결정론적 재현성, 서버 부하 트레이드오프가 달라짐

## 의사 난수 생성기(PRNG) 기초
### 핵심 원리
- PRNG는 시드(Seed) 값 하나를 넣으면 결정론적(deterministic)으로 수열을 생성함
- 즉, 동일 시드 → 동일 수열이 보장됨
- 이 성질을 이용해 서버와 클라이언트가 시드를 공유하면, 각자 독립적으로 같은 난수 수열을 생성할 수 있음
```
시드: 42
       ↓
[PRNG 상태 머신]
  ↓       ↓       ↓
 0.31   0.78   0.55  ...  (항상 동일한 수열)
```

### 주요 PRNG 알고리즘 비교
|   알고리즘    |   주기    |   속도    |   예측 불가성 |   게임 사용   |
|-------------|-----------|-----------|-------------|-------------|
|   Linear Congruential(LCG)    |   중간    |   매우 빠름   |   낮음    |   단순 게임 이펙트    |
|   Mersenne Twister(MT19937)   |   $2^{22357}$ − 1 |   빠름    |   중간    |   일반적인 게임 로직  |
|   Xorshift128+    |   $2^{128}-1$ |   매우 빠름   |   중간    |   고성능 실시간 게임  |
|   PCG(Permuted Congruential)  |   $2^{128}$   |   빠름    |   높음    |   공정성이 중요한 아이템 시스템   |
|   Cryptographic(CSPRNG)   |   -   |   느림    |   매우 높음   |   도박·확률형 아이템  |
> ⚠️ 게임에서 보안이 필요한 랜덤(확률형 아이템 드롭, 온라인 도박)과 단순 이펙트용 랜덤은 반드시 분리해서 설계해야 함

## 방식1 — 서버 권위 방식(Server-Authoritative)
### 개념
- 모든 랜덤 결과를 서버에서만 생성하고, 결과값을 클라이언트에 전달함
- 클라이언트는 수신한 값을 렌더링에만 사용하며, 직접 난수를 생성하지 않음
```
클라이언트                    서버
    |  --- "스킬 사용" 요청 -->  |
    |                           |  random() → 크리티컬 여부 결정
    |  <-- "크리티컬 발생" 응답 - |
    |  (이펙트 렌더링)            |
```

### 장단점
|   항목    |   내용    |
|----------|-----------|
|   공정성  |   서버에서만 생성하므로 조작 불가 |
|   일관성  |   클라이언트 상태와 무관하게 결정론적 |
|   레이턴시    |   왕복 지연(RTT)이 결과에 영향    |
|   서버 부하   |   모든 랜덤 연산이 서버 집중  |

### 적합한 상황
- 아이템 드롭, 크리티컬, 가챠 등 결과의 정확성이 최우선인 경우
- 레이턴시 민감도가 낮은 턴제 게임

## 방식2 — 시드 공유 방식(Shared Seed/Deterministic Simulation)
### 개념
- 서버가 게임 시작 시 시드를 생성하여 클라이언트에 전달하고, 이후 양쪽이 동일한 PRNG를 독립적으로 실행함
- 입력 이벤트만 동기화하면 클라이언트가 랜덤 결과를 로컬에서 직접 계산 가능함
```
[게임 시작]
서버: seed = generateSecureSeed()
서버 → 클라이언트: { seed: 8472901 }

[이후]
서버 PRNG(seed=8472901): 0.31, 0.78, 0.55 ...
클라이언트 PRNG(seed=8472901): 0.31, 0.78, 0.55 ...  ← 완전히 동일
```

### 구현 시 핵심 주의사항
- 서버와 클라이언트가 완전히 동일한 PRNG 구현을 사용해야 함
    - 언어(C++, Java, JavaScript)마다 기본 `random()`의 구현이 다름
    - 반드시 동일 알고리즘(예: Xorshift128+)을 직접 구현하거나 공통 라이브러리를 사용해야 함
- PRNG 호출 순서(call sequence)가 서버와 클라이언트에서 완전히 일치해야 함
    - 조건 분기로 한 쪽에서 랜덤 호출을 건너뛰면 그 시점부터 수열이 어긋남
```java
// ❌ 문제 상황: 서버에서만 특정 조건에서 random() 추가 호출
if (serverOnlyCondition) {
    double extra = rng.nextDouble(); // 클라이언트는 이 호출이 없음
}
// 이후 모든 random() 결과가 서버와 클라이언트에서 1칸씩 어긋남
```
```java
// ✅ 해결: 공통 게임 로직 외부의 랜덤은 별도 RNG 인스턴스 사용
private final Xorshift128Plus gameRng   = new Xorshift128Plus(sharedSeed);
private final Xorshift128Plus effectRng = new Xorshift128Plus(System.nanoTime()); // 독립
```

### 장단점
|   항목    |   내용    |
|----------|-----------|
|   레이턴시    |   로컬에서 계산하므로 즉각적  |
|   서버 부하   |   랜덤 연산 부하 분산 |
|   구현 복잡도 |   양쪽 PRNG 동기화 유지가 어려움  |
|   보안    |   시드가 노출되면 미래 난수 예측 가능 |

### 적합한 상황
- RTS(실시간 전략), 격투 게임 등 결정론적 시뮬레이션이 필요한 게임
- 리플레이 시스템 구현(입력 + 시드만 저장하면 완전 재현 가능)

## 방식3 — 클라이언트 예측 + 서버 검증(Client Prediction)
### 개념
- 클라이언트가 먼저 로컬에서 랜덤 결과를 예측하여 즉시 렌더링하고, 서버의 최종 결과가 도착하면 필요 시 보정(rollback)함
- FPS, 액션 RPG 등 레이턴시에 극도로 민감한 장르의 표준 패턴
```
클라이언트                          서버
    |  공격 입력                      |
    |  → 로컬 RNG → 크리티컬 예측      |
    |  → 이펙트 즉시 렌더링            |
    |                                 |
    |  --- 입력 이벤트 전송 ---------->|
    |                                 |  서버 RNG → 최종 결정
    |  <-- 서버 최종 결과 -------------|
    |                                 |
    | (예측과 다르면 보정 렌더링 수행)  |
```

### 롤백(Rollback) 처리
- 서버 결과가 클라이언트 예측과 다른 경우, 해당 시점으로 되돌아가 다시 시뮬레이션하는 **롤백 넷코드(Rollback Netcode)**를 사용
- 예측 성공률이 높을수록 사용자 체감 품질이 향상됨
```
타임라인:
  T=0: 클라이언트 입력 + 예측 렌더링
  T=50ms: 서버 결과 수신 → 예측 일치 → 아무것도 안 함
  T=50ms: 서버 결과 수신 → 예측 불일치 → T=0으로 롤백 후 재시뮬레이션
```

### 장단점
|   항목    |   내용    |
|   반응성  |   즉각적인 피드백, 높은 플레이 체감   |
|   구현 복잡도 |   롤백 매커니즘, 상태 스냅샷 관리 필요    |
|   보안    |   클라이언트 예측을 치트로 악용 가능 → 서버 최종 검증 필수    |
|   대역폭  |   입력 이벤트 송수신 오버헤드

## 방식 4 — 커밋-리빌 방식(Commit-Reveal)
### 개념
- 공정성이 극도로 중요한 상황(카드게임, 도박)에서 서버와 클라이언트 어느 쪽도 상대방의 랜덤 값을 미리 알 수 없도록 설계하는 암호학적 프로토콜
```
1. 서버: secret = secureRandom()
         commitment = SHA256(secret)
         클라이언트에게 commitment 전송

2. 클라이언트: clientSeed = secureRandom()
               서버에게 clientSeed 전송

3. 서버: secret 공개
         클라이언트 검증: SHA256(secret) == commitment ✓

4. 최종 시드: finalSeed = XOR(secret, clientSeed)
              (양쪽 모두 조작 불가)
```

- 서버가 commitment를 미리 보냈기 때문에 클라이언트의 seed를 본 두 secret을 조작하는 것이 불가능함
- 클라이언트도 서버의 secret을 미리 알 수 없어 자신의 seed를 유리하게 조작할 수 없음

### 적합한 상황
- 온라인 카드게임의 덱 셔플
- 확률형 아이템 시스템의 투명성 보장
- 블록체인 게임의 온체인 랜덤

## 방식별 비교 요약
|   방식    |   레이턴시    |   공정성  |   구현 난이도 |   주요 사용처 |
|----------|--------------|----------|--------------|-------------|
|   서버 권위   |   높음(RTT 의존)  |   최상    |   낮음    |   가챠, 턴제 RPG  |
|   시드 공유   |   낮음    |   높음    |   중간    |   RTS, 리플레이 시스템    |
|   클라이언트 예측 |   매우 낮음   |   중간    |   높음    |   FPS, 액션 게임  |
|   커밋-리빌   |   중간    |   최상    |   높음    |   카드게임, 확률형 아이템 |

## 부정행위 방지(Anti-cheat) 고려사항
- 클라이언트는 항상 신뢰할 수 없음을 전제로 설계해야 함
- 시드 공유 방식에서 시드가 노출되면 미래 난수 수열을 완전히 예측할 수 있어 어뷰징이 가능해 짐
    - 시드를 직접 노출하지 않고 서버가 순차적으로 결과만 검증하는 방식을 혼용하는 것이 권장됨
- 클라이언트 예측 방식에서는 로컬 RNG를 조작하는 메모리 패치가 가능하므로 서버의 최종 결과가 반드시 권위를 가져야 함
> ⚠️ 확률형 아이템(가챠)은 법적 규제 대상이므로 CSPRNG와 커밋-리빌 방식을 사용하고, 결과 로그를 서버에 영구 보관해야 함
