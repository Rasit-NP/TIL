## C++ 특수 멤버 함수 (Special Member Function)
- 컴파일러가 자동으로 생성해주는 멤버 함수들을 특수 멤버 함수라고 함
- 개발자가 명시적으로 정의하지 않으면 컴파일러가 기본 버전을 생성함

|   함수    |   호출 시점   |
|----------|--------------|
|   기본 생성자 |   `Matrix m;` |
|   복사 생성자 |   `Matrix b = a;` 함수 인자로 값 전달 |
|   이동 생성자 |   `Matrix b = std::move(a);` 함수에서 임시 객체 반환  |
|   복사 대입 연산자    |   `b = a;`    |
|   이동 대입 연산자    |   `b = std::move(a);` |
|   소멸자  |   객체 수명 종료 시   |

## 복사 생성자(Copy Constructor)
- 기존 객체와 동일한 값을 가진 새 객체를 생성할 때 호출됨
- 인자로 반드시 `const T&` (lvalue 참조)를 받음
- 기본 복사 생성자는 얕은 복사(Shallow Copy)를 수행함
    - 멤버를 그대로 복사하므로, Heap을 가리키는 포인터가 있으면 두 객체가 같은 메모리를 공유하게 되어 이중 해제(Double Free) 문제가 발생함
```cpp
class Matrix {
public:
    int rows, cols;
    double* data;   // Heap 메모리를 직접 소유

    // 복사 생성자: 깊은 복사(Deep Copy)
    Matrix(const Matrix& other)
        : rows(other.rows), cols(other.cols) {
            int size = rows * cols;
            data = new double[size];                        // 새로운 Heap 블록 할당
            std::copy(other.data, other.data + size, data); // 값 복사
    }
}
```

### 복사 생성자가 호출되는 세 가지 상황
```cpp
void take_by_value(Matrix m){ ... }     // 1. 함수 인자로 값 전달
Matrix make() { ... }                   // 2. 함수에서 값 반환 (RVO 미적용 시)

Matrix a(3, 3);
Matrix b = a;                           // 3. 복사 초기화
take_by_value(a);                       // 4. 1번 케이스: a를 복사해 m 생성
```
> ⚠️ 얕은 복사 함정: 포인터 멤버가 있는 클래스에서 복사 생성자를 정의하지 않으면, 두 객체가 같은 Heap을 가리키게 됨
>
> 하나가 소멸자에서 `delete`를 호출하면 나머지 객체의 포인터는 해제된 메모리를 가리키는 댕글링 포인터(Dangling Pointer)가 됨

## lvalue와 rvalue
- 이동 시맨틱을 이해하려면 값 카테고리 구분이 필수임

### lvalue (Left Value)
- 이름이 있고, 주소를 얻을 수 있는 값
- 표현식이 끝난 후에도 살아있음
- 예시 : 변수, 배열 원소, 참조 반환 함수

### rvalue (Right Value)
- 이름이 없고, 주소를 얻을 수 없는 임시 값
- 표현식이 끝나면 사라짐
- 예시 : 리터럴(42, 3.14), 임시 객체, std::move(x)

```cpp
int a = 10;
// a : lvalue
// 10 : rvalue (리터럴, 임시 값)

Matrix m(3, 3);
// m : lvalue
// Matrix(3, 3) : rvalue (임시 객체)
```

## 이동 생성자(Move Constructor)
- C++11에서 도입된 핵심 최적화 기법
- 곧 사라질 임시 객체(rvalue)에서 자원을 복사하는 대신 소유권을 탈취(steal) 해옴
- 인자로 `T&&` (rvalue 참조)를 받음
- Heap 재할당 없이 포인터만 이전하므로 $O(1)$ 비용

```cpp
class Matrix{
public:
    int rows, cols;
    double* data;

    // 이동 생성자: 자원 소유권 탈취
    Matrix(Matrix&& other) noexcept
        : rows(other.rows), cols(other.cols), data(other.data){
        // 원본 객체를 "빈 상태"로 만들어 이중 해제를 방지
        other.data = nullptr;
        other.rows = 0;
        other.cols = 0;
    }

    ~Matrix() {
        delete[] data;  // nullptr에 대한 delete는 안전함
    }
}
```

### 복사 vs. 이동 비교
```cpp
Matrix a(1000, 1000);   // 8MB Heap 할당

// 복사: 8MB를 새로 할당하고 값을 전부 memcpy
Matrix b = a;

// 이동: 포인터 하나만 이전, Heap 재할당 없음
Matrix c = std::move(a);
// 이후 a.data == nullptr, a는 더 이상 사용하면 안됨
```

|   항목    |   복사 생성자 |   이동 생성자 |
|----------|--------------|--------------|
|   비용    |   $O(N)$ — 데이터 전체 복사   |   $O(1)$ — 포인터 이전만  |
|   원본 상태   |   원본 유지   |   원본이 빈 상태가 됨 |
|   Heap 할당   |   새로 할당   |   없음    |
|   사용 조건   |   lvalue  |   rvalue(임시 객체, std::move)    |

## RVO/NRVO (Return Value Optimization)
- 함수에서 객체를 반환할 때 불필요한 복사/이동을 컴파일러가 제거하는 최적화 기법

### RVO (Return Value Optimization)
- 함수가 이름 없는 임시 객체를 반환할 때 적용됨
```cpp
Matrix make_identity(int n){
    return Matrix(n, n);        // 임시 객체 직접 반환
    // ← RVO: 호출자 측의 변수 공간에 직접 생성, 복사/이동 0회
}

Matrix m = make_identity(3);
// 복사 생성자도, 이동 생성자도 호출되지 않음
// Matrix(3, 3)이 m의 메모리에 직접 생성됨
```

### NRVO (Named Return Value Optimization)
- 함수 내부에서 이름 있는 지역 변수를 반환할 때 적용됨
```cpp
Matrix make_result(int n){
    Matrix res(n, n);       // 이름 있는 지역 변수
    // ... res를 채우는 작업 ...
    return res;             // NRVO: res가 호출자 공간에 직접 생성될 수 있음
}
```

### RVO 동작 원리
```
RVO 적용 전 (개념적 흐름)

[make_identity 스택 프레임]
    Matrix temp(n, n);   ← Heap 할당
    return temp;         ← 호출자 변수로 이동 생성자 호출 후 temp 소멸

RVO 적용 후 (실제 동작)

[호출자 스택 프레임]
    Matrix m;            ← m의 메모리 공간을 미리 확보
         ↑
[make_identity]
    m에 직접 Matrix(n, n) 생성  ← 복사/이동 없음
```
> ⚠️ RVO를 방해하는 패턴: `return std::move(res);`처럼 명시적으로 `std::move`를 쓰면 오히려 NRVO가 억제되어 성능이 저하될 수 있음
>
> 지역 변수를 반환할 때는 그냥 `return res;`로 작성하는 것이 올바름

## RAII(Resource Acquisition Is Initialization)
- 자원의 획득은 초기화 시, 자원의 해제는 소멸 시에 수행한다는 C++의 핵심 관용구
- 생성자에서 자원을 획득하고, 소멸자에서 반드시 해제함
- 스택 언와인딩(Stack Unwinding) 메커니즘과 결합하여 예외가 발생해도 자원이 반드시 해제됨

```cpp
// RAII를 직접 구현한 예시
class Matrix {
public:
    double* data;

    // 생성자: 자원 획득
    Matrix(int rows, int cols){
        data = new double[rows * cols]; // Heap 할당
    }

    // 소멸자: 자원 해제(자동 호출됨)
    ~Matrix(){
        delete[] data;
    }
}

void process(){
    Matrix m(1000, 1000);   // Heap 8MB
    // ... 작업 수행 ...
    // 예외가 발생해도,
    // 함수가 정상 종료돼도,
    // ← 여기서 ~Matrix() 자동 호출 → Heap 자동 해제
}
```

### RAII를 구현한 표준 라이브러리 타입들
|   타입    |   획득하는 자원   |   해제 방식   |
|----------|------------------|-------------|
|   `std::vector<T>`    |   Heap 메모리 |   소멸자에서 `delete[]`   |
|   `std::unique_ptr<T>` |   Heap 객체 소유권    |   소멸자에서 `delete` |
|   `std::shared_ptr<T>`    |   Heap 객체 참조 카운트   |   참조 카운트가 0이 될 때 `delete`    |
|   `std::lock_guard`   |   뮤텍스 잠금 |   소멸자에서 `unlock()`   |
|   `std::ifstream` |   파일 핸들   |   소멸자에서 파일 닫기    |

```cpp
// RAII 덕분에 안전한 코드
void safe_process(){
    std::unique_ptr<Matrix> m = std::make_unique<Matrix>(1000, 1000);

    if (some_condition){
        throw std::runtime_error("error!");
        // ← 예외 발생 → 스택 언와인딩 → unique_ptr 소멸자 → Heap 해제
    }
}   // 예외가 없어도 여기서 소멸자 자동 호출
```
> ⚠️ new/delete를 직접 쓰면 안 되는 이유: `new`로 할당 후 `delete` 호출 사이에 예외가 발생하면 메모리가 영원히 누수됨
>
> RAII를 따르는 `unique_ptr`이나 `vector`를 사용하면 이 문제를 원천 차단할 수 있음

## 정리
```
C++ 객체 모델
    │
    ├── 복사 생성자
    │     ├── const T& 인자 (lvalue)
    │     ├── 깊은 복사 → O(N) 비용
    │     └── 포인터 멤버 → 이중 해제 위험 (직접 구현 필수)
    │
    ├── 이동 생성자
    │     ├── T&& 인자 (rvalue)
    │     ├── 포인터 탈취 → O(1) 비용
    │     ├── noexcept 명시 필요 (vector 재할당 최적화)
    │     └── 원본을 nullptr로 초기화 (이중 해제 방지)
    │
    ├── RVO / NRVO
    │     ├── 컴파일러가 반환 시 복사/이동을 제거
    │     ├── return std::move(x) 는 오히려 NRVO 억제
    │     └── C++17 이후 RVO는 의무화(Guaranteed Copy Elision)
    │
    └── RAII
          ├── 생성자에서 자원 획득
          ├── 소멸자에서 자원 해제 (자동)
          ├── 스택 언와인딩 → 예외 안전성 보장
          └── vector, unique_ptr, lock_guard가 구현체
```
