# Spring Security - JWT 인증 흐름
- Spring Boot 기반 API 서버에서 Stateless 인증을 구현할 때 가장 널리 사용되는 방식
- Spring Security의 필터 체인 위에서 JWT를 발급$\cdot$검증$\cdot$재발급하는 전체 흐름을 다룸

## SecurityFilterChain 구조
- Spring Security는 서블릿 필터 체인(Servlet Filter Chain) 위에서 동작함
- 요청이 들어오면 아래 순서로 처리됨
```
클라이언트 요청
     ↓
[ DelegatingFilterProxy ]   ← 서블릿 컨테이너의 필터. 처리를 Spring Bean에 위임
     ↓
[ FilterChainProxy ]        ← ApplicationContext에 등록된 Bean. SecurityFilterChain의 진입점
     ↓
[ SecurityFilterChain ]     ← 실제 보안 필터들의 묶음. 여러 개 등록 가능
     ↓
  Filter1 → Filter2 → ... → FilterN
     ↓
  DispatcherServlet
```
- **DelegatingFilterProxy**
    - 서블릿 컨테이너에 등록되며 Spring ApplicationContext의 Bean에게 실제 처리를 위임함
- **FilterChainProxy**
    - 여러 `SecurityFilterChain` 중 요청의 `requestMatcher`에 맞는 것을 골라 실행
- **SecurityFilterChain**
    - URL 패턴별로 적용될 필터 목록을 정의하며, 여러 개를 등록해 경로별로 다른 보안 정책을 적용할 수 있음

### 주요 내장 필터
|   순서    |   필터    |   역할    |
|----------|-----------|----------|
|   1   |   `SecurityContextHolderFilter`   |   SecurityContext 초기화 및 저장  |
|   2   |   `UsernamePasswordAuthenticationFilter`  |   폼 로그인 처리  |
|   3   |   `ExceptionTranslationFilter`    |   인증/인가 예외를 HTTP 응답으로 변환 |
|   4   |   `AuthorizationFilter`   |   인가 처리   |

- JWT 커스텀 필터는 `UsernamePasswordAuthenticationFilter` 이전에 삽입하는 것이 일반적
    - 해당 필터가 실행되기 전에 이미 `SecurityContext`에 인증 정보를 채워 놓아야 이후 필터들이 인증된 요청으로 처리하기 때문

### SecurityFilterChain 등록
```java
@Configuration
@EnableWebSecurity
public class SecurityConfig{

    private final JwtAuthenticationFilter jwtAuthFilter;

    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception{
        http
            .csrf(AbstractHttpConfigurer::disable)
            .sessionManagement(session ->
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS)
            )
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/auth/**").permitAll()
                .anyRequest().authenticated()
            )
            .addFilterBefore(jwtAuthFilter, UsernamePasswordAuthenticationFilter.class);
        
        return http.build();
    }
}
```
- `STATELESS` 정책을 설정하면 Spring Security가 세션을 생성하거나 사용하지 않음
- `addFilterBefore`로 커스텀 JWT 필터를 `UsernamePasswordAuthenticationFilter` 앞에 삽입

## JWT 발급$\cdot$검증$\cdot$재발급 흐름
### JWT 구조
```
Header.Payload.Signature

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9        ← Base64Url(Header)
.eyJzdWIiOiJ1c2VyMSIsInJvbGUiOiJVU0VSIn0    ← Base64Url(Payload)
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQ     ← HMAC-SHA256(Base64Url(Header) + "." + Base64Url(Payload), secret)
```
- **Header**
    - 서명 알고리즘(`alg`)과 토큰 타입(`typ`)을 담음
- **Payload**
    - Claims를 담음
    - `sub`(주제), `iat`(발급 시각), `exp`(만료 시각), `role` 등
    - 민감 정보(비밀번호 등)는 절대 포함하지 않음
        - Base64로 인코딩된 것일 뿐 암호화가 아니기 때문
- **Signature**
    - 서버만 알고 있는 `secret` 또는 비대칭 키(RSA, EC)로 서명해 위변조를 방지

### Access Token 발급 흐름
```
[클라이언트]                         [서버]
    |                                   |
    |-- POST /api/auth/login ----------→|
    |   { id, password }                |
    |                                   |-- UserDetailsService.loadUserByUsername()
    |                                   |-- PasswordEncoder.matches()
    |                                   |-- JwtService.generateAccessToken()
    |                                   |-- JwtService.generateRefreshToken()
    |                                   |-- RefreshToken DB/Redis 저장
    |←-- 200 OK ------------------------|
    |   { accessToken, refreshToken }   |
```

### Access Token 검증 흐름
```
[클라이언트]                         [JwtAuthenticationFilter]
    |                                   |
    |-- GET /api/data -----------------→|
    |   Authorization: Bearer <token>   |
    |                                   |-- Authorization 헤더에서 토큰 추출
    |                                   |-- JwtService.validateToken()
    |                                   |    ├─ 서명 검증
    |                                   |    ├─ 만료 여부 확인 (exp claim)
    |                                   |    └─ 토큰 타입 확인 (ACCESS)
    |                                   |-- UserDetails 로드
    |                                   |-- SecurityContextHolder에 Authentication 저장
    |                                   |-- filterChain.doFilter() → 다음 필터로 진행
    |←-- 200 OK ------------------------|
```

### Refresh Token 재발급 흐름(RTR 전략)
```
[클라이언트]                         [서버]
    |                                   |
    |-- POST /api/auth/reissue --------→|
    |   { refreshToken }                |
    |                                   |-- JwtService.validateToken()
    |                                   |-- DB 저장 토큰과 비교
    |                                   |    └─ 불일치 → 토큰 탈취로 판단
    |                                   |         → 해당 유저 전체 토큰 무효화
    |                                   |-- 새 AccessToken 생성
    |                                   |-- 새 RefreshToken 생성 + DB 갱신
    |←-- 200 OK ------------------------|
    |   { accessToken, refreshToken }   |
```
- **RTR(Refresh Token Rotation)**
    - Refresh Token을 사용할 때마다 새것을 발급하고 기존 것을 폐기
    - 이미 사용된 Refresh Token으로 재발급 요청이 들어오면 탈취로 간주해 해당 유저의 모든 토큰을 무효화

### JwtService 구현
```java
@Service
@RequiredArgsConstructor
public class JwtService{

    @Value("${jwt.secret}")
    private String secretKey;

    private static final long ACCESS_TOKEN_EXPIRE = 1000*60*30;
    private static final long REFRESH_TOKEN_EXPIRE = 1000*60*60*24*14;

    private Key getSigningKey(){
        byte[] keyBytes = Decoders.BASE64.decode(secretKey);
        return Keys.hmacShaKeyFor(keyBytes);
    }

    public String generateAccessToken(UserDetails userDetails){
        return Jwts.builder()
                .setSubject(userDetails.getUsername())
                .claim("type", "ACCESS")
                .claim("role", userDetails.getAuthorities())
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + ACCESS_TOKEN_EXPIRE))
                .signWith(getSigningKey(), SignatureAlgorithm.HS256)
                .compact();
    }

    public Claims parseClaims(String token){
        return Jwts.parserBuilder()
                .setSigningKey(getSigningKey())
                .build()
                .parseClaimsJws(token)
                .getBody();
    }

    public boolean validateToken(String token){
        try{
            parseClaims(token);
            return true;
        } catch (ExpiredJwtException e){
            throw new CustomException(ErrorCode.TOKEN_EXPIRED);
        } catch (JwtException | IllegalArgumentException e){
            throw new CustomException(ErrorCode.TOKEN_INVALID);
        }
    }

    public String extractUsername(String token){
        return parseClaims(token).getSubject();
    }
}
```

## OncePerRequestFilter 커스텀 구현
- 일반 `Filter` 또는 `GenericFilterBean`은 서블릿 **forward/include** 상황에서 동일 요청에 대해 필터가 여러 번 실행될 수 있음
- `OncePerRequestFilter`는 `HttpServletRequest`에 플래그 속성을 심어 하나의 HTTP 요청 당 딱 한 번만 실행됨을 보장

### JwtAuthenticationFilter 구현
```java
@Component
@RequiredArgsConstructor
public class JwtAuthenticationFilter extends OncePerRequestFilter{
    
    private final JwtService jwtService;
    private final UserDetailsService userDetailsService;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain
    ) throws ServletException, IOException{

        String authHeader = request.getHeader("Authorization");

        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            // 토큰 없으면 통과 - 이후 AuthorizationFilter가 인가를 차단
            filterChain.doFilter(request, response);
            return;
        }

        String token = authHeader.substring(7);

        try{
            if (jwtService.validateToken(token)){
                String username = jwtService.extractUsername(token);

                // SecurityContext에 인증 정보가 없을 때만 처리
                if (username != null && SecurityContextHolder.getContext().getAuthentication() == null){
                    UserDetails userDetails = userDetailsService.loadUserByUsername(username);

                    UsernamePasswordAuthenticationToken authToken =
                        new UsernamePasswordAuthenticationToken(
                            userDetails,
                            null,
                            userDetails.getAuthorities()
                        );
                    
                    authToken.setDetails(
                        new WebAuthenticationDetailSource().buildDetails(request)
                    );

                    SecurityContextHolder.getContext().setAuthentication(authToken);
                }
            }
        } catch (CustomException e){
            SecurityContextHolder.clearContext();
            response.sendError(HttpServletResponse.SC_UNAUTHORIZED, e.getMessage());
            return;
        }

        filterChain.doFilter(request, response);
    }

    @Override
    protected boolean shouldNotFilter(HttpServletRequest request){
        // 로그인, 회원가입 등 인증이 필요 없는 경로는 제외
        return request.getServletPath().startsWith("/api/auth/");
    }
}
```

### SecurityContextHolder와 ThreadLocal
- `SecurityContextHolder`는 기본적으로 ThreadLocal 기반
    - 각 요청 스레드는 독립된 `SecurityContext`를 가짐
    - 요청 처리가 완료되면 자동으로 비워짐
- `@Async` 등 비동기 처리 시 주의
    - SecurityContext가 자식 스레드로 자동 전파되지 않음

```java
// 방법 1: SecurityContextHolder 전략 변경
SecurityContextHolder.setStrategyName(SecurityContextHolder.MODE_INHERITABLETHREADLOCAL);

// 방법 2: DelegatingSecurityContextExecutorService 사용
@Bean
public Executor taskExecutor(){
    return new DelegatingSecurityContextExecutorService(Executors.newFixedThreadPool(10));
}
```

## Spring AOP/Proxy와 Security 필터의 관계
### 실행 위치 비교
```
[HTTP 요청]
     ↓
[ Spring Security Filter Chain ]   ← 서블릿 레벨. DispatcherServlet 진입 전 동작
     ↓
[ DispatcherServlet ]
     ↓
[ Spring AOP / Proxy ]             ← 스프링 컨텍스트 레벨. Bean 메서드 호출 시 동작
     ↓
[ Controller / Service ]
```
|   구분    |   Spring Security Filter  |   Spring AOP  |
|----------|---------------------------|---------------|
|   실행 위치   |   DispatcherServlet 이전  |   Spring Bean 메서드 호출 시  |
|   적용 대상   |   모든 HTTP 요청  |   Spring Bean 메서드  |
|   주요 용도   |   인증(Authentication) + HTTP 레벨 인가(Authorization)    |   메서드 레벨 인가(Authorization) |
|   예시    |   JWT 검증, 세션 처리 |   `@PreAuthorize`, `@Secured` |

### `@PreAuthorize` 동작 원리
- `@EnableMethodSecurity`를 활성화하면 Spring AOP가 해당 어노테이션이 붙은 메서드를 프록시로 감쌈
```java
@Service
public class ArticleService{

    @PreAuthorize("hasRole('ADMIN') or #username == authentication.principal.username")
    public void deleteArticle(Long id, String username){
        // ...
    }
}
```
- 실행 흐름
```
ArticleController.delete()
     ↓
[AOP Proxy 인터셉트]
     ↓
AuthorizationManagerBeforeMethodInterceptor
     ↓
SecurityContextHolder에서 Authentication 조회
     ↓
SpEL 표현식 평가
     ↓                              ↓
  (통과)                          (실패)
실제 메서드 실행           AccessDeniedException
                                    ↓
                        ExceptionTranslationFilter
                                    ↓
                               403 응답 반환
```
- **self-invocation** 주의
    - 같은 클래스 내에서 메서드를 직접 호출하면 AOP 프록시를 거치지 않아 `@PreAuthorize`가 동작하지 않음
    - 해당 메서드를 별도 Bean으로 분리하거나 `ApplicationContext`에서 직접 꺼내 호출해야 함

### CGLIB vs. JDK 동적 프록시
```java
// Spring Boot 2.x 이전 기본 동작: 인터페이스가 있으면 → JDK 동적 프록시
public interface ArticleService { ... }
public class ArticleServiceImpl implements ArticleService { ... }

// Spring Boot 2.x 이전 기본 동작: 인터페이스가 없으면 → CGLIB (클래스 직접 상속)
public class ArticleService { ... }
```
- Spring Boot 2.x 이후 기본값이 CGLIB으로 통일되어 인터페이스 유무와 무관하게 일관된 동작이 보장됨
