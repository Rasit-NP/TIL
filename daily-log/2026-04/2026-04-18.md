# B-Tree와 B+Tree

## 인덱스(Index)
- 데이터베이스에서 특정 컬럼의 값으로 레코드를 빠르게 찾기 위한 자료구조
- 책의 색인(목차)과 동일한 원리
    - 전체 페이지를 읽지 않고 색인에서 위치를 찾아 바로 이동
- 인덱스가 없다면 DB는 테이블 전체를 처음부터 끝까지 읽는 **Full Table Scan**을 수행함
    - 1억 건 중 1건을 찾기 위해 최악의 경우 1억 번의 디스크 I/O가 발생

### 인덱스가 없을 때의 검색(**Full Table Scan**)
```
SELECT * FROM users WHERE email = 'kim@example.com';

[디스크] → 페이지1(읽기) → 페이지2(읽기) → ... → 페이지N(읽기)
                                                      ↑ 찾을 때까지 계속
```

### 인덱스가 있을 때의 검색
```
SELECT * FROM users WHERE email = 'kim@example.com';

[인덱스 트리] → 루트 → 내부 노드 → 리프 노드 → 디스크 페이지 직접 이동
                  (O(log N)번의 비교만으로 위치 특정)
```
> 인덱스는 쓰기 속도와 저장 공간을 희생하고 읽기 속도를 얻는 트레이드 오프

## 이진 탐색 트리(BST)
- 인덱스 자료구조로 BST를 쓰지 않는 이유를 이해해야 B-Tree의 탄생 배경이 보임

### BST의 문제점
```
균형 잡힌 BST (이상적)          편향된 BST (최악의 경우)
         50                              10
        /  \                              \
       30   70                            20
      / \   / \                            \
     20  40 60  80                          30
                                             \
                                              40   ← 사실상 Linked List
```
|   문제    |   내용    |
|----------|-----------|
|   디스크 I/O 비효율   |   노드 하나 = 포인터 한 개, 트리 깊이만큼 디스크를 읽어야 함  |
|   편향 가능성 |   데이터 삽입 순서에 따라 $O(N)$ 검색으로 퇴화    |
|   캐시 비효율 |   노드 간 메모리 주소가 분산되어 CPU 캐시 적중률이 낮음   |
- 디스크는 블록(페이지) 단위로 데이터를 읽기 때문에, 노드 하나에 하나의 값만 저장하는 BST는 디스크 I/O 횟수가 너무 많음
- 이 문제를 해결하기 위해 하나의 노드에 여러 키를 저장하는 B-Tree가 등장

## B-Tree
### 정의
- **Balanced Tree**의 줄임말
- **M-way** 탐색 트리의 일종으로, 하나의 노드가 최대 M개의 자식 노드를 가짐
- 모든 리프 노드가 같은 깊이에 위치하는 완전 균형 트리
- 항상 $O(\log{N})$ 검색을 보장

### 구조
```
                    [30 | 70]                   ← 루트 노드
                   /    |    \
          [10|20] [40|50|60] [80|90]            ← 내부 노드
          /  |  \    | | |     |  \
        ...  ... ...          ...  ...          ← 리프 노드
```
- 각 노드는 **키(Key)**와 **포인터(Pointer)**를 함께 저장
- 키를 기준으로 자식 포인터의 방향이 결정됨

### 노드 구조(차수 M=4일 때)
```
┌─────────────────────────────────────────┐
│ P0 │ K1 │ P1 │ K2 │ P2 │ K3 │ P3 │ Data │
└─────────────────────────────────────────┘
  ↑     ↑     ↑                  ↑
포인터  키   포인터               포인터
(K1보다 (K1~K2 (K3보다
 작은    사이)  큰 값)
  값)
```
- `P0`: K1보다 작은 값을 가진 자식 노드를 가리키는 포인터
- `Data`: 해당 키에 대응하는 실제 레코드 또는 레코드 주소

### 규칙(차수 M일 때)
|   규칙    |   내용    |
|----------|-----------|
|   키 개수 |   최소 ⌈M/2⌉ − 1개, 최대 M − 1개  |
|   자식 수 |   최소 ⌈M/2⌉개, 최대 M개  |
|   균형    |   모든 리프 노드는 동일한 깊이    |
|   정렬    |   각 노드 내 키는 오름차순 정렬   |

### 검색 과정 예시
```
목표: 키 45 검색 (루트에 [30|70]이 있다고 가정)

Step 1. 루트 노드 읽기
        [30 | 70]
         30 < 45 < 70 → 가운데 포인터 따라가기

Step 2. 내부 노드 읽기
        [40 | 50 | 60]
         40 < 45 < 50 → 두 번째 포인터 따라가기

Step 3. 리프 노드에서 45 발견 → 해당 레코드 반환
```
- 디스크 읽기 횟수 = 트리의 깊이 = $O(\log_M N)$
- M이 클수록(노드 하나에 키가 많을수록) 트리가 납작해져서 디스크 I/O가 줄어듦

### 삽입과 분할(Split)
- 노드가 꽉 찼을 때 중간 키를 부모로 올리고 노드를 둘로 분리함
```
[10 | 20 | 30 | 40] ← 차수 4, 이미 가득 참. 35 삽입 시도

       [30]                  ← 중간 키 30을 부모로 올림
      /     \
[10|20]    [35|40]           ← 분할
```

## B+Tree
### 정의
- B-Tree를 개선한 변형 구조
- 내부 노드는 키만 저장하고, 실제 데이터(레코드 포인터)는 리프 노드에만 저장
- 리프 노드들이 **연결 리스트(Linked List)**로 이어져 범위 검색이 매우 효율적

### 구조
```
                    [30 | 70]                   ← 루트 (키만 존재, 라우팅 용도)
                   /    |    \
          [10|20] [40|50|60] [80|90]            ← 내부 노드 (키만 존재)
        /             ↓            \
[10|20|30] ↔ [40|50|60|70] ↔ [80|90|...]       ← 리프 노드 (키 + 실제 데이터, 연결 리스트)
```
- 리프 노드의 `↔`는 양방향 연결 리스트(또는 단방향)를 나타냄
- 내부 노드의 키는 리프 노드로 가는 경로 안내자 역할만 함

### B-Tree vs. B+Tree 비교
|   항목    |   B-Tree  |   B+Tree  |
|----------|-----------|-----------|
|   데이터 저장 위치    |   모든 노드(내부 + 리프)  |   리프 노드에만   |
|   내부 노드의 역할    |   키 + 데이터 저장    |   키만 저장(라우팅)   |
|   리프 노드 연결  |   없음    |   연결 리스트로 연결  |
|   범위 검색   |   루트부터 매번 재탐색    |   리프 연결 리스트를 순차 스캔    |
|   포인트 검색 |   내부 노드에서 바로 반환 가능    |   항상 리프까지 내려가야 함   |
|   노드 당 키 수   |    상대적으로 적음    |   내부 노드에 키만 있어 더 많이 저장 가능 |
|   트리 높이   |   상대적으로 높음 |   내부 노드에 더 많은 키 → 더 납작한 구조 |
|   실제 DB 사용    |   MongoDB 일부 인덱스 |   MySQL(InnoDB), PostgreSQL, Oracle   |

### 범위 검색에서의 차이
```
SELECT * FROM orders WHERE amount BETWEEN 100 AND 500;

[B-Tree 방식]
루트 → 내부 노드 → 리프(100) 발견
루트 → 내부 노드 → 리프(200) 발견   ← 매번 루트부터 다시 탐색
루트 → 내부 노드 → 리프(300) 발견
... (N번의 루트부터 재탐색)

[B+Tree 방식]
루트 → 내부 노드 → 리프(100) 발견
→ → → 리프(200) [연결 리스트 순회]  ← 그냥 옆으로 이동
→ → → 리프(300)
→ → → 리프(400)
→ → → 리프(500) 까지 순차 이동 완료
```
> B+Tree는 범위 검색 시 리프 노드를 연결 리스트처럼 순차 탐색하므로 훨씬 효율적

## 실제 DB에서의 활용 — MySQL InnoDB
### Clustered Index(클러스터드 인덱스)
- InnoDB의 **기본키(PK)**는 B+Tree의 리프 노드에 실제 레코드 데이터 전체를 저장
- 레코드가 기본키 순서대로 물리적으로 정렬되어 저장됨
```
B+Tree 리프 노드 구조 (Clustered Index)

[PK: 1 | name: "김철수" | age: 30 | email: "kim@..."]
[PK: 2 | name: "이영희" | age: 25 | email: "lee@..."]
[PK: 3 | name: "박민수" | age: 28 | email: "park@..."]
↑ 리프 노드에 레코드 전체가 저장됨 (= 데이터 파일 자체가 인덱스)
```

### Secondary Index
- 기본키가 아닌 컬럼에 생성하는 인덱스
- 리프 노드에 해당 컬럼 값 + 기본키(PK) 값을 저장
- 실제 레코드를 가져오려면 PK를 통해 Clustered Index를 한 번 더 탐색함(이중 탐색)
```
email 컬럼의 Secondary Index 리프 노드

[email: "kim@..." | PK: 1]  →  Clustered Index로 PK=1 재탐색  →  실제 레코드
[email: "lee@..." | PK: 2]  →  Clustered Index로 PK=2 재탐색  →  실제 레코드
```
> 이중 탐색 비용을 피하려면 Covering Index를 활용

### Covering Index
```sql
-- users 테이블에 (email, name) 복합 인덱스가 존재할 때
SELECT name FROM users WHERE email = 'kim@example.com';
```
```
인덱스 리프 노드: [email | name | PK]
                         ↑
                 name이 인덱스에 이미 존재
                 → Clustered Index 재탐색 불필요
                 → Extra: "Using index" (실행 계획에서 확인 가능)
```

## 인덱스 선호도(Selectivity)와 카디널리티(Cardinality)
### 카디널리티
- 해당 컬럼이 가질 수 있는 유니크한 값의 수
- 카디널리티가 높을수록 인덱스 효과가 큼

|   컬럼 예시   |   카디널리티  |   인덱스 효과 |
|--------------|-------------|--------------|
|   gender(남/여)   |   낮음    |   거의 없음(Full Scan이 더 유리할 수 있음)    |
|   age |   중간    |   있음    |
|   email(유일) |   높음    |   매우 큼 |

### 복합 인덱스 컬럼 순서
```sql
-- 인덱스: (last_name, first_name, age)
-- 아래 쿼리는 인덱스를 효과적으로 사용
SELECT * FROM users WHERE last_name = '김' AND first_name = '철수';

-- 아래 쿼리는 first_name만으로 시작하면 인덱스 미사용 (선두 컬럼 없음)
SELECT * FROM users WHERE first_name = '철수';
```
- 복합 인덱스는 선두 컬럼부터 순서대로 사용되어야 효과적
- 카디널리티가 높은 컬럼을 앞에 배치하는 것이 일반적으로 유리

## 인덱스 사용 시 주의사항
### 인덱스가 무력화되는 경우
```sql
-- ❌ 컬럼에 함수 적용 → 인덱스 사용 불가
SELECT * FROM users WHERE UPPER(name) = 'KIM';

-- ❌ 타입 변환 → 묵시적 형 변환으로 인덱스 미사용
SELECT * FROM users WHERE phone = 01012345678;  -- phone은 VARCHAR

-- ❌ LIKE 검색에서 와일드카드가 앞에 오는 경우
SELECT * FROM users WHERE name LIKE '%철수';

-- ✅ 와일드카드가 뒤에 오면 인덱스 사용 가능
SELECT * FROM users WHERE name LIKE '김%';

-- ❌ OR 조건 (인덱스 머지가 발생하거나 Full Scan으로 퇴화)
SELECT * FROM users WHERE name = '김철수' OR age = 30;

-- ❌ NOT, != 조건
SELECT * FROM users WHERE age != 30;
```

### 인덱스 추가 시 부작용
```
INSERT / UPDATE / DELETE 시마다 B+Tree 재조정 발생

삽입 → 리프 노드 가득 참 → 노드 분할(Split) → 부모 노드 업데이트
삭제 → 키 제거 → 언더플로우 시 노드 병합(Merge)
수정 → 기존 키 삭제 + 새 키 삽입 (두 번의 트리 조정)
```
> 인덱스가 많을수록 쓰기 성능이 저하됨
>
> 자주 조회되고 카디널리티가 높은 컬럼에만 인덱스를 생성해야 함

## 트리의 높이와 I/O 횟수 계산
- 차수 M=1000, 총 레코드 N=1억 건일 때
트리 높이 = log_1000(1억) = log_1000(10^8) ≈ 2.67 → 약 3 depth

즉, 루트 → 내부 노드 → 리프 (3번의 디스크 읽기)만으로
1억 건 중 원하는 레코드를 찾을 수 있음
```
