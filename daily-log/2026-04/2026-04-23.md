## 메모리 영역 구조
- 프로세스가 실행될 때, OS는 가상 구조 공간을 여러 영역으로 나눠 할당함
```
고주소 ┌─────────────────────┐
       │   커널 영역          │  OS 전용 공간
       ├─────────────────────┤
       │   Stack             │  ← 높은 주소에서 낮은 주소 방향으로 성장 (↓)
       │   (지역 변수, 함수   │
       │    호출 정보)        │
       │         ↓           │
       │   (사용 가능 공간)   │
       │         ↑           │
       │   Heap              │  ← 낮은 주소에서 높은 주소 방향으로 성장 (↑)
       │   (동적 할당 메모리) │
       ├─────────────────────┤
       │   BSS               │  초기화되지 않은 전역/정적 변수
       ├─────────────────────┤
       │   Data              │  초기화된 전역/정적 변수
       ├─────────────────────┤
저주소  │   Text (Code)       │  실행 코드(읽기 전용)
       └─────────────────────┘
```

## Stack
- 컴파일 타임에 크기가 결정된 데이터를 저장하는 영역
- 함수가 호출되면 해당 함수의 스택 프레임(Stack Frame)이 쌓이고, 함수가 반환되면 자동으로 해제됨
- 할당/해제가 포인터 하나(SP, Stack Pointer)를 이동하는 것만으로 끝나기 때문에 매우 빠름
- 일반적으로 **8MB** 제한(OS마다 다름)

```cpp
void foo() {
    int x = 10;         // Stack: 함수가 끝나면 자동 해제
    int arr[100];       // Stack: 크기가 컴파일 타임에 확정됨
}                       // ← 여기서 x, arr 스택 프레임 제거
```

### 스택 프레임 구성 요소
|   항목    |   내용    |
|----------|-----------|
|   반환 주소   |   함수 종료 후 돌아갈 위치    |
|   이전 프레임 포인터  |   호출자 프레임 복원용    |
|   지역 변수   |   함수 내 선언도니 변수들 |
|   함수 인자   |   호출 시 전달될 값   |

> ⚠️**Stack Overflow**: 재귀가 너무 깊거나, 스택에 큰 배열을 할당하면 스택 공간이 초과되어 프로세스가 종료됨
>
> PS에서 `int arr[10000000]`을 함수 내부에서 선언하면 즉시 크래시가 발생하는 이유

## Heap
- 런타임에 크기가 결정되는 데이터를 저장하는 영역
- 개발자가 명시적으로 할당(`new`, `malloc`)하고, 명시적으로 해제(`delete`, `free`)하거나 소멸자/GC가 처리함
- Stack보다 할당 비용이 훨씬 큼
```cpp
void foo(){
    int* p = new int(10);       // Heap: 개발자가 직접 해제해야 함
    delete p;                   // 명시적 해제 (C++)

    std::vector<int> v(100);    // 내부적으로 Heap 할당
}                               // ← v 소멸자가 Heap 메모리 자동 해제 (RAII)
```

### Stack vs. Heap 비교
|   항목    |   Stack   |   Heap    |
|----------|------------|-----------|
|   할당 시점   |   컴파일 타임 결정    |   런타임 결정 |
|   할당 속도   |   $O(1)$, SP 이동만   |   수백 $ns$(free list 탐색 등)  |
|   해제 방식   |   함수 반환 시 자동   |   명시적 해제 또는 소멸자 |
|   크기 제한   |   ~8MB(스레드 당) |   수 GB(프로세스 가상 메모리) |
|   접근 패턴   |   연속적 → 캐시 친화적    |   분산 가능 → 캐시 미스 위험  |
|   단편화  |   없음    |   발생 가능   |

## 동적 할당(Dynamic Memory Allocation) 비용
### `new`/`malloc` 한 번의 내부 동작
```
new TreeNode() 호출 시 일어나는 일

1. free list 탐색
    Heap 내부의 빈 블록 목록을 순회하며 요청 크기에 맞는 블록 탐색

2. 블록 분할 및 메타데이터 기록
    블록 헤더에 크기, 상태(사용 중/빈 블록) 정보 저장

3. OS 시스템 콜(블록이 부족할 때만)
    brk / mmap으로 OS에게 추가 메모리 요청

4. 포인터 반환
    Heap 어딘가를 가리키는 주소 반환
```
- 1~2단계만 실행되도 수백 $ns$ 수준의 비용 발생
- 시스템 콜이 발생하면 수 $\mu s$ 이상으로 급증

### `malloc` 내부 구조(glibc 기준)
```
Heap 내부 관리 구조 (단순화)

[블록 헤더 | 사용 가능한 메모리 | 블록 풋터]
[블록 헤더 | 사용 중인 메모리            ]
[블록 헤더 | 사용 가능한 메모리 | 블록 풋터]
```
- malloc은 **빈 블록을 free list**(크기 별 버킷)로 관리함
- 요청 크기에 맞는 버킷을 찾아 블록을 꺼내거나 분할함
- 할당/해제가 반복되면 이 free list가 파편화됨

## 메모리 단편화(Memory Fragmentation)
### 외부 단편화(External Fragmentation)
- 할당·해제가 반복되면서 Heap 곳곳에 작은 빈 공간이 흩어지는 현상

```
초기 상태:
[ 빈 공간 (100 bytes 연속)                                     ]

할당 후:
[ A(20) | B(30) | C(10) | D(40)                               ]

B, D 해제 후:
[ A(20) | 빈(30) | C(10) | 빈(40)                              ]

→ 전체 빈 공간: 70 bytes
→ 그러나 50 bytes 연속 블록 요청이 들어오면 실패!
  (연속된 공간이 없으므로)
```
- 빈 공간의 총합은 충분하지만, 연속된 블록을 찾지 못해 할당이 실패하거나 느려짐
- free list 탐색 비용도 증가함

### 내부 단편화(Internal Fragmentation)
- 할당 요청 크기보다 더 큰 블록이 할당되어 내부에 낭비 공간이 생기는 현상
```
// 17 bytes 요청 → malloc은 정렬(alignment) 때문에 24 bytes 할당
// 7 bytes는 아무도 쓰지 않는 내부 낭비 공간
void* p = malloc(17);
```
- 메모리 정렬(alignment) 요구 때문에 불가피하게 발생함
- C++에서 구조체 패딩(padding)도 같은 이유로 발생

### 단편화가 성능에 미치는 영향
```
잦은 new/delete 반복 시 문제 연쇄

할당/해제 반복
    ↓
free list 파편화
    ↓
빈 블록 탐색 시간 증가 (free list 순회가 길어짐)
    ↓
할당된 주소가 Heap 곳곳에 분산
    ↓
캐시 미스 증가
```
> ⚠️ PS에서 new를 쿼리당 수만 번 호출하는 코드가 TLE를 내는 주된 이유가 여기에 있음.
>
> 알고리즘 복잡도는 맞아도, 할당 비용 자체가 병목이 됨

## 개념 정리
```
프로세스 메모리 구조
    ├── Stack
    │     ├── 자동 할당/해제
    │     ├── 빠름 (SP 이동만)
    │     └── 크기 제한 있음
    └── Heap
          ├── 동적 할당 (new / malloc)
          │     ├── free list 탐색
          │     ├── 블록 분할 및 메타데이터
          │     └── 수백 ns 비용
          └── 메모리 단편화
                ├── 외부 단편화 → 연속 블록 확보 불가
                └── 내부 단편화 → 정렬/패딩으로 인한 낭비
```
