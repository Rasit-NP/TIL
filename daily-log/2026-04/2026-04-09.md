# Design Pattern
## 디자인 패턴이란
- 소프트웨어 설계 시 자주 발생하는 문제들을 해결하기 위한 **재사용 가능한 설계 템플릿**
- 1994년 GoF(Gang of Four)가 저서 __Design Patterns: Elements of Reusable Object-Oriented Software__에서 23가지 패턴을 정리·체계화함
- 패턴 자체는 코드가 아니라 **아이디어**이므로, 언어나 프레임워크에 종속되지 않음
- 목적에 따라 크게 3가지 분류로 나뉨

|   분류    |   목적    |   대표 패턴   |
|----------|----------|--------------|
|   **생성(Creational)**    |   객체 생성 방식을 캡슐화 |   Singleton, Factory Method, Builder, Prototype, Abstract Factory |
|   **구조(Structural)**    |   클래스/객체를 조합해 더 큰 구조를 구성  |   Adapter, Decorator, Proxy, Facade, Composite, Bridge, Flyweight |
|   **행동(Behavioral)**    |   객체 간의 책임 분배와 알고리즘 캡슐화   |   Strategy, Observer, Template Method, Command, Iterator, Chain of Responsibility, State, Visitor, Mediator, Memento, Interpreter |

## 생성 패턴 (Creational Patterns)
### Singleton 패턴
> 클래스 인스턴스가 애플리케이션 전체에서 오직 하나임을 보장하는 패턴
- 인스턴스를 전역 접근 포인트로 만들어 어디서든 동일한 객체를 사용하게 함
- 공유 자원(DB 커넥션, 설정 객체 등)에 적합
```java
// Java의 Thread-safe Singleton 구현 (Double-Checked Locking)
public class AppConfig{
    private static volatile AppConfig instance;

    private AppConfig() {}

    public static AppConfig getInstance() {
        if (instance == null){
            synchronized (AppConfig.class){
                if (instance == null){
                    instance = new AppConfig();
                }
            }
        }
        return instance;
    }
}
```
### Spring에서의 Singleton
- Spring IoC 컨테이너에 등록된 빈은 기본적으로 Singleton 스코프
- 직접 구현할 필요 없이 `@Component`, `@Service`, `@Repository`, `@Bean` 등 어노테이션만으로 싱글톤 빈을 등록할 수 있음
```java
// Spring은 이 클래스의 인스턴스를 컨테이너에 딱 하나만 생성해 관리함
@Service
public class UserService {
    // ...
}
```
> **주의**: 싱글톤 빈은 상태(인스턴스 변수)를 가지면 안됨.
>
> 멀티스레드 환경에서 동시에 여러 요청이 같은 인스턴스를 공유하므로, 가변 필드가 있으면 동시성 문제가 발생함

### Factory Method 패턴
> 객체 생성의 인터페이스를 정의하되, 어떤 클래스의 인스턴스를 만들지는 서브 클래스가 결정하는 패턴
- 생성 로직을 클라이언트 코드에서 분리해 **결합도를 낮춤**
- 새로운 타입을 추가할 때 기존 코드를 수정하지 않아도 됨(OCP 원칙)
```java
// 알림 발송 예시
public interface Notification{
    void send(String message);
}

public class EmailNotification implements Notification{
    @Override
    public void send(String message){
        System.out.println("[EMAIL] " + message);
    }
}

public class SlackNotification implements Notification {
    @Override
    public void send(String message){
        System.out.println("[SLACK] " + message);
    }
}

// Factory
public class NotificationFactory{
    public static Notification create(String type){
        return switch (type){
            case "email" -> new EmailNotification();
            case "slack" -> new SlackNotification();
            default -> throw new IllegalArgumentException("Unknown type: " + type);
        };
    }
}
```

### Spring에서의 활용
- `BeanFactory`, `ApplicationContext` 자체가 Factory 패턴의 구현체
- `@Bean` 메서드는 특정 조건에 따라 다른 구현체를 반환하는 팩토리로 활용할 수 있음
```java
@Configuration
public class NotificationConfig{

    @Value("${notification.type}")
    private String notificationType;

    @Bean
    public Notification notification() {
        return switch (notificationType){
            case "email" -> new EmailNotification();
            case "slack" -> new SlackNotification();
            default -> throw new IllegalArgumentException("Unknown type");
        };
    }
}
```

### Builder 패턴
> 복잡한 객체의 생성 과정을 단계별로 분리하여, 동일한 생성 절차에서 다양한 객체를 만들 수 있는 패턴
- 생성자 매개변수가 많거나, 선택적 필드가 많은 객체에 적합
- `new Object(a, null, null, b, null, c)`같은 가독성이 낮은 코드를 방지

```java
public class HttpRequest{
    private final String url;
    private final String method;
    private final String body;
    private final int timeout;

    private HttpRequest(Builder builder){
        this.url        = builder.url;
        this.method     = builder.method;
        this.body       = builder.body;
        this.timeout    = builder.timeout;
    }

    public static class Builder{
        private final String url;       // 필수
        private String method = "GET";  // 기본값
        private String body;
        private int timeout = 3000;

        public Builder(String url){ this.url = url; }

        public Builder method(String method)    { this.method = method;     return this; }
        public Builder body(String body)        { this.body = body;         return this; }
        public Builder timeout(int timeout)     { this.timeout = timeout;   return this; }

        public HttpRequest build() { return new HttpRequest(this); }
    }
}

// 사용
HttpRequest request = new HttpRequest.Builder("https://api.example.com")
        .method("POST")
        .body("{\"key\": \"value\"}")
        .timeout(5000)
        .build();
```

### Spring에서의 활용
- Lombok의 `@Builder`를 통해 보일러플레이트 코드 없이 적용 가능
- Spring Security의 `HttpSecurity`, `WebClient.Builder` 등 Spring 자체 API도 Builder 패턴을 많이 사용함
```java

        .username("hong")
        .email("hong@example.com")
        .role("ADMIN")
        .build();
```

## 구조 패턴 (Structural Patterns)
### Proxy 패턴
> 다른 객체에 대한 접근을 제어하기 위해 대리자(Proxy) 객체를 두는 패턴
- 원본 객체에 접근하기 전후에 부가 로직(로깅, 트랜잭션, 캐싱, 보안 검사 등)을 삽입할 수 있음
- 클라이언트는 원본 객체와 프록시를 동일한 인터페이스로 사용하므로, 코드 변경 없이 기능을 추가할 수 있음

```java
public interface UserRepository{
    User findById(Long id);
}

public class UserRepositoryImpl implements UserRepository{
    @Override
    public User findById(Long id){
        // DB 조회 로직
        return new User(id, "hong");
    }
}

// 로깅을 추가하는 Proxy
public class LoggingUserRepositoryProxy implements UserRepository{
    private final UserRepository target;

    public LoggingUserRepositoryProxy(UserRepository target){
        this.target = target;
    }

    @Override
    public User findById(Long id){
        System.out.println("[LOG] findById 호출: id=" + id);
        User result = target.findById(id);
        System.out.println("[LOG] 결과: " + result);
        return result;
    }
}
```

### Spring에서의 활용
- Spring AOP의 핵심 구현 방식
- `@Transactional`, `@Cacheable`, `@Async` 등의 어노테이션은 모두 프록시 객체를 통해 동작함
- 두 가지 프록시 생성 방식이 있음
|   방식    |   조건    |   Spring Boot 기본    |
|----------|-----------|----------------------|
|   JDK Dynamic Proxy   |   인터페이스가 있을 때    |   -   |
|   CGLIB   |   클래스 상속 방식(인터페이스 불필요) |   ✅ (2.x 이후)   |
```java
// Spring이 아래 클래스에 대한 프록시를 자동 생성해 @Transactional을 처리함
@Service
public class OrderService{

    @Transactional  // 실제로는 프록시가 트랜잭션을 열고 닫음
    public void placeOrder(Order order){
        // ...
    }
}
```
> **Self-invocation 주의**: 같은 클래스 내에서 `this.method()`로 호출하면 프록시를 거치지 않아 `@Transactional`, `@Cacheable` 등이 무시됨
>
> 관련 메서드를 별도 빈으로 분리하는 것이 권장됨

### Decorator 패턴
> 객체에 동적으로 책임을 추가하는 패턴
>
> 서브클래스 대신 래퍼(Wrapper)로 기능을 확장함
- 상속 없이 기능을 조합할 수 있어, 클래스 폭발 문제를 방지함
- Java의 `BufferedInputStream(new FileInputStream(...))`이 대표적인 예시
```java
public interface TextProcessor {
    String process(String text);
}

public class PlainTextProcessor implements TextProcessor{
    @Override
    public String process(String text){ return text; }
}

// Decorator: 대문자 변환 추가
public class UpperCaseDecorator implements TextProcessor{
    private final TextProcessor wrapped;
    public UpperCaseDecorator(TextProcessor wrapped){ this.wrapped = wrapped; }

    @Override
    public String process(String text){
        return wrapped.process(text).toUpperCase();
    }
}

// Decorator: 트림 추가
public class TrimDecorator implements TextProcessor{
    private final TextProcessor wrapped;
    public TrimDecorator(TextProcessor wrapped){ this.wrapped = wrapped; }

    @Override
    public String process(String text){
        return wrapped.process(text.trim());
    }
}

// 조합 사용
TextProcessor processor = new UpperCaseDecorator(new TrimDecorator(new PlainTextProcessor()));
processor.process(" hello world ");
```

### Spring에서의 활용
- Spring Security의 필터 체인이 Decorator 패턴의 연속 적용과 유사한 구조
- `HttpServletRequestWrapper`, `HttpServletResponseWrapper`를 상속해 요청/응답 객체를 커스터마이징할 때 사용
```java
// 요청 본문을 여러 번 읽기 위한 Wrapper (Spring에서 자주 사용되는 패턴)
public class CachingRequestBodyWrapper extends HttpServletRequestWrapper{
    private final byte[] cachedBody;

    public CachingRequestBodyWrapper(HttpServletRequest request) throws IOException{
        super(request);
        this.cachedBody = StreamUtils.copyToByteArray(request.getInputStream());
    }

    @Override
    public ServletInputStream getInputStream(){
        return new ByteArrayServletInputStream(this.cachedBody);
    }
}
```

### Adapter 패턴
> 호환되지 않는 인터페이스를 가진 클래스들을 함께 동작할 수 있도록 변환해주는 패턴
- 기존 클래스를 수정하지 않고 새로운 인터페이스를 맞춤
- 외부 라이브러리나 레거시 코드를 통합할 때 주로 사용
```java
// 외부 결제 SDK (변경 불가)
public class TossPaymentSDK{
    public boolean requestPayment(String orderId, long amount){
        System.out.println("Toss 결제 요청: " + orderId + ", " + amount);
        return true;
    }
}

// 내부 표준 인터페이스
public interface PaymentGateway{
    boolean pay(String orderId, long amount);
}

// Adapter: 외부 SDK를 내부 인터페이스에 맞게 변환
public class TossPaymentAdapter implements PaymentGateway {
    private final TossPaymentSDK sdk;

    public TossPaymentAdapter(TossPaymentSDK sdk){ this.sdk = sdk; }

    @Override
    public boolean pay(String orderId, long amount){
        return sdk.requestPayment(orderId, amount);
    }
}
```

### Spring에서의 활용
- `HandlerAdapter`: 다양한 컨트롤러 타입(`@Controller`, `HttpRequestHandler` 등)을 DispatcherServlet이 동일하게 처리할 수 있도록 어댑터 패턴으로 통합
- Spring Security의 `UserDetailService`가 기존 유저 서비스를 Spring Security 인증 체계에 연결하는 역할

## 행동 패턴 (Behavioral patterns)
### Strategy 패턴
> 알고리즘을 인터페이스로 정의하고, 런타임에 원하는 구현체를 교체할 수 있게 하는 패턴
- `if-else`/`switch` 분기를 제거하고 확장에 열린 구조를 만듦
- OCP(개방-폐쇄 원칙)를 가장 직접적으로 구현하는 패턴
```java
// 할인 정책 Strategy 인터페이스
public interface DiscountStrategy {
    long apply(long price);
}

public class NoDiscount implements DiscountStrategy{
    @Override
    public long apply(long price){ return price; }
}

public class FixedDiscount implements DiscountStrategy{
    private final long amount;
    public FixedDiscount(long amount){ this.amount = amount; }

    @Override
    public long apply(long price){ return Math.max(0, price-amount); }
}

public class PercentDiscount implements DiscountStrategy {
    private final double rate;
    public PercentDiscount(double rate){ this.rate = rate; }

    @Override
    public long apply(long price) { return (long) (price * (1-rate)); }
}

// Context: Strategy를 사용하는 클래스
public class OrderService{
    private final DiscountStrategy discountStrategy;

    public OrderService(DiscountStrategy discountStrategy){
        this.discountStrategy = discountStrategy;
    }

    public long calculatePrice(long originalPrice){
        return discountStrategy.apply(originalPrice);
    }
}
```

### Spring에서의 활용
- Spring의 빈 주입과 결합하면 강력한 Strategy 패턴을 구성할 수 있음
- `Map<String, Strategy>` 주입으로 동적 전략 선택이 가능
```java
@Component("fixed")
public class FixedDiscountStrategy implements DiscountStrategy { ... }

@Component("percent")
public class PercentDiscountStrategy implements DiscountStrategy { ... }

@Service
@RequiredArgsConstructor
public class OrderService {

    // Spring이 DiscountStrategy 구현체들을 이름 → 빈으로 자동 주입
    private final Map<String, DiscountStrategy> strategies;

    public long calculatePrice(String discountType, long price) {
        DiscountStrategy strategy = strategies.getOrDefault(discountType, new NoDiscount());
        return strategy.apply(price);
    }
}
```

### Observer 패턴
> 객체의 상태 변화를 다른 객체들에게 자동으로 알리는 패턴. Pub/Sub 패턴이라고도 함
- 발행자(Subject)와 구독자(Observer)간 의존성을 최소화
- 이벤트 기반 아키텍처의 기반이 되는 패턴
```java
public interface EventListener{
    void onEvent(String event);
}

public class EventBus{
    private final List<EventListener> listeners = new ArrayList<>();

    public void subscribe(EventListener listener) { listeners.add(listener); }

    public void publish(String event){
        listeners.forEach(l -> l.onEvent(event));
    }
}
```

### Spring에서의 활용
- Spring의 `ApplicationEvent`/`ApplicationEventPublisher`로 느슨하게 결합된 이벤트 시스템을 구성할 수 있음

public class OrderPlacedEvent{
    private final Long orderId;
    public OrderPlacedEvent(Long orderId){ this.orderId = orderId; }

// 2. 이벤트 발행

public class OrderService{

@Component

}

### Template Method 패턴
> 알고리즘의 뼈대를 상위 클래스에 정의하고, 세부 구현은 서브 클래스에서 오버라이드하는 패턴
- 공통 흐름은 재사용하고, 변하는 부분만 오버라이드
- 상속 기반이라 Strategy 패턴보다 결합도가 높음. 가능하면 Strategy를 우선 고려
```java
// 데이터 파싱 파이프라인 뼈대 정의
public abstract class DataParser{

    // 템플릿 메서드: 전체 흐름 고정
    public final void parse(String data){
        String validated = validate(data);      // 유효성 검사
        String processed = process(validated);  // 처리
        save(processed);                        // 저장
    }

    protected abstract String validate(String data);
    protected abstract String process(String data);

    // 공통 로직 (오버라이드 불필요)
    private void save(String data){
        System.out.println("DB 저장: " + data);
    }
}

public class JsonDataParser extends DataParser {
    @Override
    protected String validate(String data){
        // JSON 유효성 검사 ...
        return data;
    }

    @Override
    protected String process(String data){
        // JSON 파싱 처리...
        return data;
    }
}
```

### Spring에서의 활용
- `JdbcTemplate`, `RestTemplate`, `RedisTemplate` 등 Spring의 `*Template` 클래스들이 대표적인 구현
    - DB 연결, 쿼리 실행, 자원 해제 같은 공통 흐름은 템플릿이 처리하고, SQL이나 콜백 로직만 사용자가 제공

```java
@Repository
@RequiredArgsConstructor
public class UserJdbcRepository{
    private final JdbcTemplate jdbcTemplate;

    public User findById(Long id){

        // 실제 SQL과 매핑 로직만 람다로 제공
        return jdbcTemplate.queryForObject(
            "SELECT * FROM users WHERE id = ?",
            (rs, rowNum) -> new User(rs.getLong("id"), rs.getString("username")),
            id
        );
    }
}
```

### Chain of Responsibility 패턴
> 요청을 처리할 수 있는 객체들을 체인으로 연결하고, 요청을 체인을 따라 전달하는 패턴
- 요청을 발신자와 수신자의 결합을 끊음
- 요청을 처리할 객체를 런타임에 결정할 수 있음
```java
public abstract class RequestHandler{
    protected RequestHandler next;

    public RequestHandler setNext(RequestHandler next){
        this.next = next;
        return next;
    }

    public abstract void handle(HttpRequest request);
}

public class AuthHandler extends RequestHandler{
    @Override
    public void handle(HttpRequest request){
        if (request.getHeader("Authorization") == null){
            System.out.println("인증 실패");
            return;
        }
        System.out.println("인증 통과");
        if (next != null) next.handle(request);
    }
}

public class RateLimitHandler extends RequestHandler{
    @Override
    public void handle(HttpRequest request){
        System.out.println("Rate Limit 체크 통과");
        if (next != null) next.handle(request);
    }
}
```

### Spring에서의 활용
- Spring Security의 필터 체인(`SecurityFilterChain`)이 Chain of Responsibility의 전형적인 구현
- Servlet 필터(`OncePerRequestFilter`)를 체인으로 구성해 인증/로깅/CORS 처리를 순서대로 적용
```java
@Component
@Order(1)
public class LoggingFilter extends OncePerRequestFilter{

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain filterChain) throws ServletException, IOException{
        System.out.println("[LOG] 요청 수신: " + request.getRequestURI());
        filterChain.doFilter(request,response);     // 체인의 다음 필터로 전달
        System.out.println("[LOG] 응답 완료");
    }
}
```

## SOLID 원칙과의 연관성
- 디자인 패턴은 대부분 SOLID 원칙을 구현하는 구체적인 방법

|   SOLID 원칙  |   관련 패턴   |
|---------------|-------------|
|   SRP(단일 책임)  |   Facade, Command |
|   OCP(개방-폐쇄)  |   Strategy, Decorator, Observer   |
|   LSP(리스코프 치환)  |   Template Method, Factory Method |
|   ISP(인터페이스 분리)   |    Adapter |
|   DIP(의존성 역전)    |   Strategy, Observer, Proxy   |

## FAQ
- **Proxy vs. Decorator**
    - Proxy는 **접근 제어**가 목적(캐싱, 보안, 트랜잭션)
    - Decorator는 **기능 확장**이 목적(동적으로 책임 추가)
    - 구조는 유사하지만 의도가 다름
- **Strategy vs. Template Method**
    - Strategy는 **컴포지션(조합)** 기반, 런타임에 교체 가능
    - Template Method는 **상속** 기반, 컴파일 타임에 결정
    - 현대적인 설계에서는 Strategy 방식을 선호
- **Spring에서 Singleton이 Thread-safe한가?**
    - Spring 빈 자체의 생성은 컨테이너가 보장하지만, 빈 내부의 가변 상태는 개발자가 직접 관리해야 함
    - 가변 인스턴스 변수를 갖는 싱글톤 빈은 동시성 문제가 발생할 수 있음
