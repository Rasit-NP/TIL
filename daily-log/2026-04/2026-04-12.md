## 1. 입출력 최적화

### `ios::sync_with_stdio(false)` + `cin.tie(NULL)`
```cpp
# include <iostream>
using namespace std;

int main(){
    ios::sync_with_stdio(false);
    cin.tie(NULL);

    int n;
    cin >> n;
    cout << n << "\n";
    return 0;
}
```
- `ios::sync_with_stdio(false)`
    - C 표준 입출력(`scanf`, `printf`)과 C++ 스트림(`cin`, `cout`)의 **동기화를 해제**하여 속도를 높임
- `cin.tie(NULL)`
    - `cin`과 `cout`의 **묶음(tie)을 해제**함
    - 기본적으로 `cin`은 입력 전 `cout`을 flush하는데, 이를 끊어 불필요한 flush를 방지함
- 위 두 줄을 사용할 경우 `scanf`/`printf`와 `cin`/`cout`을 혼용하면 안 됨

### `"\n"` vs. `endl`
```cpp
// 느림 - flush 발생
cout << "hello" << endl;

// 빠름 - flush 없음
cout << "hello" << "\n";
```
- `endl`은 개행 문자를 출력 후 **버퍼를 강제 flush**하기 때문에 느림
- `"\n"`을 사용하면 flush 없이 개행만 수행하여 빠름
- 특히 출력이 매우 많은 문제에서 차이가 두드러짐

### `scanf`/`printf`
```cpp
# include <cstdio>

int main(){
    int n;
    scanf("%d", &n);
    printf("%d\n", n);
    return 0;
}
```
- `sync_with_stdio(false)`를 쓰지 않는 상황이거나, C 스타일을 선호하는 경우 `scanf`/`printf`가 기본 `cin`/`cout`보다 빠름
- 단, `sync_with_stdio(false)` 이후에는 혼용 금지

### 직접 버퍼 사용 (극단적 최적화)
```cpp
# include <cstdio>
# include <cstring>

char buf[1<<22];    // 4MB 버퍼
int bufPos = 0;

void writeInt(int x){
    if (x < 0)  { buf[bufPos++] = '-'; x = -x; }
    if (x == 0) { buf[bufPos++] = '0'; return; }
    char tmp[12];   int len = 0;
    while (x)   { tmp[len++] = '0' + x%10;  x /= 10; }
    for (int i=len-1; i>=0; i--)    buf[bufPos++] = tmp[i];
}

int main(){
    // ...로직...
    fwrite(buf, 1, bufPos, stdout);
    return 0;
}
```
- 출력을 버퍼에 모아 `fwrite`로 한 번에 출력하는 방법
- 일반 PS에서는 거의 필요 없으나, 초고속 입출력이 필요한 대회 환경에서 사용됨

## 2. 자료형 선택과 비트 연산

### 자료형 선택
```cpp
// long long 대신 가능하면 int 사용(long long보다 빠름)
int a = 1000000;

// 꼭 필요할 때만 long long 사용
long long b = 1e18;

// 부동소수점 연산: double보다 float이 빠른 경우 있음 (단, 정밀도 주의)
float c = 3.14f;
```
- `long long`은 `int`보다 연산이 미세하게 느릴 수 있음
    - 범위가 충분하면 `int` 사용을 권장
- 나눗셈, 모듈러 연산은 정수 자료형에서 비용이 큰 편이므로 불필요한 나눗셈을 줄이는 것이 유리

### 비트 연산 활용
```cpp
// 2의 배수 곱셈/나눗셈 → 비트 시프트로 대체
int a = n * 2;  // 느림
int b = n << 1; // 빠름

int c = n / 4;  // 느림
int d = n >> 2; // 빠름

// 홀짝 판별
if (n % 2 == 1) { ... } // 느림
if (n & 1)      { ... } // 빠름

// 절댓값 없이 부호 확인
bool isNeg = (n >> 31) & 1;
```
- 2의 거듭제곱 관련 곱셈/나눗셈은 비트 시프트로 대체하면 더 빠름
- 컴파일러 최적화(O2 이상)에서는 자동으로 처리하는 경우도 많지만, 명시적으로 쓰면 의도를 명확히 전달할 수 있음

## 3. 메모리 및 배열 관리

### 전역 배열 vs. 지역 배열
```cpp
# include <iostream>
using namespace std;

// 전역 배열: BSS 세그먼트에 올라가며, 스택 오버플로 없음
int dp[1000001];
int arr[1000001];

int main(){
    // 지역 배열: 스택에 할당 → 크기가 크면 스택 오버플로 위험
    // int local[1000001];  // 위험

    return 0;
}
```
- 큰 배열(수십만~수백만)은 **전역 변수**로 선언하는 것이 안전하고, 스택 메모리를 아낄 수 있음
- 전역 배열은 0으로 자동 초기화됨(지역 배열은 쓰레기 값)

### `memset`/`fill`을 이용한 빠른 초기화
```cpp
# include <cstring>
# include <algorithm>

int dp[1001][1001];

int main(){
    // memset: 바이트 단위 초기화 (0 또는 -1에 유용)
    memset(dp, 0, sizeof(dp));
    memset(dp, -1, sizeof(dp)); // 모든 바이트를 0xFF로 → int 기준 -1

    // fill: 정확한 값으로 초기화 (0이 아닌 값에 안전)
    fill(&dp[0][0], &dp[0][0] + 1001*1001, 987654321);

    return 0;
}
```
- `memset(arr, 0, sizeof(arr))`으로 매우 빠른 0 초기화
- `memset`으로 `-1` 초기화도 가능(모든 비트가 1인 값)
- `0`이나 `-1`이 아닌 특정 값으로 초기화할 때는 `fill`이나 `fill_n` 사용

### 메모리 접근 패턴
```cpp
// 구조체 배열(AoS) vs 배열의 구조체(SoA)
// 특정 필드만 반복 접근할 땐 SoA가 캐시 효율 좋음

// pair/tuple 대신 구조체 (가독성 + 접근 명확성)
struct Edge { int u, v, w; };
```
- 특정 필드만 반복 접근하는 경우, AoS(Array of Structs)보다 SoA(Struct of Arrays)가 캐시 지역성 면에서 유리함

## 4. STL 컨테이너 및 반복문 최적화

### STL 컨테이너 선택과 최적화
```cpp
# include <vector>

int main(){
    // vector reserve로 재할당 방지
    vector<pair<int,int>> v;
    v.reserve(1000000);  // ✅ 미리 공간 확보

    // emplace_back vs. push_back
    v.emplace_back(1, 2);   // ✅ 직접 생성 (복사 없음)
    v.push_back({1, 2});    // 임시 객체 생성 후 복사

    // sort 후 unique로 중복 제거 (set보다 빠름)
    sort(v.begin(), v.end());
    v.erase(unique(v.begin(), v.end()), v.end());
}
```
- `vector`는 용량 초과 시 내부적으로 재할당(reallocation)이 발생하여 느려짐
- 최대 크기를 미리 알고 있다면 `reserve(n)`으로 미리 공간을 확보해 재할당 비용을 없앰
- `unordered_map`/`unordered_set`은 `map`/`set`($O(\log{N})$)보다 평균($O(1)$)로 해시 충돌이 적은 상황에서 빠름

### 반복문 최적화
```cpp
// 전위 증감 vs. 후위 증감
++i;    // ✅ (복사본 생성 없음)
i++;    // int에선 컴파일러가 최적화하지만, 이터레이터에선 차이남

// 반복문 내 불변값은 밖으로
int sz = v.size();
for (int i=0; i<sz; i++){...}       // ✅
for (int i=0; i<v.size(); i++){...} // ❌ 매번 호출
```
- 전위 증감(`++i`)은 복사본을 생성하지 않으므로 이터레이터 타입에서 특히 유리함
- 반복문 조건식에서 매번 계산되는 불변값은 루프 밖으로 꺼내는 것이 좋음

## 5. 컴파일러 설정

### `#pragma GCC optimize`
```cpp
# pragma GCC optimize("O3, unroll-loops")
# pragma GCC target("avx2, bmi, bmi2, popcnt")
# include <bits/stdc++.h>
using namespace std;

int main(){
    // ...
    return 0;
}
```
- `O3` : 컴파일러 최고 수준 최적화 적용
- `unroll-loops` : 루프 언롤링으로 반복문 오버헤드 감소
- `avx2` : SIMD 벡터 연산 활성화(지원하는 저지에서만 유효)
- BOJ 등 일부 온라인 저지에서 효과가 있으나, **모든 환경에서 동작을 보장하지 않음**
    - 테스트 후 사용 권장

### `bits/stdc++.h`
```cpp
# include <bits/stdc++.h>
using namespace std;
```
- GCC 환경에서 사용 가능한 헤더로, 거의 모든 표준 라이브러리를 한 번에 포함
- 컴파일 시간은 길어지지만 **PS에서는 코드 작성 시간**을 줄이는 데 유리
- MSVC(Visual Studio) 환경에서는 지원하지 않음
- `#pragma GCC optimize`와 함께 쓰이는 경우가 많음
