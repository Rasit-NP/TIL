# 통합 테스트 자동화
## 단위 테스트와 통합 테스트의 구분
- 단위 테스트는 함수나 클래스 하나를 격리해 그 내부 동작만 검증하는 테스트
- 통합 테스트는 여러 구성 요소를 실제로 결합해 그들 사이의 상호작용까지 검증하는 테스트
- 서버 프로그램의 통합 테스트는 서버 프로세스를 실제로 띄우고, 별도의 클라이언트가 네트워크를 통해 붙어 동작을 확인하는 형태가 됨
- 이 방식은 소켓 생성, 바인딩, 이벤트 분배, 응답 전송까지의 경로 전체를 한 번에 통과시킴
- 함수 단위로는 드러나지 않는 결함, 예를 들어 프로세스 종료 지연이나 유휴 상태의 비정상적인 자원 점유는 이 층위에서만 관측됨

## CTest 테스트 등록
- CTest는 CMake가 제공하는 테스트 실행기로, 등록된 테스트를 일괄 실행하고 성공과 실패를 집계하는 도구임
- 테스트는 `add_test` 명령으로 등록하며, 실행할 명령과 인자를 지정함
- 등록된 각 테스트는 지정한 명령을 실행해 종료 코드가 0이면 성공, 0이 아니면 실패로 판정됨
```cmake
# tests/integration/CMakeLists.txt
add_executable(echo_client echo_client.cpp)
target_compile_features(echo_client PRIVATE cxx_std_20)

add_executable(integration_driver driver.cpp)
target_compile_features(integration_driver PRIVATE cxx_std_20)

# NAME: 테스트 식별자, COMMAND: 실행할 명령과 인자
add_test(NAME echo_roundtrip COMMAND integration_driver $<TARGET_FILE:http-server>)
```
- `$<TARGET_FILE:http-server>`는 제너레이터 표현식으로, 빌드된 서버 실행 파일의 실제 경로로 치환됨
- 드라이버가 서버 경로를 인자로 받으면, 빌드 구성이나 디렉토리 구조가 달라져도 경로를 고정하지 않아도 됨

### 테스트 속성 설정
- `set_tests_properties`는 등록된 테스트에 부가 속성을 부여하는 명령임
- 통합 테스트에서 자주 쓰이는 속성은 다음과 같음

| 속성 | 의미 |
|-----|-----|
| `TIMEOUT` | 지정한 초를 넘기면 테스트를 강제 종료하고 실패 처리함 |
| `WILL_FAIL` | 종료 코드가 0이 아닐 때를 성공으로 뒤집어 판정함 |
| `FIXTURES_SETUP` | 이 테스트가 다른 테스트의 사전 준비 단계임을 표시함 |
| `FIXTURES_REQUIRED` | 이 테스트가 특정 준비 단계를 선행 조건으로 요구함을 표시함 |
| `RESOURCE_LOCK` | 같은 이름의 자원을 요구하는 테스트끼리 동시 실행을 막음 |

```cmake
set_tests_properties(echo_roundtrip PROPERTIES TIMEOUT 30)
```

### 픽스처
- 픽스처는 테스트 본체가 실행되기 위해 미리 갖춰져야 하는 환경을 가리키는 용어
- CTest 픽스처는 준비 테스트와 본 테스트를 의존 관계로 묶어, 준비가 끝난 뒤에만 본 테스트가 돌도록 만드는 장치임
- `FIXTURES_SETUP`로 표시한 테스트가 먼저 실행되고, 그 이름을 `FIXTURES_REQUIRED`로 요구하는 테스트가 뒤따름
- 서버 기동을 별도의 준비 테스트로 분리하고 검증 테스트가 그것을 요구하게 만드는 패턴이 가능함
- 다만 이 패턴은 서버 프로세스의 수명이 테스트 경계를 넘어 유지되어야 하므로 관리가 복잡해짐
- 대안으로 단일 드라이버 하나가 서버 기동, 검증, 종료를 모두 책임지는 패턴이 있음
- 단일 드라이버 방식은 프로세스 수명이 드라이버 안에 닫혀 있어 누수 가능성이 작고, 본 문서는 이 방식을 따름

## 자식 프로세스 생명주기 제어
- 드라이버가 서버를 직접 띄우고 종료시키려면 자식 프로세스를 다루는 시스템 콜을 사용함
- 관련 호출은 `fork`, `execv`, `kill`, `waitpid`이며 각각의 역할은 다음과 같음
- `fork`는 현재 프로세스를 복제해 자식 프로세스를 생성하는 호출임
    - 호출 한 번이 부모와 자식 양쪽에서 각각 반환됨
    - 부모에게는 자식의 PID를, 자식에게는 0을 반환해 두 흐름을 구분함
- `execv`는 현재 프로세스의 메모리 이미지를 지정한 실행 파일로 교체하는 호출임
    - 교체가 성공하면 원래 코드로 돌아오지 않음
    - 자식 측에서 호출해 복제된 드라이버 이미지를 서버 실행 파일로 덮어씌움
- `kill`은 지정한 PID의 프로세스에 시그널을 전달하는 호출임
    - 이름과 달리 종료 전용이 아니라 임의의 시그널을 보내는 일반 통로임
- `waitpid`는 지정한 자식 프로세스가 종료될 때까지 대기하고 그 종료 상태를 회수하는 호출임

### 좀비 프로세스
- 좀비 프로세스는 실행은 끝났으나 부모가 종료 상태를 회수하지 않아 프로세스 테이블에 남은 항목을 가리키는 용어임
- 자식이 종료되면 커널은 그 종료 상태를 부모가 읽어 갈 때까지 보관하며, 이 상태의 항목이 좀비임
- 부모가 `waitpid`로 상태를 회수해야 비로소 항목이 제거됨
- 회수하지 않으면 좀비가 누적되어 프로세스 테이블을 잠식하고, 반복 실행되는 테스트 환경에서 점진적으로 자원을 고갈시킴

```cpp
// driver.cpp - 서버 기동과 정리의 전체 흐름
pid_t spawn_server(const char* server_path, int port){
    pid_t pid = ::fork();                       // 프로세스 복제
    if (pid == 0){
        // 자식: 서버 이미지로 교체
        std::string port_str = std::to_string(port);
        char* argv[] = {
            const_cast<char*>(server_path),
            const_cast<char*>(port_str.c_str()),
            nullptr
        };
        ::execv(server_path, argv);             // 성공하면 아래로 돌아오지 않음
        ::_exit(127);                           // execv 실패 시에만 도달
    }
    return pid;                                 // 부모: 자식 PID 반환
}

void shutdown_server(pid_t pid){
    ::kill(pid, SIGTERM);                       // 정상 종료 요청
    int status = 0;
    ::waitpid(pid, &status, 0);                 // 종료까지 대기, 상태 회수
}

int main(int argc, char* argv[]){
    const char* server_path = argv[1];
    int port = 18080;

    pid_t pid = spawn_server(server_path, port);
    bool ok = run_echo_roundtrip(port);         // 검증 본체
    shutdown_server(pid);                       // 좀비 방지: 반드시 회수

    return ok ? 0 : 1;
}
```
- `execv` 실패 경로에서 `_exit`를 쓰는 이유는, 복제된 자식이 부모의 정리 로직이나 버퍼를 건드리지 않고 즉시 사라지게 하기 위함임
- `exit`는 등록된 종료 핸들러와 표준 출력 버퍼 플러시를 수행하므로, 부모와 공유된 상태를 중복 정리할 위험이 있음

## 동적 포트 할당
- 고정 포트를 강제하면 같은 포트를 쓰는 다른 프로세스나 동시 실행되는 테스트와 충돌함
- 충돌을 피하는 정석은 운영체제에게 빈 포트를 직접 고르게 하는 방식임
- `bind`에 포트 번호 0을 넘기면 운영체제가 현재 비어 있는 포트 하나를 임의로 할당함
- 이렇게 할당받은 포트 번호는 호출자가 미리 알 수 없으므로 별도로 조회해야 함

### getsockname을 통한 포트 발견
- `getsockname`은 소켓에 실제로 바인딩된 주소와 포트를 회수하는 호출임
- 포트 0으로 바인딩한 직후 이 호출로 운영체제가 정한 실제 포트 번호를 알아냄

```cpp
sockaddr_in addr{};
addr.sin_family      = AF_INET;
addr.sin_addr.s_addr = INADDR_ANY;
addr.sin_port        = htons(0);                // 0 → OS가 빈 포트 선택
::bind(listen_fd, reinterpret_cast<sockaddr*>(&addr), sizeof(addr));

sockaddr_in actual{};
socklen_t len = sizeof(actual);
::getsockname(listen_fd, reinterpret_cast<sockaddr*>(&actual), &len);
uint16_t real_port = ntohs(actual.sin_port);    // OS가 정한 실제 포트
```

### 포트 주입과 포트 발견의 설계 분기
- 통합 테스트에서 클라이언트가 서버에 붙으려면 포트 번호를 알아야 함
- 포트를 결정하고 전달하는 방식은 두 갈래로 나뉨

| 방식 | 포트 결정 주체 | 클라이언트의 포트 인지 방법 |
|-----|-------------|----------------------|
| 고정 포트 주입 | 드라이버가 인자로 지정 | 같은 값을 클라이언트에 그대로 전달 |
| 동적 포트 발견 | 운영체제가 `bind(0)`로 선택 | 서버가 실제 포트를 출력하고 드라이버가 읽음 |
- 고정 포트 주입은 구현이 단순하나, 하필 그 포트가 점유 중이면 충돌함
- 동적 포트 발견은 충돌이 원칙적으로 없으나, 서버가 실제 포트를 표준 출력으로 알리고 드라이버가 그 줄을 파싱하는 절차가 추가됨

# 테스트용 블로킹 클라이언트
## 블로킹 클라이언트 소켓
- 테스트 클라이언트는 단순함을 우선해 블로킹 소켓으로 구현함
- 블로킹 소켓은 데이터가 준비될 때까지 호출이 반환하지 않고 대기하는 소켓
- 클라이언트는 `socket`으로 소켓을 만들고, `connect`로 서버에 연결한 뒤, `send`로 보내고, `recv`로 받음

### 부분 수신 루프
- `recv`는 요청한 바이트 수를 한 번에 다 채워 주지 않고, 그 시점에 도착해 있는 만큼만 반환할 수 있음
- 따라서 N바이트를 기대한다면 받은 누계가 N에 도달할 때까지 반복 호출해야 함
- 이 반복을 빠뜨리면 일부만 받은 상태로 비교하게 되어, 대용량 응답에서 잘못된 검증 결과를 냄

```cpp
// 정확히 expected 바이트를 받을 때까지 반복
bool recv_exact(int fd, char* buf, size_t expected){
    size_t total = 0;
    while (total < expected){
        ssize_t r = ::recv(fd, buf + total, expected - total, 0);
        if (r > 0){
            total += static_cast<size_t>(r);    // 받은 만큼 누계에 더함
        } else if (r == 0){
            return false;                       // 상대가 연결을 닫음
        } else {
            if (errno == EINTR) continue;       // 시그널 중단은 재시도
            return false;                       // 실제 오류
        }
    }
    return true;
}
```

### 연결 준비 확인 재시도 루프
- 드라이버가 서버를 띄운 직후 곧바로 `connect`하면, 서버가 아직 `listen` 단계에 이르지 못해 `connect`가 `ECONNREFUSED`로 실패할 수 있음
- 고정된 시간 동안 잠들고 연결하는 방식은 환경에 따라 너무 짧거나 너무 길어 불안정함
- 짧은 간격으로 `connect`를 재시도하면서 성공할 때까지 기다리면, 서버 준비를 능동적으로 확인할 수 있음
```cpp
// 서버가 listen할 때까지 짧은 backoff로 재시도
int connect_with_retry(uint16_t port, int max_attempts){
    for (int attempt = 0; attempt < max_attempts; ++attempt){
        int fd = ::socket(AF_INET, SOCK_STREAM, 0);

        sockaddr_in addr{};
        addr.sin_family = AF_INET;
        addr.sin_port   = htons(port);
        ::inet_pton(AF_INET, "127.0.0.1", &addr.sin_addr);

        if (::connect(fd, reinterpret_cast<sockaddr*>(&addr), sizeof(addr)) == 0){
            return fd;          // 연결 성공
        }
        ::close(fd);            // 실패한 fd는 닫고
        ::usleep(10000);        // 10ms 대기 후 재시도
    }
    return -1;                  // 끝내 실패
}
```

## SIGPIPE 차단
- SIGPIPE는 이미 닫힌 연결 데이터를 쓰려 할 때 커널이 송신 측 프로세스에 보내는 시그널
- 이 시그널의 기본 동작은 프로세스 강제 종료이므로, 처리하지 않으면 클라이언트가 검증 도중 갑자기 죽을 수 있음
- 차단하는 방법은 두 가지
    - 프로세스 전역으로 `signal(SIGPIPE, SIG_IGN)`을 설정해 시그널 자체를 무시
    - `send` 호출마다 `MSG_NOSIGNAL` 플래그를 주어 그 호출에 한해 시그널 발생을 막음

```cpp
::signal(SIGPIPE, SIG_IGN);             // 프로세스 전역 무시
// 또는
::send(fd, buf, len, MSG_NOSIGNAL);     // 호출 단위 차단
```
- 전역 무시는 간단하나 프로세스 전체의 동작을 바꾸므로, 라이브러리 코드에서는 호출 단위 플래그가 더 안전함

# 자원 사용량 검증
## 유휴 CPU 측정
- 유휴 상태란 연결이 하나도 없어 서버가 처리할 일이 없는 상태를 가리킴
- 이벤트 대기 기반 서버는 유휴 상태에서 CPU를 거의 쓰지 않아야 함
- 만약 빈 루프를 계속 도는 바쁜 대기 구조라면 유휴 상태에서도 CPU를 100% 가까이 점유함
- 따라서 유휴 CPU 측정은 이벤트 대기 구조가 바쁜 대기로 회귀하지 않았음을 검증하는 수단이 됨

### proc stat을 통한 누적시간 조회
- `/proc/<pid>/stat`은 리눅스가 프로세스별 상태를 노출하는 가상 파일
- 이 파일의 14번째 필드 `utime`과 15번째 필드 `stime`이 각각 사용자 모드와 커널 모드에서 소비한 CPU 시간임
- 두 값의 단위는 클럭 틱이며, `sysconf(_SC_CLK_TCK)`로 초당 틱 수를 얻어 초로 환산함
- 이 파일은 리눅스 전용이므로 다른 운영체제에는 존재하지 않음
```cpp
// /proc/<pid>/stat에서 utime+stime(클럭 틱)을 읽어 합산
long read_cpu_ticks(pid_t pid){
    std::string path = "/proc/" + std::to_string(pid) + "/stat";
    std::ifstream f(path);
    std::string content;
    std::getline(f, content);

    // 명령 이름에 공백/괄호가 있을 수 있으므로 마지막 ')' 뒤부터 파싱
    size_t pos = content.rfind(')');
    std::istringstream iss(content.substr(pos+2));

    std::string field;
    long utime = 0, stime = 0;
    // ')' 이후 첫 필드(state)는 3번째 → utime은 그로부터 11번째, stime은 12번째
    for (int i=3; i<=15; ++i){
        iss >> field;
        if (i == 14) utime = std::stol(field);
        if (i == 15) stime = std::stol(field);
    }
    return utime + stime;
}
```

### 구간 델타 측정
- 한 시점의 값만 보면 측정 노이즈에 휘둘려 판정이 흔들림
- 일정 구간의 시작과 끝에서 누적 CPU 시간을 각각 읽고 그 차이를 보면 안정적인 판정이 가능함
- 예를 들어 연결이 없는 2초 구간 동안의 CPU 시간 증가분이 임계값 이하인지 확인함
- `top`이나 `ps`의 순간 점유율은 표본 추출 시점에 따라 출렁이므로 누적시간 델타가 더 신뢰할 만함
```cpp
// 유휴 2초 구간의 CPU 누적시간 증가분 측정
long before = read_cpu_ticks(pid);
::sleep(2);                             // 연결 없는 유휴 구간
long after  = read_cpu_ticks(pid);

long ticks_per_sec = ::sysconf(_SC_CLK_TCK);
double cpu_seconds = static_cast<double>(after - before) / ticks_per_sec;
// cpu_seconds가 작은 임계값(예: 0.1초) 이하이면 유휴 CPU 정상
```

## 종료 타이밍 측정
- 활성 연결을 유지한 상태에서 종료 시그널을 보낸 뒤, 실제 종료까지 걸린 시간을 재는 검증임
- 종료 절차가 차단 호출에 갇혀 지연되면 이 시간이 비정상적으로 길어짐
- 임계값(예: 1초) 이내 종료를 요구해, 종료 경로가 즉시 응답함을 고정함

### 단조 시계
- 시간 측정에는 `clock_gettime`에 `CLOCK_MONOTONIC`을 지정해 사용함
- `CLOCK_MONOTONIC`은 임의의 고정 시점부터 단조 증가하는 시계로, 도중에 뒤로 가지 않음
- 벽시계인 `CLOCK_REALTIME`은 시스템 시각 보정이나 NTP 동기화로 값이 갑자기 점프하거나 역행할 수 있음
- 두 시점의 차이로 경과 시간을 구하는 측정에서는 역행 가능성이 있는 벽시계가 음수 구간을 만들 수 있으므로, 단조 시계를 써야 정확함
```cpp
// SIGTERM 전송 후 실제 종료까지의 경과 시간 측정
timespec t0{}, t1{};
::clock_gettime(CLOCK_MONOTONIC, &t0);      // 측정 시작

::kill(pid, SIGTERM);                       // 종료 요청
int status = 0;
::waitpid(pid, &status, 0);                 // 실제 종료까지 대기

::clock_gettime(CLOCK_MONOTONIC, &t1);      // 측정 끝
double elapsed = (t1.tv_sec - t0.tv_sec)
                + (t1.tv_nsec - t0.tv_nsec) / 1e9;
// elapsed가 임계값(예: 1.0초) 이하이면 종료 타이밍 정상
```

## CI 환경과 로컬 환경의 측정 편차
- 같은 테스트라도 실행 환경에 따라 통과 여부가 갈릴 수 있음
- CI 러너는 코어 수가 적거나 다른 작업과 부하를 공유하므로, 로컬에서 통과한 시간 임계값이 CI에서는 초과될 수 있음
- 따라서 종료 타이밍과 CPU 임계값은 빠듯하게 잡지 않고 여유를 두어야 함
- `TIMEOUT` 속성 또한 러너의 처리 지연을 감안해 넉넉하게 설정함
> 타이밍에 의존하는 검증은 환경 편차를 항상 전제해야 함
>
> 임계값을 이론적 최소에 붙이면 환경이 조금만 느려져도 거짓 실패가 발생함

## 테스트 간 포트 점유 충돌
- CTest는 등록된 테스트를 병렬로 실행할 수 있음
- 여러 통합 테스트가 동시에 같은 고정 포트를 쓰려 하면 바인딩 단계에서 충돌함
- 충돌을 막는 방법은 두 가지
    - 각 테스트가 `bind(0)`로 서로 다른 포트를 받아 격리
    - `RESOURCE_LOCK` 속성으로 같은 포트 자원을 요구하는 테스트들의 동시 실행을 직렬화

```cmake
# 같은 포트 자원을 요구하는 테스트끼리 동시 실행 차단
set_tests_properties(echo_roundtrip     PROPERTIES RESOURCE_LOCK server_port)
set_tests_properties(large_payload      PROPERTIES RESOURCE_LOCK server_port)
```
