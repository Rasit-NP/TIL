## 핸들러가 직접 할 수 없는 일
- 단순 서버의 종료 구조는 시그널 핸들러가 종료 플래그를 세우고 메인 루프가 그 플래그를 검사해 빠져나오는 방식
- 이 구조는 `accept` 한 곳에서만 잠드는 서버에서는 정상 동작함
    - 시그널이 블로킹 중인 `accept`를 `EINTR`로 중단시킴
    - 중단으로 깨어난 루프가 플래그를 재검사해 종료함

```cpp
std::atomic<bool> g_should_stop{false};

extern "C" void handle_signal(int){
    g_should_stop.store(true, std::memory_order_relaxed);
}
```
- epoll 이벤트 루프로 넘어오면 이 전제가 무너짐
- `epoll_wait`에 타임아웃을 `-1`로 주면 감시 중인 fd에 이벤트가 없는 한 무한히 잠듦

```cpp
while(!g_should_stop.load(std::memory_order_relaxed)){
    int n = ::epoll_wait(epoll_fd_, events, MAX_EVENTS, -1);    // 무한 대기
    // ...
}
```
- 종료 시그널이 도착해도 `epoll_wait`가 그것을 인지하지 못하는 상황이 발생함

```
epoll_wait(-1) ──── 무한 대기 (감시 fd에 이벤트 없음) ─────────▶ 시간
        │
        │  SIGINT 도착
        ▼
   핸들러 실행 → g_should_stop = true
        │
        ▼
   epoll_wait는 여전히 잠들어 있음
        │
        │  (감시 fd 중 어느 것에도 이벤트가 없으므로)
        ▼
   다음 클라이언트 데이터가 도착할 때까지 플래그가 평가되지 않음
```
- 플래그는 세워졌지만 `epoll_wait`가 그것을 보지 못함
- 루프 조건 `!g_should_stop`은 `epoll_wait`가 반환된 뒤에야 평가되는데, `epoll_wait` 자체가 반환할 이유를 갖지 못함
- `epoll_wait`는 SA_RESTART 설정과 무관하게 시그널 핸들러에 중단되면 항상 `EINTR`로 반환됨
    - `man 7 signal`은 파일 디스크립터 다중화 인터페이스를 SA_RESTART 여부와 관계없이 절대 재시작되지 않는 호출로 명시함
    - 따라서 `epoll_wait`가 시그널로 중단되어 `EINTR`을 반환하는 경로 자체는 존재함
- 그러나 이 `EINTR` 경로에만 의존하면 깨우기 유실(lost wakeup)이 발생함
    - 루프가 `!g_should_stop`을 검사해 거짓을 확인한 직후, `epoll_wait`에 진입하기 직전의 창에서 시그널이 처리될 수 있음
    - 이 창에서 핸들러가 플래그를 세워도 `epoll_wait`는 그 사실을 모른 채 진입해 잠듦
    - 이미 처리가 끝난 시그널은 `epoll_wait`를 `EINTR`로 중단시키지 못하므로, 깨우기가 유실되어 다음 이벤트가 올 때까지 종료가 지연됨

```
[ lost wakeup이 발생하는 타이밍 창 ]

메인 루프                          시그널
   │
   while (!g_should_stop)  ← false 확인
   │
   │  ◀───────────────────── SIGINT 도착, 핸들러 실행
   │                          g_should_stop = true
   │                          (그러나 epoll_wait는 아직 진입 전)
   ▼
   epoll_wait(-1) 진입 → 이미 처리된 시그널은 여기서 깨우지 못함
   │
   ▼
   다음 이벤트가 올 때까지 잠듦 (종료 지연)
```

> 플래그 검사와 `epoll_wait` 진입 사이의 창에서 시그널이 처리되면 깨우기가 유실됨
>
> 시그널을 파이프 바이트로 영속화하면, 그 바이트가 남아 있는 한 다음 `epoll_wait`가 진입 즉시 readable 이벤트로 깨어나므로 타이밍 창과 무관하게 깨우기가 보존됨

## async-signal-safe 함수의 범위
- 핸들러는 메인 코드의 임의 시점에 끼어듦
- 메인 코드가 `malloc`의 내부 락을 쥔 채 힙 구조를 절반쯤 갱신하던 순간에 핸들러가 진입할 수 있음
- 이 상태에서 핸들러가 다시 `malloc`을 호출하면 같은 락을 재진입하려다 교착에 빠지거나 깨진 힙 구조를 더 훼손함
- 핸들러 안에서 호출해도 안전한 함수를 async-signal-safe 함수라 부름
- POSIX는 이 목록을 명시적으로 규정하며, `man 7 signal-safety`에서 전체를 확인할 수 있음

| 함수 | 핸들러 호출 가능 | 이유 |
|-----|--------------|-----|
| `write`, `read`, `close` | 안전 | 시스템 콜이며 재진입 안전성이 보장됨 |
| `_exit`, `sigaction` | 안전 | 사용자 공간 락이나 가변 전역 상태에 의존하지 않음 |
| `printf`, `fprintf` | 위험 | 내부 버퍼와 락이 재진입 시 깨짐 |
| `malloc`, `free` | 위험 | 힙 메타데이터 락을 재진입하면 교착 |
| `std::cout`, `std::string` 생성 | 위험 | 동적 할당과 스트림 락에 의존 |

- `write`가 안전한 근거는 그것이 커널의 시스템 콜이고 사용자 공간의 락이나 버퍼 상태에 의존하지 않는다는 점에 있음
- 한 번의 `write` 호출은 내부에서 원자적으로 처리되며, 메인 흐름이 보유한 사용자 공간 상태와 충돌하지 않음
- 이 성질이 다음 해법의 토대가 됨

> 핸들러 안에서 `printf`로 로그를 남기거나 `new`로 객체를 만드는 것은 모두 잠복성 버그의 원인이 됨
>
> 핸들러가 안전하게 수행할 수 있는 작업은 원자 변수 저장과 fd에 대한 `write` 정도임

## 해법의 방향
- 핸들러가 직접 `epoll_wait`를 깨울 수는 없음
- 그러나 `epoll_wait`가 감시하는 fd 중 하나에 이벤트를 일으킬 수는 있음
- 이 발상이 두 해법의 공통 골격임
```
시그널 도착 → 핸들러 → 특정 fd를 readable 상태로 전환
                              │
                              ▼
                  epoll_wait가 그 fd 이벤트로 깨어남
                              │
                              ▼
                  그 fd를 처리하며 종료 플래그 평가 → 루프 탈출
```
- 핵심은 비동기적으로 도착하는 시그널을 epoll이 다룰 수 있는 동기적 fd 이벤트로 변환하는 것임
- 이 변환의 구현 방식이 self-pipe trick과 signalfd로 나뉨

## self-pipe trick
- `pipe`로 읽기 끝과 쓰기 끝 한 쌍을 만든 뒤, 읽기 끝을 epoll에 등록하고 핸들러는 쓰기 끝에 1 바이트를 보냄
```
        ┌─────────────────────────────────────────────┐
        │                  파이프                       │
        │                                             │
  핸들러  │  write end ───────────────────▶ read end    │ ◀── epoll 감시
   (1B  │  (핸들러만 접근)                 (메인 루프가 비움) │     (EPOLLIN)
  write)│                                             │
        └─────────────────────────────────────────────┘

SIGINT → 핸들러: write(write_fd, "x", 1)
                        │
                        ▼
            read end가 readable 상태가 됨
                        │
                        ▼
            epoll_wait가 read end의 EPOLLIN으로 깨어남
                        │
                        ▼
            read end를 비우고 g_should_stop 평가 → 루프 탈출
```

### 파이프의 방향성
- 파이프는 단방향 통신 채널
- 쓰기 끝에 기록한 데이터는 읽기 끝에서만 읽힘
- 핸들러가 쓰기 끝만 접근하고 메인 루프가 읽기 끝만 접근하는 역할 분리는 의도적
    - 핸들러의 책임은 깨우기 신호를 보내는 것까지임
    - 그 신호를 소비하고 의미를 해석하는 책임은 메인 루프에 있음
- 읽기 끝을 epoll에 등록하는 근거는 epoll이 감시하는 대상이 읽을 데이터의 도착 여부라는 점
- 핸들러가 쓰기 끝에 바이트를 기록하면 읽기 끝이 readable 상태가 되고 그 순간 epoll이 깨어남

### 양쪽 끝의 논블로킹 설정
- 읽기 끝과 쓰기 끝 모두 `O_NONBLOCK`으로 설정해야 함
- 쓰기 끝의 논블로킹 설정 근거
    - 시그널이 짧은 시간에 폭주해 파이프 버퍼가 가득 찼을 때 핸들러의 `write`가 그 자리에서 잠드는 것을 막음
    - 핸들러가 블로킹되면 메인 흐름이 영원히 복귀하지 못함
    - 버퍼가 가득 차 `write`가 `EAGAIN`으로 실패해도 이미 들어 있는 1바이트만으로 깨우기 목적은 달성되므로 실패를 무시해도 무방함
- 읽기 끝의 논블로킹 설정 근거
    - 메인 루프가 파이프를 비울 때 더 읽을 것이 없는데도 잠드는 상황을 막음
    - `EAGAIN`이 날 때까지 읽어 비우고 빠져나오는 패턴을 사용함

```cpp
# include <fcntl.h>
# include <unistd.h>

int g_wake_fds[2];      // [0] = read end, [1] = write end

void setup_self_pipe(){
    if (::pipe(g_wake_fds) < 0){
        perror("pipe");
        return;
    }
    for (int i=0; i<2; i++){
        int flags = ::fcntl(g_wake_fds[i], F_GETFL);
        ::fcntl(g_wake_fds[i], F_SETFL, flags | O_NONBLOCK);
    }
}
```

### 핸들러에서의 errno 보존
- 핸들러는 메인 흐름의 임의 시점에 끼어들므로 메인 흐름이 방금 설정한 `errno`를 덮어쓰면 안 됨
- `write`는 실패 시 `errno`를 변경함
- 핸들러 진입 직전에 메인 흐름이 어떤 시스템 콜의 `errno`를 읽으려던 참이었다면, 핸들러의 `write`가 그 값을 오염시켜 메인 흐름이 다른 오류로 오인할 수 있음
- 이를 막기 위해 핸들러는 진입 시 `errno`를 지역 변수에 저장하고 복귀 직전에 복원함
```cpp
# include <atomic>
# include <cerrno>
# include <csignal>
# include <unistd.h>

std::atomic<bool> g_should_stop{false};
extern int g_wake_fds[2];

extern "C" void handle_signal(int){
    int saved_errno = errno;        // 1. 메인 흐름의 errno 보존

    g_should_stop.store(true, std::memory_order_relaxed);

    const char byte = 'x';
    ssize_t r = ::write(g_wake_fds[1], &byte, 1);   // 2. 깨우기 신호 (async-signal-safe)
    (void)r;                        // 버퍼가 가득 차 실패해도 무시

    errno = saved_errno;            // 3. errno 복원
}
```
> 핸들러가 `errno`를 오염시키는 버그는 발생 빈도가 낮고 재현이 어려움
>
> 시그널이 메인 흐름의 `errno` 검사 직전에 도착해야만 드러나므로 잠복하다 드물게 발현됨

## signalfd
- `signalfd`는 리눅스 전용 시스템 콜로, 시그널을 읽을 수 있는 파일 디스크립터를 생성함
- 이 fd를 epoll에 등록하면 시그널 도착 시 fd가 readable 상태가 되어 epoll이 깨어남
- self-pipe와 달리 별도의 시그널 핸들러 함수가 필요 없음
    - 시그널이 fd로 직접 유입되기 때문

### 시그널 블록이라는 전제
- `signalfd`를 사용하기 전에 해당 시그널을 반드시 블록해야 함
- 블록하지 않으면 기본 동작이나 등록된 핸들러가 시그널을 먼저 가로채 signalfd로는 시그널이 유입되지 않음
- 시그널 블록은 해당 시그널을 즉시 처리하지 않고 대기 상태로 보류하라는 의미이며, 보류된 시그널을 signalfd가 fd 읽기로 수거함
```
sigprocmask로 SIGINT/SIGTERM 블록
        │
        ▼
시그널 도착 → 기본 처리되지 않고 pending 상태로 보류
        │
        ▼
signalfd가 readable 상태가 됨
        │
        ▼
epoll_wait 깨어남 → signalfd를 read → 시그널 종류 식별
```
- 이 블록 단계의 누락이 signalfd에서 가장 흔히 발생하는 함정
- 블록을 빠뜨리면 시그널이 기본 동작인 프로세스 강제 종료로 처리되거나 핸들러로 새어나가 signalfd가 이벤트를 받지 못함

```cpp
# include <sys/signalfd.h>
# include <signal.h>
# include <unistd.h>

int setup_signalfd(){
    sigset_t mask;
    sigemptyset(&mask);
    sigaddset(&mask, SIGINT);
    sigaddset(&mask, SIGTERM);

    // 1. 해당 시그널 블록(누락 시 signalfd로 유입되지 않음)
    if (::sigprocmask(SIG_BLOCK, &mask, nullptr) < 0){
        perror("sigprocmask");
        return -1;
    }

    // 2. 블록한 시그널을 읽을 수 있는 fd 생성
    int sfd = ::signalfd(-1, &mask, SFD_NONBLOCK | SFD_CLOEXEC);
    if (sfd < 0){
        perror("signalfd");
        return -1;
    }
    return sfd;
}
```
- 이벤트 발생 시 `struct signalfd_siginfo`를 읽어 시그널 종류를 식별함
```cpp
# include <sys/signalfd.h>

void on_signalfd_readable(int sfd, std::atomic<bool>& stop){
    struct signalfd_siginfo si;
    while(::read(sfd, &si, sizeof(si)) == sizeof(si)){
        if (si.ssi_signo == SIGINT || si.ssi_signo == SIGTERM){
            stop.store(true, std::memory_order_relaxed);
        }
    }
    // 논블로킹이므로 EAGAIN으로 빠져나옴
}
```
- 멀티스레드 환경에서는 `sigprocmask` 대신 `pthread_sigmask`를 사용해 스레드별 시그널 마스크를 제어함
    - 시그널이 어느 스레드로 전달될지를 통제해야 하기 때문

## 두 방식의 비교
| 구분 | self-pipe trick | signalfd |
|-----|-----------------|----------|
| 이식성 | POSIX 전반에서 동작 | 리눅스 전용 |
| 핸들러 필요성 | 필요 (write 수행) | 불필요(fd로 직접 수신) |
| 코드 복잡도 | 파이프 생성·논블로킹·errno 보존으로 번거로움 | 블록 후 fd 생성으로 간결 |
| 시그널 식별 | 별도 전역 변수로 관리 | signalfd_siginfo로 fd에서 직접 식별 |
| 전제 조건 | 별도 사전 조건 없음 | 시그널을 먼저 블록해야 함 |

- self-pipe trick은 운영체제에 무관하게 동작하며 시그널을 fd로 변환하는 원리를 가장 명시적으로 드러냄
    - 파이프 생성, 논블로킹 설정, 핸들러의 `write`와 `errno` 보존, 읽기 끝 비우기까지 모든 단계가 코드에 노출됨
- signalfd는 핸들러 자체가 사라져 코드가 간결하지만 리눅스에 종속되며, 시그널 블록이라는 전제를 정확히 이해하지 못하면 동작하지 않음

## 깨우기 fd의 epoll 통합
- self-pipe의 읽기 끝을 epoll에 등록하면 이벤트 루프는 세 종류의 fd를 구분해야 함
- 기존 구조에서 리스닝 소켓은 `data.ptr`을 `nullptr`로 두어 표식으로 삼았고, 연결 소켓은 `data.ptr`에 `Connection*`을 실었음
- 여기에 깨우기 fd라는 제 3의 종류가 추가됨

```
data.ptr == nullptr        → 리스닝 소켓 (새 연결 도착)
data.ptr == WAKE_MARKER    → 깨우기 fd (종료 신호 도착)
data.ptr == Connection*    → 연결 소켓 (읽을 데이터 도착)
```

### 식별 방식의 두 선택지
- 깨우기 fd의 식별 방법에는 두 가지가 있음
- 첫째는 sentinel 포인터 방식임
    - 정적 저장 기간을 가진 더미 객체의 주소를 약속된 표식으로 사용함
    - `Connection*`과 결코 겹치지 않는 고유한 주소이므로 안전하게 구별됨

```cpp
// event_loop.cpp
namespace {
    char g_wake_sentinel;           // 주소만 사용하며 내용은 의미 없음
}
constexpr void* WAKE_MARKER = &g_wake_sentinel;
```
- 둘째는 `data.ptr` 대신 `data.fd`를 사용해 별도로 처리하는 방식
    - 현재 구조가 연결 소켓 식별에 `data.ptr`을 사용하므로, 깨우기 fd만 `data.fd`를 쓰면 같은 union의 서로 다른 멤버를 fd별로 혼용하게 되어 분기가 지저분해짐
- union의 일관성을 유지하려면 sentinel 포인터 방식이 정돈됨

```cpp
// 깨우기 fd 등록
epoll_event wev{};
wev.events   = EPOLLIN;
wev.data.ptr = WAKE_MARKER;             // sentinel: 깨우기 fd 표식
::epoll_ctl(epoll_fd_, EPOLL_CTL_ADD, g_wake_fds[0], &wev);
```
- 이벤트 처리 분기는 세 갈래로 확장됨
```cpp
void EventLoop::run(std::atomic<bool>& stop){
    constexpr int MAX_EVENTS = 64;
    epoll_event events[MAX_EVENTS];

    while(!stop.load(std::memory_order_relaxed)){
        int n = ::epoll_wait(epoll_fd_, events, MAX_EVENTS, -1);
        if (n<0){
            if (errno == EINTR) continue;
            break;
        }

        for (int i=0; i<n; ++i){
            void* tag = events[i].data.ptr;

            if (tag == nullptr){
                accept_new();               // 리스닝 소켓: 새 연결
                continue;
            }
            if (tag == WAKE_MARKER){
                drain_wake_fd();            // 깨우기 fd: 파이프 비우기
                continue;                   // 루프 조건이 stop을 재검사
            }
            auto* conn = static_cast<Connection*>(tag);
            if (!conn->on_readable()){      // 연결 소켓: 데이터 처리
                close_conn(conn->fd());
            }
        }
    }
}

void EventLoop::drain_wake_fd(){
    char buf[64];
    while (true){
        ssize_t n = ::read(g_wake_fds[0], buf, sizeof(buf));
        if (n > 0){
            continue;
        }
        if (n == 0){
            break;
        }
        if (errno == EAGAIN || errno == EWOULDBLOCK){
            break;
        }
        if (errno == EINTR){
            continue;
        }
        break;
    }
}
```
- 깨우기 fd를 비우는 근거는 레벨 트리거 모드에서 읽지 않은 데이터가 남으면 같은 이벤트가 반복 통지된다는 점에 있음
    - 파이프를 비우지 않으면 `epoll_wait`가 즉시 다시 깨어나 바쁜 반복에 빠짐
- 파이프를 비운 직후 곧장 종료를 단정하지 않고 `continue`로 루프 상단에 복귀해 `while` 조건이 `stop`을 재검사하도록 둠
    - 종료 판단은 루프 조건 한 곳에 집중됨
    - 깨우기 fd 처리부는 파이프를 비우는 역할만 담당함

## 검증 포인트
- 이 단계의 목표는 진행 중인 연결이 살아 있는 상태에서 종료 신호를 받았을 때 즉시 종료되는 것을 실측하는 것
```
1. 서버 기동, nc로 연결 하나를 맺어 active 상태로 유지
2. 서버에 kill -INT 전송
3. 종료까지 걸린 시간 측정

[ self-pipe 적용 전 ]
epoll_wait(-1)가 다음 데이터를 기다리며 잠듦
→ 다음 클라이언트 입력이 도착할 때까지 종료 지연

[ self-pipe 적용 후 ]
핸들러가 파이프에 write → epoll_wait 즉시 깨어남
→ 종료 지연이 0에 수렴
```
- 블로킹 구조에서 관측되던 종료 지연이 사실상 사라지는 것을 직접 확인하는 것이 검증의 핵심
- 이전 측정값과 비교해 지연이 0에 수렴함을 수치로 확인함
