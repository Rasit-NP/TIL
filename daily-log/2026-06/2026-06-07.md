## 동기 호출의 한계
- 서비스 A가 B를 직접 호출하는 구조에서는 A가 B의 응답을 기다리는 동안 묶여 있게 됨
- B가 느려지거나 죽으면 그 지연과 장애가 A까지 전파되는 강한 결합이 발생함
- 트래픽이 순간적으로 몰리면 B의 한계가 곧바로 A의 실패로 이어짐

```
[동기 호출]
주문 서비스 ──호출──> 결제 서비스 ──호출──> 알림 서비스
  (대기)             (대기)             (처리)
    ▲                                   │
    └──────── 전체 응답 시간 = 세 단계의 합 ──┘
```
> 핵심 문제는 호출자와 피호출자가 시간적으로 같이 살아 있어야 한다는 점
>
> 한 단계라도 느려지면 사슬 전체가 함께 느려짐

## 비동기 메시징의 발상
- 호출자는 처리 요청을 메시지로 만들어 중간 저장소에 넣고 곧바로 자기 일을 끝냄
- 처리자는 자신의 속도에 맞춰 메시지를 꺼내 작업을 수행함
- 생산자와 소비자가 시간적으로 분리되므로 한쪽의 부하가 다른 쪽으로 직접 번지지 않음

```
[비동기 메시징]
주문 서비스 ──넣기──> [ 메시지 브로커 ] ──꺼내기──> 결제/알림 서비스
(즉시 반환)             (버퍼 역할)            (각자 속도로 소비)
```

### 이 구조가 주는 이점
- 결합도 완화
    - 생산자는 소비자가 누구인지, 살아 있는지 몰라도 됨
- 부하 평준화
    - 순간 폭주를 브로커가 버퍼로 흡수해 소비자를 보호함
- 확장 용이성
    - 소비자 인스턴스를 늘리면 처리량을 수평으로 키울 수 있음

### 메시지 큐와 메시지 브로커
- 메시지 브로커
    - 생산자와 소비자 사이에서 메시지를 받아 보관하고 전달하는 중간 시스템
- 전달 모델은 크게 두 가지로 나뉨
    - 큐 모델
        - 하나의 메시지를 여러 소비자 중 한 명만 가져가는 작업 분배 방식
    - 발행-구독 모델
        - 하나의 메시지를 구독한 모든 소비자가 각자 받는 방송 방식
- Kafka는 토픽과 소비자 그룹 개념으로 위 두 모델을 동시에 표현할 수 있음

## Kafka의 핵심 구성 요소
- 토픽(Topic)
    - 메시지가 쌓이는 논리적 이름 공간이며 카테고리 역할을 함
- 파티션(Partition)
    - 하나의 토픽을 여러 조각으로 나눈 단위이며 병렬 처리와 순서 보장의 기본 축
- 오프셋(Offset)
    - 파티션 안에서 메시지가 가지는 순번이며 소비 위치를 가리키는 좌표
- 프로듀서(Producer)
    - 토픽으로 메시지를 보내는 주체임
- 컨슈머(Consumer)
    - 토픽에서 메시지를 읽어가는 주체임
- 컨슈머 그룹(Consumer Group)
    - 같은 목적을 가진 컨슈머들의 묶음이며 파티션을 그룹 내에서 나눠 가짐
- 브로커(Broker)
    - 파티션 데이터를 실제로 저장하고 요청을 처리하는 서버 노드

```
Topic: orders
┌────────────── Partition 0 ──────────────┐
│  off0  off1  off2  off3  off4  ...      │ ← 추가는 항상 끝에만(append)
└─────────────────────────────────────────┘
┌────────────── Partition 1 ──────────────┐
│  off0  off1  off2  ...                  │
└─────────────────────────────────────────┘
        ▲                          │
   Producer가 키 기준 분배      Consumer가 오프셋 추적하며 읽음
```

## 로그 기반 동작 방식
- Kafka는 메시지를 꺼내면 사라지는 큐가 아니라 끝에만 덧붙이는 로그(append-only log)로 저장함
- 소비는 메시지를 삭제하는 행위가 아니라 자신의 오프셋을 앞으로 옮기는 행위임
- 보관 기간(retention) 동안 데이터가 남아 있어 여러 그룹이 같은 데이터를 독립적으로 다시 읽을 수 있음
> 전통적 큐는 보통 브로커가 소비자에게 밀어주는(push) 방식이지만, Kafka는 소비자가 직접 당겨오는(pull) 방식임
>
> 소비 속도를 소비자가 제어하므로 느린 소비자가 브로커에 떠밀려 무너지는 상황을 피할 수 있음

### 순서 보장의 범위
- 순서는 토픽 전체가 아니라 파티션 단위에서만 보장됨
- 같은 키를 가진 메시지는 같은 파티션으로 보내져 키 단위 순서가 유지됨
- 따라서 순서가 중요한 데이터는 키 설계로 같은 파티션에 모이도록 해야 함

## 전통적 메시지 큐와의 비교
| 구분 | 전통적 큐(RabbitMQ 등) | Kafka |
|-----|---------------------|-------|
| 저장 모델 | 소비 후 삭제 | 로그로 일정 기간 보관 |
| 전달 방식 | 브로커가 push | 소비자가 pull |
| 재처리 | 기본적으로 어려움 | 오프셋 되감기로 가능 |
| 강점 | 복잡한 라우팅, 낮은 지연 | 높은 처리량, 대용량 스트림 |
