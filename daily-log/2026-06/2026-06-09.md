## 순회 폴링의 한계
- 소켓을 논블로킹으로 바꾸면서, 데이터가 없는 fd는 건너뛰고 다음 fd로 넘어가는 순회가 가능해짐
- 이 순회는 준비 여부와 무관하게 모든 fd를 매 바퀴 검사한다는 약점을 가짐
- 연결 만 개 중 실제로 데이터가 도착한 것이 하나뿐이어도, 나머지 9,999개를 전부 헛되이 확인함
- 대부분의 호출이 `EAGAIN`만 돌려받으므로 CPU가 의미 없는 시스템 콜에 소모됨
- 이는 "기다리지 않는 대신 쉬지도 않는" 바쁜 대기(busy waiting)이며, 블로킹과는 정반대 방향의 비효율임
> 논블로킹 순회는 한 연결에 갇히는 문제를 풀었지만, 준비되지 않은 fd까지 매번 훑는 새로운 낭비를 만듦
>
> 필요한 것은 "준비된 fd만 커널이 골라서 알려주는" 통지 방식

## select와 poll, 그리고 epoll의 차이
- 준비된 fd만 받아 오려는 시도는 `select`와 `poll`에서 먼저 동작함
- 두 인터페이스는 호출할 때마다 감시 대상 fd 목록 전체를 커널에 다시 전달해야 함
- 커널은 매 호출마다 그 목록 전체를 선형으로 훑어 준비 상태를 확인하므로, 비용이 fd 수에 비례함
- epoll은 감시 대상을 커널 내부 자료 구조에 미리 등록해 두고, 준비된 fd만 반환하는 방식으로 이 비용을 떼어 냄

| 항목 | `select`/`poll` | `epoll` |
|-----|-----------------|---------|
| 감시 목록 전달 | 호출마다 전체 재전달 | 최초 1회 등록 후 유지 |
| 준비 여부 검사 | 매번 전체 스캔, $O(n)$ | 준비된 fd만 수집, 평균 $O(1)$ |
| 반환 형태 | 전체 목록을 받아 직접 순회 | 준비된 이벤트만 골라서 반환 |
| 대규모 연결 | fd 수에 비례해 급격히 느려짐 | 연결 수와 거의 무관하게 일정 |

- 핵심 차이는 감시 대상의 상태를 호출자가 매번 들고 다니느냐, 커널이 보관하느냐에 있음
- epoll은 등록을 한 번 해두면 그 정보가 커널에 남으므로, 이후 호출은 결과만 가져오는 가벼운 연산이 됨

## epoll의 세 가지 시스템 콜
- epoll은 생성·등록·대기의 세 단계로 나뉜 별도의 시스템 콜을 가짐

### epoll 인스턴스 생성
- `epoll_create1`
    - 커널 안에 감시 정보를 담을 epoll 인스턴스를 만들고, 그것을 가리키는 파일 디스크립터를 반환
    - 인자로 `EPOLL_CLOEXEC`를 주면 `exec` 시 이 fd가 자동으로 닫혀 자식 프로세스로 새지 않음
    - 반환된 epfd 역시 fd이므로, 다 쓰면 `close`로 닫아야 함

### 감시 대상 등록과 변경
- `epoll_ctl`
    - 어떤 fd를 어떤 이벤트로 감시할지 epoll 인스턴스에 등록하거나 수정하거나 해제함
    - 동작은 세 가지 연산으로 구분됨

| 연산 | 의미 |
|-----|-----|
| `EPOLL_CTL_ADD` | 새 fd를 감시 목록에 추가함 |
| `EPOLL_CTL_MOD` | 이미 등록된 fd의 관심 이벤트를 바꿈 |
| `EPOLL_CTL_DEL` | fd를 감시 목록에서 제거함 |

### 준비된 이벤트 대기
- `epoll_wait`
    - 등록된 fd 중 준비된 것이 생길 때까지 대기하고, 준비된 이벤트 목록을 채워서 반환함
    - 반환값은 준비된 이벤트의 개수이며, 0이면 타임아웃, 음수면 오류를 의미함
    - 타임아웃을 `-1`로 주면 무한 대기, `0`으로 주면 즉시 반환함

> 블로킹이 사라지는 것이 아니라, 여러 fd에 흩어져 있던 블로킹이 `epoll_wait` 한 곳으로 모임
>
> 단일 스레드가 한 지점에서만 잠들면서도 수많은 연결을 동시에 지켜볼 수 있게 됨

## struct epoll_event 구조
- 등록할 때와 결과를 받을 때 모두 `epoll_event` 구조체를 매개로 사용함
```c
typedef union epoll_data {
    void        *ptr;       // 사용자 정의 객체를 가리키는 포인터
    int         fd;         // 감시 대상 파일 디스크립터
    uint32_t    u32;
    uint64_t    u64;
} epoll_data_t;

struct epoll_event {
    uint32_t        events; // 관심 이벤트 / 발생한 이벤트의 비트 마스크
    epoll_data_t    data;   // 이벤트와 함께 돌려받을 사용자 데이터
};
```

### events 필드
- 어떤 종류의 준비 상태에 관심이 있는지를 비트 플래그로 지정함
| 플래그 | 의미 |
|------|-----|
| `EPOLLIN` | 읽을 데이터가 준비됨 |
| `EPOLLOUT` | 블로킹 없이 쓸 수 있는 상태가 됨 |
| `EPOLLRDHUP` | 상대가 연결을 절반 닫음 |
| `EPOLLERR` | 오류 발생(등록하지 않아도 항상 통지됨) |
| `EPOLLHUP` | 연결이 끊김(등록하지 않아도 항상 통지됨) |
| `EPOLLET` | 엣지 트리거 모드로 전환(이번 단계에서는 사용하지 않음) |

### data 필드와 fd 대 ptr의 선택
- `epoll_wait`가 이벤트를 돌려줄 때, 그 이벤트가 어느 fd에서 왔는지는 등록 시 넣어 둔 `data` 값으로 식별함
- `data`는 공용체이므로 `fd`와 `ptr` 중 하나만 의미 있게 채워서 사용함
- `data.fd`
    - 디스크립터 번호만 보관하는 가장 단순한 방식
    - 단일 연결이나 상태가 거의 없는 경우에 적합함
- `data.ptr`
    - 연결별 상태(읽기 버퍼, 쓰기 대기열, 연결 단계 등)를 담은 객체를 가리킴
    - 이벤트가 도착하면 곧장 그 객체로 점프해 해당 연결의 문맥을 복원할 수 있음
    - 연결마다 상태를 들고 다녀야 하는 본격적인 서버에서 사실상 필수가 됨

> 오늘은 단일 연결만 다루므로 `data.fd`로 시작하고, 연결별 상태가 필요해지는 시점에 `data.ptr`로 옮겨 가는 것이 자연스러움

## 레벨 트리거와 엣지 트리거
- epoll의 통지에는 두 가지 모드가 있으며, 둘의 차이는 "언제 알려 주는가"에 있음
- 레벨 트리거(Level-Triggered)
    - 조건이 충족된 상태가 유지되는 한, 호출할 때마다 계속 통지함
    - 읽을 데이터가 남아 있으면 이번에 다 비우지 않아도 다음 `epoll_wait`가 다시 알려 줌
- 엣지 트리거(Edge-Triggered)
    - 상태가 바뀌는 순간에만 한 번 통지함
    - 통지를 받았을 때 버퍼를 끝까지 비우지 않으면, 남은 데이터에 대한 통지는 다시 오지 않음

| 구분 | 레벨 트리거 | 엣지 트리거 |
|-----|----------|----------|
| 통지 시점 | 조건이 참인 동안 계속 | 상태 전이가 일어난 순간 한 번 |
| 처리 부담 | 한 번에 다 안 비워도 안전 | `EAGAIN`이 날 때까지 반드시 다 비워야 함 |
| 통지 횟수 | 상대적으로 많음 | 최소화됨 |

> 레벨 트리거는 처리 누락에 너그러운 모드이며, 이 너그러움 덕분에 한 이벤트 당 한 번만 읽어도 동작이 무너지지 않음
>
> 엣지 트리거는 통지가 한 번뿐이므로 반드시 끝까지 비우는 루프를 갖춰야 함

## 직선 흐름에서 통지 기반 분기로
- 이전 단계까지의 서버는 `accept → read → write`가 위에서 아래로 한 줄로 이어지는 직선 구조
- 이 구조에서는 한 호출이 블로킹되면 흐름 전체가 그 자리에 멈춤
- epoll을 도입하면 흐름의 중심이 "다음에 무엇을 할까"에서 "어떤 fd가 준비됐는가"로 옮겨 감
- 코드는 더 이상 순서대로 내려가지 않고, `epoll_wait`가 돌려준 이벤트를 보고 그때그때 처리를 분기함

```
[이전: 직선 흐름]
accept() ─블로킹─▶ read() ─블로킹─▶ write() ─▶ 다음 연결
   한 호출에서 잠들면 그 동안 다른 fd는 보지 못함

[epoll: 통지 기반 분기]
        ┌────────────────────────────────┐
        │  epoll_wait() — 준비된 fd만 반환  │ ◀── 블로킹은 이 한 지점에만
        └────────────────┬───────────────┘
                         │ 준비된 이벤트 목록
             ┌───────────┴────────────┐
             ▼                        ▼
       fd == listen_fd           fd == client_fd
        accept()                  read() → write()
        새 연결 fd를 ADD            (echo 처리)
```
- 같은 fd라도 어떤 처리를 할지는 등록해 둔 `data`로 식별해 결정함
- listen 소켓의 준비 이벤트는 "새 연결이 도착했다"는 뜻이고, 연결 소켓의 준비 이벤트는 "읽을 데이터가 왔다"는 뜻으로 갈라짐

## 최소 구현
- 목표는 listen 소켓 하나를 epoll에 등록해 감시하고, 연결이 들어오면 그 연결 소켓까지 등록해 echo를 수행하는 것
- 연결은 하나만 다루지만, 흐름이 통지 기반으로 바뀌는 것을 확인하는 데 의의가 있음
- 레벨 트리거를 사용하므로 한 이벤트마다 `read`를 한 번만 호출해도 무방함
```cpp
# include <sys/epoll.h>
# include <sys/socket.h>
# include <unistd.h>
# include <cerrno>
# include <cstdio>

constexpr int MAX_EVENTS = 64;

int main(){
    Socket listen_sock{create_listen_socket(8080)};
    set_nonblocking(listen_sock.get());

    // 1. epoll 인스턴스 생성
    int epfd = ::epoll_create1(EPOLL_CLOEXEC);
    if (epfd < 0){
        perror("epoll_create1");
        return 1;
    }
    Socket epoll_sock{epfd};

    // 2. listen 소켓을 읽기 이벤트로 등록 (레벨 트리거)
    struct epoll_event ev{};
    ev.events   = EPOLLIN;
    ev.data.fd = listen_sock.get();
    if (::epoll_ctl(epfd, EPOLL_CTL_ADD, listen_sock.get(), &ev) < 0){
        perror("epoll_ctl: listen");
        return 1;
    }

    struct epoll_event events[MAX_EVENTS];

    // 3. 이벤트 루프
    while (!g_should_stop.load(std::memory_order_relaxed)){
        int n = ::epoll_wait(epfd, events, MAX_EVENTS, -1);     // 준비될 때까지 대기
        if (n < 0){
            if (errno == EINTR) continue;
            perror("epoll_wait");
            break;
        }

        for (int i=0; i<n; i++){
            int fd = events[i].data.fd;

            if (fd == listen_sock.get()){
                // listen 소켓의 이벤트 == 새 연결 도착
                int client_fd = ::accept(listen_sock.get(), nullptr, nullptr);
                if (client_fd < 0){
                    if (errno == EAGAIN || errno == EWOULDBLOCK)    continue;
                    perror("accept");
                    continue;
                }
                set_nonblocking(client_fd);

                struct epoll_event cev{};
                cev.events  = EPOLLIN;          // 이 연결의 이벤트도 감시
                cev.data.fd = client_fd;
                ::epoll_ctl(epfd, EPOLL_CTL_ADD, client_fd, &cev);
            }
            else {
                // 연결 소켓의 이벤트 = 읽을 데이터 도착
                char buf[1024];
                ssize_t r = ::read(fd, buf, sizeof(buf));
                if (r > 0){
                    ::write(fd, buf, r);
                }
                else if (r == 0){
                    ::epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);
                    ::close(fd);
                }
                else {
                    if (errno == EAGAIN || errno == EWOULDBLOCK)    continue;
                    if (errno == EINTR) continue;
                    ::epoll_ctl(epfd, EPOLL_CTL_DEL, fd, nullptr);
                    ::close(fd);
                }
            }
        }
    }
    return 0;
}
```

- 연결을 닫기 직전에 `EPOLL_CTL_DEL`로 감시를 해제하는 이유는, 닫힌 fd가 감시 목록에 남아 엉뚱한 통지를 일으키는 것을 막기 위함임
- 다만 fd를 `close`하면 커널이 epoll 등록도 함께 정리하므로, 단순한 경우에는 `DEL`을 생략해도 동작은 함
- 명시적으로 `DEL`을 호출하는 습관을 들이면, 같은 fd 번호가 재사용될 때 생기는 혼동을 예방할 수 있음

## 동작 확인과 트리거 체감
- 한 터미널에서 `nc localhost 8080`으로 접속해 문자열을 입력하면 동일한 문자열이 즉시 되돌아옴
- 이 때 흐름이 직선이 아니라 `epoll_wait`의 통지를 받고서야 `read`가 호출되는 구조임을 의식하면서 확인함
> 레벨 트리거에서는 한 통지에 다 비우지 못한 데이터가 사라지지 않고, 조건이 유지되는 한 다시 통지됨

## 주의할 점
- 레벨 트리거는 처리되지 않은 준비 상태를 계속 통지하므로, 처리할 수 없는 이벤트를 방치하면 같은 통지가 반복되어 바쁘게 돌 수 있음
    - 예를 들어 `EPOLLOUT`을 등록해 둔 채 쓸 데이터가 없으면, 쓰기 가능 상태가 매번 통지되어 불필요한 반복이 생김
    - 따라서 `EPOLLOUT`은 보낼 데이터가 쌓였을 때만 `EPOLL_CTL_MOD`로 켜고, 다 보내면 다시 끄는 운용이 일반적임
- 이번 단계에서는 echo가 즉시 `write`로 끝나므로 `EPOLLOUT` 없이 `EPOLLIN`만으로 충분함
