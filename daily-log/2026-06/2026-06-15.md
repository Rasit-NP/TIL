## 부분 전송의 이유
- `::write(fd, buf, len)`은 요청한 `len`만큼 항상 다 써 준다는 보장이 없음
- 커널은 소켓마다 송신 버퍼(send buffer)를 두고, 애플리케이션이 건넨 데이터를 일단 그곳에 복사한 뒤 네트워크로 내보냄
- `write`가 실제로 옮기는 양은 그 순간 송신 버퍼에 남은 빈 공간에 달려 있음
- 블로킹 소켓이라면 공간이 빌 때까지만 호출 안에서 잠들어 결국 전량을 밀어 넣지만, 논블로킹 소켓은 잠들지 않고 지금 가능한 만큼만 쓰고 즉시 반환함
- 따라서 논블로킹 `write`의 반환값은 네 갈래로 나뉨

| 반환 형태 | 의미 | 처리 |
|---------|-----|-----|
| `w == len` | 요청 전량이 송신 버퍼에 들어감 | 전송 완료 |
| `0 < w < len` | 일부만 들어감 (부분 전송) | 남은 `len - w`를 보관 후 재시도 |
| `w < 0 && errno == EAGAIN` | 송신 버퍼가 가득 참 | 한 바이트도 못 씀, 쓰기 가능 시점을 기다림 |
| `w < 0 && errno == EINTR` | 시그널에 의한 중단 | 같은 데이터로 재시도 |

> 부분 전송과 `EAGAIN`은 오류가 아니라 논블로킹 소켓의 정상 동작임
>
> 받은 만큼만 회신하던 `read`의 short read와 같은 결의 문제이며, 방향만 송신으로 뒤집힌 것

## 잔여분이 방치되는 송신 코드
- 송신 코드가 `write` 한 번의 결과로 쓴 만큼만 버퍼에서 제거하고 함수를 빠져나가는 형태라면, 부분 전송된 잔여분이 처리되지 못함
```cpp
// 잔여분을 다시 보낼 트리거가 없는 송신부
if (!write_buf_.empty()){
    ssize_t w = ::write(sock_.get(), write_buf_.data(), write_buf_.size());
    if (w > 0){
        write_buf_.erase(0, static_cast<size_t>(w));    // 쓴 만큼만 제거
    }
    // 다 비우지 못했을 때 남은 데이터를 보낼 트리거가 없음
}
```
- `write_buf_`에 잔여분이 남아도 그 fd가 `EPOLLIN`만 감시한다면, 읽기 이벤트가 다시 오지 않는 한 송신 핸들러가 호출되지 않음
- 그 결과 남은 데이터는 영영 전송되지 못하고 버퍼에 갇힘
- echo 페이로드가 작을 때는 한 번의 `write`로 전부 들어가 이 문제가 드러나지 않음

## 송신 버퍼와 EPOLLOUT 토글
- 핵심 아이디어는 보낼 데이터를 곧장 소켓에 밀어 넣지 않고, 일단 송신 버퍼에 적재한 뒤 가능한 만큼만 흘려보내는 것
- 다 비우지 못하면 그 fd의 관심 이벤트에 `EPOLLOUT`을 추가해, 쓰기 가능 상태가 되었을 때 커널이 다시 깨워 주도록 함
```
[1] 보낼 데이터 → write_buf_ 에 append
[2] write_buf_ 를 가능한 만큼 ::write
        │
        ├─ 전량 전송 → write_buf_ 비움 → EPOLLOUT 끔(켜져 있었다면)
        │
        └─ 부분 전송 / EAGAIN → 잔여분 보관 → EPOLLOUT 켬(EPOLL_CTL_MOD)
                                        │
                            [3] 소켓이 writable 해짐
                                        │
                            [4] epoll_wait 가 EPOLLOUT 통지 → 잔여분 마저 전송
                                        │
                            [5] 다 비우면 EPOLLOUT 끔 → 다시 EPOLLIN 만 감시
```
- 읽기 이벤트는 항상 감시하되, 쓰기 이벤트는 보낼 잔여분이 있는 동안에만 한시적으로 감시함

## EPOLLOUT 상시 등록과 바쁜 대기
- 소켓은 송신 버퍼에 여유 공간만 있으면 거의 항상 쓰기 가능 상태로 판정됨
- 따라서 `EPOLLOUT`을 상시 등록해 두면, 보낼 데이터가 전혀 없어도 레벨 트리거가 쓰기 가능 상태를 매 루프마다 통지함
- `epoll_wait`가 끊임없이 깨어나지만 쓸 것이 없어 아무 일도 하지 않고 다시 도는, 바쁜 대기(busy waiting)에 빠짐
- 이는 논블로킹 순회에서 준비되지 않은 fd를 매번 훑던 낭비와 동일한 구조이며, 그 낭비가 쓰기 방향에서 재현되는 것임
- 따라서 운용 원칙은 보낼 잔여분이 생겼을 때만 켜고, 다 비우는 즉시 끄는 것으로 고정됨

| 운용 방식 | 동작 | 결과 |
|---------|-----|-----|
| `EPOLLOUT` 상시 등록 | 쓸 게 없어도 매번 통지 | 바쁜 대기, CPU 낭비 |
| 잔여분 있을 때만 등록 | 보낼 게 있을 때만 통지 | 필요한 순간에만 깨어남 |

> 레벨 트리거에서 `EPOLLOUT`은 다루기 까다로운 이벤트
>
> 읽기 이벤트는 데이터가 오지 않으면 통지되지 않지만, 쓰기 이벤트는 가만히 두면 거의 항상 통지되기 때문

## 토글의 책임 경계
- `EPOLL_CTL_MOD` 호출은 epoll 파일 디스크립터를 단독으로 소유한 `EventLoop`의 권한
- 반면 송신 버퍼가 비었는지, 잔여분이 남았는지는 그 버퍼를 들고 있는 `Connection`만 알 수 있음
- 두 객체의 책임이 갈라져 있으므로, 둘 사이를 잇는 인터페이스가 필요함
- 한 가지 방법은 `Connection`이 쓰기 관심 여부를 외부에 노출하고, `EventLoop`가 그 신호를 받아 `EPOLL_CTL_MOD`를 수행하는 것

```cpp
// connection.h -- 쓰기 관심 상태를 외부에 노출
class Connection {
private:
    Socket      sock_;              // 연결 소켓 (RAII, 단독 소유)
    std::string read_buf_;          // 수신 버퍼
    std::string write_buf_;         // 송신 버퍼 (아직 다 못 보낸 잔여분)
    bool        writing_ = false;   // 현재 EPOLLOUT을 켜 둔 상태인가

public:
    explicit Connection(Socket sock) : sock_(std::move(sock)) {}
    int fd() const noexcept { return sock_.get(); }

    bool on_readable();             // 반환: true 유지 / false 종료
    bool on_writable();             // 반환: true 유지 / false 종료

    // EventLoop가 EPOLL_CTL_MOD 여부를 판단하기 위한 질의
    bool want_write() const noexcept { return !write_buf_.empty(); }
    bool is_writing() const noexcept { return writing_; }
    void set_writing(bool on) noexcept { writing_ = on; }

private:
    bool flush_write();             // 송신 버퍼를 가능한 만큼 전송하는 공통 경로
};
```
- 부분 전송과 `EAGAIN`을 다루는 송신 로직은 한 군데로 모아 두는 것이 일관됨

```cpp
// connection.cpp -- 송신 버퍼를 가능한 만큼 흘려보내는 공통 경로
// 반환: true면 연결 유지, false면 연결 종료 요청
bool Connection::flush_write() {
    while (!write_buf_.empty()){
        ssize_t w = ::write(sock_.get(), write_buf_.data(), write_buf_.size());
        if (w > 0){
            write_buf_.erase(0, static_cast<size_t>(w));    // 쓴 만큼 제거
            continue;                                       // 더 보낼 수 있는지 재시도
        }
        if (w < 0 && (errno == EAGAIN || errno == EWOULDBLOCK)){
            break;                                          // 송신 버퍼 가득 참: 잔여분 보관
        }
        if (w < 0 && errno == EINTR){
            continue;                                       // 시그널 중단: 같은 데이터로 재시도
        }
        return false;                                       // 실제 오류: 연결 종료
    }
    return true;
}
```
- 읽기 핸들러는 받은 데이터를 송신 버퍼에 적재한 뒤 즉시 한 번 흘려보냄

```cpp
bool Connection::on_readable(){
    char buf[4096];
    while (true){
        ssize_t n = ::read(sock_.get(), buf, sizeof(buf));
        if (n > 0){
            write_buf_.append(buf, static_cast<size_t>(n)); // echo: 받은 만큼 적재
            continue;
        }
        if (n == 0)                                     return false;   // FIN 수신
        if (errno == EAGAIN || errno == EWOULDBLOCK)    break;          // 더 읽을 것 없음
        if (errno == EINTR)                             continue;       // 시그널 중단
        return false;                                                   // 실제 오류
    }
    return flush_write();       // 적재분을 곧장 가능한 만큼 전송
}

// 쓰기 가능 통지를 받았을 때: 남은 잔여분을 마저 전송
bool Connection::on_writable(){
    return flush_write();
}
```
- `EventLoop`는 핸들러 호출 직후 쓰기 관심 상태가 바뀌었는지 비교해, 필요할 때만 `EPOLL_CTL_MOD`를 호출함
```cpp
// event_loop.cpp -- 디스패치 후 EPOLLOUT 토글 판단
void EventLoop::update_interest(Connection* conn){
    bool want = conn->want_write();     // 보낼 잔여분이 있는가
    bool now  = conn->is_writing();     // 지금 EPOLLOUT을 켜 둔 상태인가
    if (want == now){
        return;                             // 상태 변화 없음: MOD 불필요
    }

    epoll_event ev{};
    ev.events   = EPOLLIN | (want ? EPOLLOUT : 0);      // 잔여분 있으면 쓰기도 검사
    ev.data.ptr = conn;
    ::epoll_ctl(epoll_fd_, EPOLL_CTL_MOD, conn->fd(), &ev);
    conn->set_writing(want);            // 현재 등록 상태 갱신
}

void EventLoop::run(std::atomic<bool>& stop){
    constexpr int MAX_EVENTS = 64;
    epoll_event events[MAX_EVENTS];

    while (!stop.load(std::memory_order_relaxed)){
        int n = ::epoll_wait(epoll_fd_, events, MAX_EVENTS, -1);
        if (n < 0){
            if (errno == EINTR) continue;
            break;
        }

        for (int i=0; i<n; ++i){
            if (events[i].data.ptr == nullptr){
                accept_new();
                continue;
            }

            auto* conn = static_cast<Connection*>(events[i].data.ptr);

            bool alive = true;
            if (events[i].events & EPOLLIN)
                alive = conn->on_readable();
            if (alive && (events[i].events & EPOLLOUT))
                alive = conn->on_writable();

            if (!alive){
                close_conn(conn->fd());     // 종료 요청 시 정리
                continue;
            }
            update_interest(conn);          // 쓰기 관심 변화 반영
        }
    }
}
```
- `EventLoop`는 누구에게 일이 생겼는가만 알고, `EPOLLOUT`을 켜고 끄는 판단은 `Connection`이 노출한 상태에 위임함
- I/O 다중화 계층과 연결 상태 관리 계층의 책임 경계를 유지하면서, 쓰기 관심 토글이라는 협의 항목 하나가 추가되는 구조임
> 한 이벤트 안에서 `EPOLLIN`과 `EPOLLOUT`이 함께 통지될 수 있으므로, 둘을 비트 마스크로 각각 검사해 순서대로 처리함
>
> 읽기를 처리하다 연결이 끊겼다면 쓰기를 시도하지 않도록 `alive` 플래그로 가드를 둠

## 검증
- 작은 echo는 송신 버퍼에 한 번에 다 들어가므로 부분 전송 경로가 실행되지 않음
- 부분 전송을 강제로 재현하려면 두 가지 방향이 있음
    - 송신 버퍼 용량을 넘어서는 큰 페이로드(수백 KB에서 수 MB)를 한 번에 보냄
    - 클라이언트가 의도적으로 천천히 읽어 서버의 송신 버퍼가 채워지도록 만듦
- 송신 버퍼가 가득 차면 `write`가 `EAGAIN`을 돌려주거나 일부만 쓰게 되고, 그제야 `EPOLLOUT` 등록 경로가 발동함
- 최소 목표는 송신 버퍼 용량을 초과하는 페이로드를 보내도 한 바이트의 유실 없이 그대로 echo되어 돌아오는 것
- 송신 버퍼의 현재 용량은 `getsockopt`의 `SO_SNDBUF`로 확인할 수 있으며, 그보다 큰 데이터를 던지는 것이 가장 단순한 재현 방법임
