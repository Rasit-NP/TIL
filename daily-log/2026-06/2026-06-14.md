## 책임 분리
- epoll을 단일 파일 디스크립터에만 적용하던 코드는 `main` 함수 하나에 모든 것이 뭉쳐 있음
    - epoll 인스턴스 생성과 대기, 새 연결 수락, 연결별 읽기·쓰기, 정리가 한 함수에 섞여 있음
- 연결이 하나일 때는 문제가 드러나지 않지만, 여러 연결을 동시에 다루는 순간 두 종류의 관심사가 충돌함
    - 하나는 "어떤 fd에 이벤트가 왔는가"를 판별하고 분배하는 일
    - 다른 하나는 "이 연결 하나가 지금 어떤 상태이고 무엇을 읽고 써야 하는가"를 관리하는 일
- 이 둘을 분리하지 않으면 연결 수가 늘어날수록 분기와 지역 변수가 폭발적으로 늘어남
> 책임 분리는 단순한 코드 정리가 아님
>
> 프로토콜 파서를 끼워 넣을 자리를 미리 비워 두기 위한 구조적 준비

## 두 객체의 책임 경계
- `EventLoop`는 "전체 시스템의 교통 관제" 역할을 맡음
    - epoll 파일 디스크립터를 단독으로 소유함
    - `epoll_wait`로 깨어난 이벤트들을 해당 연결로 분배(dispatch)함
    - 리스닝 소켓에서 새 연결을 수락하고 등록함
    - 끊어진 연결을 `EPOLL_CTL_DEL`로 해제하고 소멸시킴
- `Connection`은 "연결 하나의 살림" 역할을 맡음
    - 그 연결의 `Socket`(fd)을 소유함
    - 입력·출력 버퍼를 보유함
    - 연결의 현재 상태(읽는 중, 닫히는 중 등)를 들고 있음
- `EventLoop`는 "누구에게 일이 생겼는가"를 알고, `Connection`은 "그 일을 어떻게 처리하는가"를 앎

```
        ┌─────────────────────────────────────────┐
        │              EventLoop                  │
        │  - epoll fd 소유                         │
        │  - epoll_wait → 이벤트 디스패치             │
        │  - accept → Connection 생성·등록          │
        │  - EPOLL_CTL_DEL → Connection 소멸       │
        └───────────────┬─────────────────────────┘
                        │ 소유 (unordered_map)
        ┌───────────────┼─────────────────┬─────────────────┐
        ▼               ▼                 ▼                 ▼
   ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
   │Connection│     │Connection│     │Connection│     │Connection│
   │ Socket   │     │ Socket   │     │ Socket   │     │ Socket   │
   │ 버퍼      │     │ 버퍼      │     │ 버퍼      │     │ 버퍼      │
   │ 상태      │     │ 상태      │     │ 상태      │     │ 상태      │
   └──────────┘     └──────────┘     └──────────┘     └──────────┘
```

## 소유권 모델을 확정
- "`Connection` 객체를 누가 언제까지 소유하는가"를 못 박아야 함
- 이 결정을 미루면 이후 모든 코드가 수명 관리의 모호함 위에 쌓이게 됨
- 후보를 비교하면 선택은 분명해짐

| 방식 | 소유 주체 | 수명 관리 | 문제점 |
|-----|---------|--------|-------|
| 원시 포인터 `Connection*` | 불명확 | 수동 `delete` | 누수·이중 해제 위험 |
| 값으로 보관 `map<int, Connection>` | map | 자동 | 이동 비용, 주소 불안정 |
| `map<int, unique_ptr<Connection>>` | map(단독) | 자동 | 사실상 없음 |
| `shared_ptr` | 공유 | 참조 카운트 | 소유 주체가 흐려짐 |

- 결론은 `std::unordered_map<int, std::unique_ptr<Connection>>`
    - 키는 fd, 값은 그 연결 객체의 단독 소유권을 쥔 스마트 포인터
- 장점
    - `unique_ptr`은 단독 소유를 코드로 강제하므로 "누가 주인인가"가 타입 차원에서 명확함
    - map에서 원소를 지우는 순간 `unique_ptr` 소멸자가 `Connection`을 파괴하고, 그 안의 `Socket` 소멸자가 fd를 닫음
    - 객체 본체는 힙에 있으므로 map이 재해싱되어도 `Connection`의 주소는 변하지 않음
- 이는 fd 하나의 수명을 책임지던 RAII 소켓 래퍼를 한 단계 끌어올린 형태
    - 단일 `Socket`은 fd 하나의 수명을 책임짐
    - map은 여러 `Connection`의 수명을 한꺼번에 책임짐

> 수명의 핵심 규칙
>
> `Connection` 객체는 `EPOLL_CTL_DEL`이 끝나는 시점까지 반드시 살아 있어야 함
>
> epoll에서 등록을 해제하기 전에 객체를 파괴하면, 커널은 이미 사라진 객체를 가리키는 이벤트를 돌려줄 수 있음

## `data.fd`와 `data.ptr`
- `epoll_ctl`로 fd를 등록할 때 함께 넘기는 `struct epoll_event`에는 `data`라는 union이 들어 있음
- 이 union은 커널이 이벤트를 돌려줄 때 그대로 다시 실어주는 "사용자 정의 꼬리표"임
```c
struct epoll_event{
    uint32_t        events;     // EPOLLIN, EPOLLOUT 등
    epoll_data_t    data;       // 사용자가 임의로 채워 두는 union
};

typedef union epoll_data {
    void        *ptr;
    int         fd;
    uint32_t    u32;
    uint64_t    u64;
} epoll_data_t;
```
- 단일 fd만 다룰 때는 `data.fd`에 fd를 실으면 충분함
    - `epoll_wait`가 깨어나면 `events[i].data.fd`로 어떤 fd인지 식별함
    - 연결이 하나뿐이라 추가 조회가 필요 없음
- 다중 연결에서는 깨어난 fd로부터 그 연결의 `Connection` 객체를 찾아야 함
    - `data.fd`만 쓰면 `map.find(fd)`로 매번 조회해야 함
    - `data.ptr`에 `Connection*`를 실어 두면 조회 없이 곧장 객체에 접근할 수 있음

| 항목 | `data.fd` | `data.ptr` |
|-----|-----------|------------|
| 싣는 값 | fd 정수 | `Connection*` |
| 객체 접근 | map 조회 필요 | 즉시 역참조 |
| 조회 비용 | 해시 탐색 1회 | 없음 |
| 위험 요소 | 적음 | use-after-free |

- 더 빠른 `data.ptr`에는 그만큼 무거운 책임이 따름
    - 커널이 돌려주는 포인터가 가리키는 객체는 그 이벤트를 처리하는 순간까지 반드시 유효해야 함
    - 만약 연결을 닫으면서 `Connection`을 먼저 파괴하고 `EPOLL_CTL_DEL`을 나중에 하면, 그 사이에 도착한 이벤트의 `ptr`은 이미 해제된 메모리를 가리킴
    - 이것이 use-after-free이며, 즉시 죽지 않고 잠복하다 터지는 가장 까다로운 버그 부류임
- 그래서 `unique_ptr`를 사용
    - `data.ptr`에는 map에 보관된 `unique_ptr`이 관리하는 객체의 raw 포인터(`get()`)를 실음
    - 객체의 실제 수명은 여전히 map의 `unique_ptr`이 단독으로 책임지므로, 포인터가 가리키는 대상의 생존 보장이 한 곳으로 일원화됨

```
[등록 시]
  map[fd] = make_unique<Connection>(...)   // 객체 생성, map이 소유
  ev.data.ptr = map[fd].get()              // raw 포인터를 꼬리표로 첨부
  epoll_ctl(ADD)

[이벤트 처리 시]
  epoll_wait → ev.data.ptr 를 Connection* 로 캐스팅 → 즉시 사용

[해제 시 - 순서가 생명]
  epoll_ctl(DEL)          // 1. 먼저 커널 등록 해제 (이후 이벤트 안 옴)
  map.erase(fd)           // 2. 그 다음 객체 파괴 (unique_ptr 소멸 → Socket close)
```
> 닫는 순서를 거꾸로 하면 안됨
>
> 반드시 `EPOLL_CTL_DEL`을 먼저 하고, 그 후에 map에서 지워 객체를 파괴함
>
> "커널에게 이 객체 일은 알리지 마라"고 전달한 뒤에 객체를 치워야 안전함

## Connection 설계
- `Connection`은 연결 하나의 상태를 응집한 클래스임
- RAII 소켓 래퍼 `Socket`을 멤버로 들고, 입력·출력 버퍼와 상태를 더함

```cpp
// connection.h
# include <string>
# include "socket.h"        // RAII Socket 래퍼

class Connection {
private:
    Socket          sock_;          // 연결 소켓 (단독 소유, 이동으로만 받음)
    std::string     read_buf_;      // 수신 버퍼
    std::string     write_buf_;     // 송신 버퍼 (아직 다 못 보낸 잔여분)

public:
    explicit Connection(Socket sock) : sock_(std::move(sock)) {}

    int fd() const noexcept { return sock_.get(); }

    // 데이터 처리의 단일 진입점 -- 이 자리에 이후 프로토콜 파서가 들어옴
    // 반환값: true면 연결 유지, false면 연결 종료 요청
    bool on_readable();
};
```

```cpp
// connection.cpp
# include "connection.h"
# include <unistd.h>
# include <cerrno>

bool Connection::on_readable() {
    char buf[4096];
    while (true) {                      // 레벨 트리거라도 버퍼는 비우는 게 안전
        ssize_t n = ::read(sock_.get(), buf, sizeof(buf));
        if (n > 0){
            // ── 데이터 처리 지점 ──
            // 지금은 echo: 받은 만큼 그대로 송신 버퍼에 적재
            // 이후 이 자리에 프로토콜 파서가 들어옴
            write_buf_.append(buf, static_cast<size_t>(n));
            continue;
        }
        if (n == 0){
            return false;               // FIN 수신: 연결 종료 요청
        }
        if (errno == EAGAIN || errno == EWOULDBLOCK){
            break;                      // 더 읽을 것 없음 (정상)
        }
        if (errno == EINTR){
            continue;                   // 시그널 중단 (정상)
        }
        return false;                   // 실제 오류: 연결 종료
    }

    // 적재된 송신 버퍼를 내보냄 (echo)
    if (!write_buf_.empty()){
        ssize_t w = ::write(sock_.get(), write_buf_.data(), write_buf_.size());
        if (w > 0){
            write_buf_.erase(0, static_cast<size_t>(w));
        }
    }

    return true;                        // 연결 유지
}
```
- 연결마다 fd를 받아 그 자리에서 끝까지 처리하던 자유 함수가, 여기서는 `Connection::on_readable` 멤버 함수로 응집됨
    - 자유 함수는 한 번 호출되면 연결이 끊길 때까지 그 안에 머물렀음
    - 멤버 함수는 한 번의 읽기 가능 알림마다 호출되고, 버퍼와 상태는 객체에 남아 다음 호출로 이어짐

## 데이터 처리 지점
- `on_readable` 안의 "데이터 처리 지점" 주석이 이번 설계의 핵심 의도
- 지금은 echo이지만, 이후 그 자리에서 HTTP같은 프로토콜 파서가 들어올 예정임
- 중요한 점은 그 교체가 `EventLoop`를 전혀 건드리지 않고 이뤄져야 한다는 점
```
                현재                       이후
              ┌──────────────┐         ┌──────────────┐
 EventLoop ──▶│ on_readable  │         │ on_readable  │◀── EventLoop (그대로)
              │  ┌────────┐  │         │  ┌────────┐  │
              │  │ echo   │  │   →     │  │ HTTP   │  │   교체 지점만 바뀜
              │  │ 로직    │  │         │  │ parser │  │
              │  └────────┘  │         │  └────────┘  │
              └──────────────┘         └──────────────┘
```
- 파서가 `EventLoop`에 직접 결합되어 있다면, 파서를 바꿀 때마다 이벤트 분배 로직까지 흔들림
- 파서를 `Connection` 내부의 한 지점에 가두면, `EventLoop`는 "읽을 게 있으니 처리하라"고 호출만 할 뿐 그 안에서 무엇을 하는지 알 필요가 없음

> I/O 다중화 계층(EventLoop)과 프로토콜 해석 계층(parser)을 분리하는 감각
>
> 이것이 확장 가능한 서버 구조의 출발점

## EventLoop 설계
- `EventLoop`는 epoll fd와 연결 맵을 소유하고, 영원히 도는 디스패치 루프를 돌림
```cpp
// event_loop.h
# include <sys/epoll.h>
# include <unordered_map>
# include <memory>
# include <atomic>
# include "socket.h"
# include "connection.h"

class EventLoop{
private:
    void accept_new();                  // 리스닝 소켓에서 새 연결 수락·등록
    void close_conn(int fd);            // EPOLL_CTL_DEL 후 map에서 제거

    Socket  listen_sock_;
    int     epoll_fd_;
    std::unordered_map<int, std::unique_ptr<Connection>> conns_;    // 소유의 단일 지점

public:
    EventLoop(Socket listen_sock, int epoll_fd)
        : listen_sock_(std::move(listen_sock)), epoll_fd_(epoll_fd) {}
    ~EventLoop() { if(epoll_fd_ >= 0) ::close(epoll_fd_); }

    void run(std::atomic<bool>& stop);  // 시그널 핸들러가 세팅하는 종료 플래그
};
```
```cpp
// event_loop.cpp
# include "event_loop.h"
# include "socket.h"
# include <cerrno>

void EventLoop::accept_new(){
    while (true){
        int client_fd = ::accept(listen_sock_.get(), nullptr, nullptr);
        if (client_fd < 0){
            if (errno == EAGAIN || errno == EWOULDBLOCK)    break;      // 더 받을 것 없음
            if (errno == EINTR)                             continue;
            break;
        }
        set_nonblocking(client_fd);

        auto conn = std::make_unique<Connection>(Socket{client_fd});

        epoll_event ev{};
        ev.events   = EPOLLIN;              // 레벨 트리거 (기본)
        ev.data.ptr = conn.get();           // map의 unique_ptr이 관리하는 객체의 raw 포인터
        ::epoll_ctl(epoll_fd_, EPOLL_CTL_ADD, client_fd, &ev);

        conns_.emplace(client_fd, std::move(conn));     // 소유권을 map으로 이전
    }
}

void EventLoop::close_conn(int fd){
    ::epoll_ctl(epoll_fd_, EPOLL_CTL_DEL, fd, nullptr);     // 1. 커널 등록 먼저 해제
    conns_.erase(fd);                                       // 2. 객체 파괴 (Socket close)
}

void EventLoop::run(std::atomic<bool>& stop){
    constexpr int MAX_EVENTS = 64;
    epoll_event events[MAX_EVENTS];

    while (!stop.load(std::memory_order_relaxed)){
        int n = ::epoll_wait(epoll_fd_, events, MAX_EVENTS, -1);
        if (n < 0){
            if (errno == EINTR) continue;       // 시그널 중단: 루프 조건 재검사 후 종료
            break;
        }

        for (int i=0; i<n; ++i){
            if (events[i].data.ptr == nullptr){
                // 리스닝 소켓은 ptr 대신 약속된 표식으로 구분
                accept_new();
                continue;
            }
            auto* conn = static_cast<Connection*>(events[i].data.ptr);
            if (!conn->on_readable()){
                close_conn(conn->fd());         // 종료 요청 시 정리
            }
        }
    }
}
```
- 리스닝 소켓과 연결 소켓을 구분하는 방법이 하나 필요함
    - 위 예시에서는 리스닝 소켓 등록 시 `data.ptr`을 `nullptr`로 두어 표식으로 삼았음
    - 더 견고하게는 리스닝 소켓도 하나의 특수한 `Connection`으로 모델링하는 방법이 있으나, 지금 단계에서는 단순한 분기로 충분함

```cpp
// 리스닝 소켓 등록 (run 이전, 생성 지점)
epoll_event lev{};
lev.events      = EPOLLIN;
lev.data.ptr    = nullptr;              // 약속: nullptr이면 리스닝 소켓
::epoll_ctl(epoll_fd_, EPOLL_CTL_ADD, listen_sock_.get(), &lev);
```

## 단일 스레드 동시 처리
- 블로킹 단일 스레드 서버는 첫 연결의 `read`에 갇혀 두 번째 연결을 처리하지 못함
- 이번 구조에서는 `nc`를 세 개 이상 동시에 띄워도 모두 즉시 echo됨

```
[ 블로킹 단일 스레드 ]                 [ epoll 이벤트 루프 ]

 conn A  read() ─ 잠듦 ──┐            epoll_wait ─ 깨어남
 conn B  (대기, 못 받음)   │             ├─ A 읽기 가능 → on_readable(A)
 conn C  (대기, 못 받음)   │             ├─ B 읽기 가능 → on_readable(B)
                        ▼             └─ C 읽기 가능 → on_readable(C)
   A가 끝나야 B 차례          한 번의 깨어남에 준비된 모두를 순서대로 훑음
```
- 핵심은 "동시(simultaneous)"가 아니라 "교대로 매우 빠르게(interleaved)"임
    - 단일 스레드는 여전히 한 순간에 하나의 `read`만 수행함
    - 다만 어떤 fd도 데이터가 없을 때 그 자리에서 잠들지 않음
- 잠듦이 사라진 자리를 epoll이 메움
    - `epoll_wait`는 "준비된 fd가 하나라도 생길 때까지"만 잠들고, 깨어나면 준비된 fd들의 목록을 한꺼번에 돌려줌
    - 루프는 그 목록을 순서대로 훑으며 각 `Connection`의 `on_readable`을 호출함
    - 각 호출은 논블로킹이라 데이터를 다 비우면 `EAGAIN`으로 즉시 빠져나와 다음 연결로 넘어감
- 결과적으로 세 연결이 거의 동시에 응답받는 것처럼 보이지만, 실제로는 한 스레드가 셋 사이를 매우 빠르게 오가는 것임

> 동시성(concurrency)과 병렬성(parallelism)의 차이가 여기서 체감됨
>
> 이 서버는 병렬이 아니라 동시적임 — 하나의 일꾼이 여러 일을 번갈아 처리함

## 정리
```
이벤트 루프와 연결 객체의 책임 분리
    │
    ├── 책임 분리
    │     ├── EventLoop : epoll fd 소유, 디스패치, accept, DEL
    │     └── Connection : Socket + 버퍼 + 상태
    │
    ├── 소유권 모델
    │     ├── unordered_map<int, unique_ptr<Connection>>
    │     ├── 단독 소유 → 타입으로 강제
    │     └── 객체 본체는 힙 → 재해싱에도 주소 불변
    │
    ├── data.fd vs data.ptr
    │     ├── data.fd  : map 조회 필요, 안전
    │     ├── data.ptr : 즉시 역참조, 빠름
    │     └── use-after-free 주의 → DEL 먼저, erase 나중
    │
    ├── 다중 연결 동시 처리
    │     ├── 단일 스레드 유지
    │     ├── 잠듦 제거 → epoll_wait만 대기
    │     └── 동시성(interleaved) ≠ 병렬성
    │
    └── 인터페이스 설계
          ├── on_readable = 데이터 처리 단일 진입점
          ├── echo → 프로토콜 parser 교체 지점
          └── I/O 다중화 계층과 프로토콜 계층 분리
```
