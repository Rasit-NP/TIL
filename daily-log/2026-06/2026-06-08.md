## 블로킹은 실제로 어떤 일인가
- `read`나 `accept`가 "멈춘다"는 표현은 비유가 아니라 커널이 그 프로세스를 실제로 재운다는 뜻
- 운영체제 입장에서 프로세스는 항상 하나의 상태를 가지며, 그중 핵심은 실행 가능(runnable)과 대기(blocked/sleeping)임
- 블로킹 호출이 일어나는 과정을 단계로 풀면 다음과 같음
```
프로세스가 read(fd) 호출
        │
        ▼
커널: 수신 버퍼에 데이터가 있는가?
        │
        ├─ 있음 ──▶ 복사해서 즉시 반환 (프로세스 계속 실행)
        │
        └─ 없음 ──▶ 1. 프로세스를 'blocked' 상태로 전환
                    2. 실행 큐(run queue)에서 빼냄  → CPU를 다른 프로세스에 양보
                    3. 해당 fd의 대기 큐(wait queue)에 등록
                          │
                          ▼  (데이터 도착 = 네트워크 인터럽트)
                    4. 커널이 대기 큐의 프로세스를 깨워 'runnable'로 되돌림
                    5. 스케줄러가 다시 CPU를 줄 때 read가 반환
```
- 핵심은 "기다리는 동안 CPU를 한 사이클도 쓰지 않는다"는 점
- 블로킹은 비효율의 상징이 아니라, 할 일이 없을 때 자원을 깔끔하게 양보하는 매우 효율적인 대기 방식
- 문제는 효율이 아니라 동시성
- 프로세스가 한 `read`에서 잠들면, 그 스레드는 잠든 동안 다른 어떤 fd도 들여다볼 수 없음
> 단일 스레드 서버가 한 연결에 갇혀 다른 연결을 못 받았던 한계의 근본 원인이 바로 이 "잠듦"임
>
> 한 곳에서 잠들면 나머지 전부가 함께 멈춤

## O_NONBLOCK
- 논블로킹은 시스템 콜을 다른 함수로 바꾸는 것이 아님
- 똑같은 `read`·`accept`를 호출하되, 위 흐름도에서 "없음" 분기의 동작만 바꾸는 설정
- 블로킹에서는 데이터가 없을 때 프로세스를 재웠지만, 논블로킹에서는 재우지 않고 즉시 제어를 돌려줌
```
커널: 수신 버퍼에 데이터가 있는가?
        │
        ├─ 있음 ──▶ 복사해서 반환 (블로킹과 동일)
        │
        └─ 없음 ──▶ 재우지 않음
                    즉시 -1 반환 + errno = EAGAIN
                    ("지금은 줄 게 없으니 다음에 다시 와라")
```
- 즉 `O_NONBLOCK`은 "기다림"을 "즉시 보고"로 바꾸는 스위치임
- 덕분에 한 fd에 데이터가 없어도 그 자리에 묶이지 않고, 곧장 다른 fd를 확인하러 갈 수 있게 됨
- 이 fd는 비어 있으니 넘어가고 저 fd를 보러 가는 식의 "순회"가 비로소 가능해짐

## EAGAIN
- `read`가 `-1`을 반환하면 본능적으로 오류로 읽기 쉽지만, `EAGAIN`은 실패가 아니라 상태 보고
- 이름 그대로 "again", 즉 "아직 준비 안 됐으니 나중에 다시 시도하라"는 뜻
- 같은 음수 반환이라도 그 안에 담긴 의미는 `errno`로 갈라지며, 이것을 구분하는 것이 논블로킹 처리의 전부라 해도 과언이 아님

| 상황 | 반환값 / errno | 진짜 의미 | 올바른 처리 |
|-----|---------------|--------|----------|
| 데이터 받음 | `> 0` | 그만큼 읽음 | 처리 후 계속 |
| 상대가 연결 종료 | `0` | FIN 수신, 더 읽을 것 없음 | 연결 정리 |
| 아직 데이터 없음 | `-1`/`EAGAIN`·`EWOULDBLOCK` | 준비 안 됨(정상) | 넘어가고 다음에 재시도 |
| 시그널로 중단 | `-1`/`EINTR` | 시스템 콜이 끊김(정상) | 재시도 |
| 실제 오류 | `-1`/그 외 | 회복 불가 | 로깅 후 연결 정리 |

> 핵심 분기는 "반환값이 음수인가"가 아니라 "음수일 때 `errno`가 무엇인가"임
>
> `EAGAIN`·`EINTR`을 오류로 오인해 연결을 끊으면, 멀쩡한 연결이 죽거나 서버가 통째로 종료됨

## 플래그 설정 방법
- 소켓 fd의 동작은 파일 상태 플래그(file status flags)라는 비트 집합으로 제어되며, `fcntl`로 읽고 씀
- `O_NONBLOCK`은 그 비트 중 하나일 뿐이므로, 기존 비트를 보존한 채 이 비트만 더해야 함
- 그래서 "현재 값 읽기 → 비트 추가 → 다시 쓰기"의 read-modify-write 순서를 지킴
```cpp
# include <fcntl.h>
# include <cstdio>

void set_nonblocking(int fd){
    int flags = ::fcntl(fd, F_GETFL, 0);                    // 1. 현재 플래그 읽기
    if (flags < 0){
        perror("fcntl F_GETFL");
        return;
    }
    if (::fcntl(fd, F_SETFL, flags | O_NONBLOCK) < 0){      // 2. 기존 값에 O_NONBLOCK만 추가
        perror("fcntl F_SETFL");
    }
}
```
> `F_GETFL` 없이 `F_SETFL, O_NONBLOCK`만 쓰면 다른 플래그가 전부 지워짐
>
> 반드시 읽어온 값에 비트 OR로 덧붙여야 함

## 리스닝 소켓과 연결 소켓 양쪽에 적용
- `accept`로 새로 만들어진 연결 소켓은 리스닝 소켓의 논블로킹 속성을 자동으로 물려받지 않음
- 리스닝 소켓에만 걸면 `accept`는 즉시 반환되지만, 정작 데이터를 주고받는 `read`·`write`는 여전히 블로킹임
- 그러면 새 연결은 잘 받으면서 한 연결의 `read`에서 다시 잠드는, 절반만 전환된 상태가 됨
- 따라서 리스닝 소켓에 한 번, `accept`로 받은 연결마다 한 번씩 `set_nonblocking`을 호출해야 함
```cpp
# include <cerrno>

Socket listen_sock{create_listen_socket(8080)};
set_nonblocking(listen_sock.get());         // 리스닝 소켓: accept를 즉시 반환시킴

while (!g_should_stop.load(std::memory_order_relaxed)){
    int client_fd = ::accept(listen_sock.get(), nullptr, nullptr);
    if (client_fd < 0){
        if (errno == EAGAIN || errno == EWOULDBLOCK)    continue;
        if (errno == EINTR)                             continue;
        perror("accept");
        break;
    }

    Socket client{client_fd};
    set_nonblocking(client.get());          // 연결 소켓: read/write도 즉시 반환시킴
    handle_client(client.get());
}
```
- 연결 소켓이 논블로킹 되었으므로, 그 fd를 다루는 `handle_client`의 `read`도 같은 세 갈래를 처리해야 일관됨

```cpp
# include <unistd.h>
# include <cerrno>

// 한 연결의 echo 세션 -- 연결 소켓이 논블로킹이므로 read도 EAGAIN을 다뤄야 함
void handle_client(int fd){
    char buf[1024];
    while (true){
        ssize_t n = ::read(fd, buf, sizeof(buf));
        if (n > 0){
            ::write(fd, buf, n);            // 받은 만큼만 회신 (short read 대비)
            continue;
        }
        if (n == 0) break;                  // FIN: 상대가 정상 종료
        if (errno == EAGAIN || errno == EWOULDBLOCK)    continue;   // 아직 없음 (정상)
        if (errno == EINTR)                             continue;   // 시그널 중단 (정상)
        perror("read");
        break;
    }
}
```
> `accept`만 고치고 `read`를 블로킹 가정 그대로 두면, 핸들러가 `EAGAIN`을 오류로 오인해 멀쩡한 연결을 끊음
