## 시그널과 안전한 종료
- 시그널은 커널이 프로세스에 비동기적으로 전달하는 소프트웨어 인터럽트
- 종료 시그널을 가로채지 않으면 프로세스는 기본 동작인 즉시 강제 종료로 끝남
- 강제 종료는 처리 중이던 연결, 열린 소켓, 점유한 포트를 정리하지 못한 채 끊어버림
- 안전한 종료란 종료 신호를 받으면 새 연결 수신을 멈추고, 진행 중인 작업을 마무리한 뒤 자원을 정리하고 끝내는 절차를 의미함
- C++에서는 이 자원 정리를 소멸자에 위임하는 RAII로 자연스럽게 표현할 수 있음

## 주요 종료 시그널
- 서버가 다루어야 하는 대표적인 종료 시그널은 다음과 같음

| 시그널 | 발생 상황 | 가로채기 가능 여부 |
|-------|---------|---------------|
| `SIGINT` | 터미널에서 Ctrl+C 입력 | 가능 |
| `SIGTERM` | `kill`, 오케스트레이터의 정상 종료 요청 | 가능 |
| `SIGKILL` | `kill -9`, 강제 종료 | 불가능 |

> `SIGKILL`은 핸들러로 가로챌 수 없으므로 소멸자를 포함한 어떤 정리 로직도 기대할 수 없음
>
> 컨테이너 환경에서는 보통 `SIGTERM`을 먼저 보내고, 유예 시간이 지나면 `SIGKILL`을 보냄

## 시그널 핸들러와 종료 플래그
- 핸들러는 메인 흐름과 무관한 시점에 끼어들어 실행되므로, 그 안에서 할 수 있는 일이 엄격히 제한됨
- 핸들러 내부에서 호출해도 안전한 함수를 async-signal-safe 함수라 부름
- `std::cout`, `new`, `std::string` 생성 등은 락이나 내부 상태가 깨질 수 있어 핸들러 안에서 사용하면 안 됨
- 핸들러에서 안전하게 다룰 수 있는 타입은 `volatile std::sig_atomic_t`와 락-프리 원자 타입으로 한정됨
- C++에서는 락-프리가 보장되는 `std::atomic<bool>`을 종료 플래그로 사용함

```cpp
# include <atomic>
# include <csignal>

std::atomic<bool> g_should_stop{false};
static_assert(std::atomic<bool>::is_always_lock_free);      // 시그널 안정성 전제

extern "C" void handle_signal(int /*signo*/){
    g_should_stop.store(true, std::memory_order_relaxed);   // 안전한 원자 저장만 수행
}
```
- `extern "C"`로 선언해 C 링크지로 만드는 이유는, 시그널 핸들러가 C 호출 규약을 따르도록 보장하기 위함임
- `static_assert`로 플래그가 항상 락-프리임을 컴파일 타임에 확정해, 핸들러에서의 접근 안정성을 명시함
- 등록은 동작이 명확한 POSIX `sigaction`을 사용함

```cpp
# include <cstring>

void setup_signal_handlers(){
    struct sigaction sa{};
    sa.sa_handler = handle_signal;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = 0;                    // SA_RESTART 미설정 → accept()가 EINTR로 깨어남
    sigaction(SIGINT,   &sa, nullptr);
    sigaction(SIGTERM,  &sa, nullptr);
}
```

## RAII 소켓 래퍼
- 파일 디스크립터는 명시적으로 닫지 않으면 누수되는 자원임
- 소켓을 클래스로 감싸고 소멸자에서 닫으면, 스코프를 벗어나는 순간 자동으로 해제됨
- 예외가 발생하거나 루프가 어떤 경로로 빠져나가든 정리가 보장된다는 점이 RAII의 핵심임

```cpp
# include <unistd.h>

class Socket{
public:
    explicit Socket(int fd) : fd_(fd) {}
    ~Socket(){ if (fd_ >= 0) ::close(fd_); }

    Socket(const Socket&) = delete;
    Socket& operator=(const Socket&) = delete;

    Socket(Socket&& o) noexcept : fd_(o.fd_){ o.fd_ = -1; }
    int get() const noexcept { return fd_; }
private:
    int fd_;
};
```
> 복사를 막고 이동만 허용하는 이유는, 같은 디스크립터를 두 객체가 소유해 `close`가 두 번 호출되는 사고를 방지하기 위함임

## 메인 루프 종료 패턴
- 핸들러는 플래그만 바꾸고, 실제 종료 판단은 메인 루프가 담당함
- 루프 조건에서 원자 플래그를 검사하면 다음 반복으로 넘어갈 때 자연스럽게 빠져나옴
```cpp
# include <cerrno>

Socket listen_sock{create_listen_socket(8080)};

while(!g_should_stop.load(std::memory_order_relaxed)){
    int client_fd = ::accept(listen_sock.get(), nullptr, nullptr);
    if (client_fd < 0){
        if (errno == EINTR) continue;
        continue;
    }
    Socket client{client_fd};
    handle_client(client.get());
}
```
- 클라이언트 소켓을 `Socket`으로 받으면 매 반복 끝에서 자동으로 닫혀 누수가 생기지 않음
- 루프를 빠져나오면 `listen_sock`의 소멸자가 호출되어 점유하던 포트가 해제됨

## 블로킹 시스템 콜과 EINTR
- `accept`는 연결이 들어올 때까지 대기하는 블로킹 호출임
- 대기 중 시그널이 도착하면 핸들러가 실행된 뒤 블로킹 호출이 중단됨
- `SA_RESTART`를 설정하지 않은 경우, `accept`는 `-1`을 반환하고 `errno`를 `EINTR`로 설정함
- 이 때, `EINTR`를 정상 흐름으로 처리하지 않으면 오류로 오인해 루프가 깨질 수 있음

```
정상 대기 중                Ctrl+C 입력
   │                          │
accept() 블로킹  ──────────▶  핸들러 실행 (g_should_stop=true)
   │                          │
   ▼                          ▼
accept() 반환 -1, errno=EINTR
   │
   ▼
errno==EINTR → continue → while 조건 재검사 → 종료
```
> `EINTR` 처리는 "시그널로 인한 중단"과 "실제 오류"를 구분하는 핵심 분기임
>
> 분기 없이 `accept` 실패를 곧장 에러로 취급하면 종료 신호가 비정상 종료로 변질됨

## SO_REUSEADDR와 포트 재사용
- 서버를 내린 직후 같은 포트로 재시작하면 종종 "Address already in use"가 발생함
- 능동적으로 연결을 닫은 쪽 소켓은 일정 시간 `TIME_WAIT` 상태로 남으며, 이 동안 해당 포트가 묶여 있음
- `TIME_WAIT`는 네트워크에 떠도는 지연 패킷이 새 연결에 섞이는 것을 막기 위한 정상적인 상태임
- `SO_REUSEADDR` 옵션을 켜면 `TIME_WAIT` 상태의 주소에도 새 소켓을 바인딩할 수 있어서 즉시 재시작이 가능해짐

```cpp
int opt = 1;
::setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
```
> 설정 시점이 중요함
>
> `bind` 이후에 옵션을 켜면 이미 바인딩이 끝난 뒤라 효과가 없음

## 전체 종료 흐름
- 지금까지의 요소를 하나로 묶으면 다음 순서로 동작함
```
[기동]
  소켓 생성 → SO_REUSEADDR 설정 → bind → listen → Socket으로 감쌈
  시그널 핸들러 등록 (SIGINT, SIGTERM)
        │
        ▼
[정상 운영]
  while (!g_should_stop) { accept → Socket client → handle_client }
        │  ← Ctrl+C / SIGTERM
        ▼
[핸들러] g_should_stop = true   (원자 저장만 수행)
        │
        ▼
[종료 절차]
  accept() EINTR 반환 → 루프 조건 재검사 → 탈출
  Socket 소멸자 → close → 포트 해제
        │
        ▼
[재시작] 동일 포트 즉시 바인딩 성공
```
