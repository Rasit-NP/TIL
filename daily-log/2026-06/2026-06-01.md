## Echo 서버
- 클라이언트가 보낸 데이터를 그대로 되돌려 보내는 가장 단순한 형태의 TCP 서버
- 네트워크 프로그래밍 학습에서 디버깅·연결 동작 검증의 기준점으로 자주 쓰임
- "한 번 응답 후 종료" 구조를 "여러 메시지를 주고받는 지속 연결"로 확장하는 첫 단계

## 두 겹의 무한 루프 구조
> 외부 루프는 새 연결을 받고, 내부 루프는 한 연결 안에서 메시지를 주고 받음
```
외부 루프 -- 연결 수락
  └─ accept()           // 새 클라이언트 대기 (블로킹)
     │
     └─ 내부 루프 -- 데이터 송수신
        ├─ read()       // 클라이언트로부터 수신 (블로킹)
        ├─ n == 0  -> 정상 종료, 내부 루프 탈출
        ├─ n  < 0  -> 에러, 내부 루프 탈출
        └─ write()      // 받은 만큼 그대로 회신
     │
     └─ close(client_fd)   // 연결 정리 후 다음 accept로 복귀
```
- 외부 루프는 리스닝 소켓을 재사용하며 무한히 연결을 받음
- 내부 루프는 클라이언트가 끊을 때까지만 유지되는 일회성 루프임

## `read`의 반환값
> 시스템 콜의 반환값 세 가지를 구분해야 올바른 종료 처리가 가능함
| 반환값 | 의미 | 처리 |
|-------|------|-----|
| `> 0` | 실제 읽은 바이트 수 | 그 길이만큼 `write`로 회신 |
| `0` | EOF, 상대가 정상 종료 | 내부 루프 탈출 후 `close` |
| `< 0` | 시스템 콜 에러 | errno 확인 후 적절히 처리 |
- `read`가 0을 반환하는 시점은 TCP FIN을 받아 더 이상 읽을 데이터가 없을 때임
- `< 0`인 경우에도 `EINTR`(시그널로 인한 중단)처럼 재시도 가능한 에러와 치명적 에러를 구분해야 함

### Short read에 주의
- 요청한 길이보다 적게 읽어 오는 경우가 매우 흔함
- 따라서 `write`에 넘기는 길이는 `sizeof(buf)`가 아니라 `read`가 반환한 `n`을 사용해야 함
- TCP는 바이트 스트림 프로토콜이므로 "메시지 경계"가 보장되지 않음을 의미함

## 구현 예시
```cpp
# include <sys/socket.h>
# include <netinet/in.h>
# include <unistd.h>
# include <cerrno>
# include <cstring>
# include <cstdio>
# include <cstdlib>

int main(){
    int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
    if (listen_fd < 0) {
        perror("socket");
        exit(EXIT_FAILURE);
    }

    int opt = 1;
    if (setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt)) < 0){
        perror("setsockopt");
        exit(EXIT_FAILURE);
    }

    sockaddr_in addr{};
    addr.sin_family = AF_INET;
    addr.sin_addr.s_addr = htonl(INADDR_ANY);
    addr.sin_port = htons(8080);

    if (bind(listen_fd, reinterpret_cast<sockaddr*>(&addr), sizeof(addr)) < 0) {
        perror("bind");
        exit(EXIT_FAILURE);
    }

    if (listen(listen_fd, 10) < 0) {
        perror("listen");
        exit(EXIT_FAILURE);
    }

    // 외부 루프 -- 새 연결을 계속 받음
    while (true){
        sockaddr_in client_addr{};
        socklen_t client_len = sizeof(client_addr);
        int client_fd = accept(listen_fd, reinterpret_cast<sockaddr*>(&client_addr), &client_len);
        if (client_fd < 0){
            perror("accept");
            continue;       // 한 연결의 실패가 서버 전체를 죽이지 않도록 함
        }

        // 내부 루프 -- 한 클라이언트와의 echo 세션
        char buf[1024];
        while (true){
            ssize_t n = read(client_fd, buf, sizeof(buf));
            if (n == 0){
                // 클라이언트가 정상 종료 (FIN 수신)
                break;
            }
            if (n < 0){
                if (errno == EINTR)  continue;   // 시그널로 중단된 경우 재시도
                perror("read");
                break;
            }
            // 받은 바이트 수만큼만 회신 -- short read 대비
            ssize_t written = write(client_fd, buf, n);
            if (written < 0){
                perror("write");
                break;
            }
        }

        close(client_fd);
    }

    close(listen_fd);
    return 0;
}
```

## 동작 확인 절차
- 첫 번째 터미널에서 `nc localhost 8080` 접속 후 임의의 문자열을 입력하면 동일한 문자열이 즉시 되돌아옴
- 입력창에서 `Ctrl+D`를 누르면 nc가 FIN을 전송하고, 서버의 `read`가 0을 반환하며 내부 루프가 탈출함
- 서버는 다시 외부 루프의 `accept`로 돌아가 다음 연결을 대기함

## 단일 스레드 서버의 한계
- 첫 번째 nc 접속 중에 두 번째 터미널에서 또 다른 nc로 동시 접속을 시도함
- 두 번째 nc는 입력을 보내도 echo 응답이 오지 않음
    - 서버는 첫 번째 클라이언트의 내부 루프 안에 갇혀 `read`에서 블로킹 중이기 때문
- 첫 번째 nc를 종료하는 순간, 서버가 외부 루프로 복귀해 `accept`를 호출함
    - 이미 대기 중이던 두 번째 연결이 즉시 처리되며 응답이 한꺼번에 쏟아져 나옴

### 원인
- 단일 스레드는 한 시점에 하나의 블로킹 시스템 콜만 수행할 수 있음
- 첫 번째 연결의 `read`가 데이터를 기다리는 동안, 그 어떤 코드도 진행되지 않음
- 두 번째 연결은 TCP 스택의 accept 큐(`listen`이 만든 큐)에 그대로 적재되어 대기함
    - 즉, 연결 자체는 커널이 이미 완료해 두지만 애플리케이션이 꺼내 가지 못하는 상태임

### `listen`의 backlog가 보이는 순간
- accept 큐의 길이가 곧 `listen(fd, backlog)`에서 지정한 값임
- 이 단계에서는 동시 접속을 1개도 처리하지 못하지만, 큐 덕분에 연결 자체는 거부되지 않음
- backlog를 초과하는 연결이 쌓이면 그제서야 새 연결이 거부되거나 지연됨
