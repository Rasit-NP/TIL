# package와 import
## 패키지
- 클래스의 묶음
- 여러 클래스 또는 인터페이스를 포함할 수 있으며, 서로 관련된 클래스를 그룹 단위로 묶어 효율적으로 관리할 수 있음
- 같은 이름의 클래스라도 서로 다른 패키지에 존재하는 것이 가능
    - 따라서 자신만의 패키지 체계를 유지함으로써 다른 개발자가 개발한 클래스 라이브러리의 클래스와 이름이 충돌하는 것을 피할 수 있음
- 클래스의 실제 이름은 패키지 이름을 포함하고 있음
- 클래스가 물리적으로 하나의 클래스 파일(`.class`)인 것과 같이 패키지는 물리적으로 하나의 디렉토리임
    - 패키지에 속한 클래스는 해당 디렉토리에 존재하는 클래스 파일(`.class`)이어야 함
- 디렉토리가 하위 디렉토리를 가질 수 있는 것처럼 패키지도 다른 패키지를 포함할 수 있으며 점(`.`)으로 구분함
- 하나의 소스파일에는 첫 번째 문장으로 단 한 번의 패키지 선언만을 허용

## 패키지의 선언
```java
package package_name;
```
- 패키지 선언문은 반드시 소스 파일에서 주석과 공백을 제외한 첫 번째 문장이어야 함
- 패키지 이름은 대소문자를 모두 허용하지만, 클래스 명과 쉽게 구분하기 위해서 소문자로 하는 것을 원칙으로 하고 있음
- `Java`에서 기본적으로 이름없는 패키지(`unnamed package`)를 제공하고 있음
    - 소스 파일에 자신이 속할 패키지를 지정하지 않는 모든 클래스는 자동적으로 이름없는 패키지에 속하게 됨
- 간단한 프로그램은 패키지를 지정하지 않아도 문제가 없음
    - 큰 프로젝트나 Java API와 같은 라이브러리를 작성하는 경우에는 미리 패키지를 구성하여 적용해야 함

## `import` 문
- 소스 코드를 작성할 때 다른 패키지의 클래스를 사용하려면 패키지 이름이 포함된 클래스 이름을 사용해야 함
- 소스 코드를 작성하기 전, `import` 문으로 사용하고자 하는 클래스의 패키지를 미리 명시해주면 소스 코드에 사용되는 클래스 이름에서 패키지 이름은 생략할 수 있음
- `import` 문의 역할은 컴파일러에게 소스 파일에 사용된 클래스의 패키지에 대한 정보를 제공하는 것
- 컴파일 시에 컴파일러는 `import` 문을 통해 소스 파일에 사용된 클래스들의 패키지를 알아낸 다음, 모든 클래스 이름 앞에 패키지 이름을 붙임

## `import` 문의 선언
```java
import package_name.ClassName;
import package_name.*;
```
- 모든 소스 파일(`.java`)에서 `import` 문은 `package` 문 다음에, 그리고 클래스 선언문 이전에 위치해야 함
- `import` 문은 `package` 문과 달리 한 소스 파일에 여러 번 선언할 수 있음
- 클래스 이름을 지정해주는 대신 `*`을 사용하면, 컴파일러는 해당 패키지에서 일치하는 클래스 이름을 찾아야 하는 수고를 더 해야 함
    - 하지만, 실행 시 성능 상의 차이는 전혀 없음
- `import` 문에서 클래스의 이름 대신 `*`를 사용하는 것이 하위 패키지의 클래스까지 포함하는 것은 아님
    ```java
    import java.util.*;
    import java.text.*;
    ```
    - 위의 선언들을 아래로 대체할 수 없음
        ```java
        import java.*;
        ```

## `static import` 문
```java
import static java.lang.Integer.*;
import static java.lang.Math.random;
import static java.lang.System.out;
```
- `static import` 문을 사용하면 `static` 멤버를 호출할 때 클래스 이름을 생략할 수 있음
- 특정 클래스의 `static` 멤버를 자주 사용할 때 편리함
